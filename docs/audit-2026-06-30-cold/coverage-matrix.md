## Gapped ADRs detail

### B1-01 [P2, upgraded from P3] ADR 007 — sideload revisit-trigger never recorded in ADR 007 itself, despite being explicitly tracked to do so

ADR 007 states its own revisit trigger: a native build path that bypasses store review (sideloaded Android, enterprise iOS) re-opens the ADR. That trigger has fired — Phase-1 Android distribution is direct APK sideload per `docs/roadmap.md`/`docs/tranche-1-launch.md`. The 06-15 audit's `remediation-plan.md:56` named **both** ADR-027 and ADR-007 as needing this dated decision. Commit `009f598a` (PR #1449) executed half of that — it amended **ADR 027** with a dated "2026-06-16 trigger review" — but never touched `docs/adr/007-native-projects-source-of-truth.md` (no substantive edit since 2026-04-19). This audit's own `raw/v-mobile.md` re-checked CF-36 and declared it closed, but only verified the ADR-027 half — the audit process itself missed the ADR-007 half twice now. Secondary: ADR 007's overlay enumeration (3 files) is stale against ~40 real overlay files / ~15 overlay actions (corroborated by `v-mobile.md` M-02).
**Severity:** P2 — upgraded from the prior P3 because this is now a tracked-and-half-executed remediation, not fresh staleness; a process-control failure layered on a documentation gap, for a release-distribution security decision.

### B1-02 [P2] ADR 005 item 4 — in-memory rate limiting's stated trigger ("when we horizontally scale") has already fired

`flyctl scale show -a loopfinance-api` shows 2 machines live since 2026-04-17 — the same date as ADR 004/005. No Redis-backed limiter shipped; per-IP rate limits are live-bypassable today by distributing requests across machines. Directly compounds AUTH-01/ADM-01 findings from the vertical audits (per the tranche-reconciliation section below).

### B1-03 [P3] ADR 005 item 11 — "secret token transport via upstream" framing materially understates the now-live Loop-native auth posture

Production (`fly.toml:38`, `LOOP_AUTH_NATIVE_ENABLED=true`) runs Loop-native HS256 auth by default with atomic refresh rotation, reuse-detection + full-session revocation, and DSR-triggered revocation — exactly the controls ADR-005 frames as a future Phase-2 trigger. Posture is better than documented, not worse, but the doc is materially stale for anyone making decisions from it.

### B1-04 [P3] ADR 005 item 2 — barcode redemption (Phase 2) was silently shipped, ADR/roadmap not updated

`apps/backend/src/orders/barcode-fields.ts` + `get-handler.ts:198-212` (PR #1317, tested) ship exactly the wiring the ADR says doesn't exist. `docs/roadmap.md:84,281` is also stale.

### B1-05 [P3] ADR 005 item 7 — vitest jsdom enumeration is stale and partly factually wrong

111/150 web test files now opt into jsdom, including route-level components and 3 of the 4 hooks the ADR explicitly names as deliberate jsdom holdouts.

### B1-06 [P2] ADR 004 — "currently single-replica" claim contradicted by live infra since the ADR's own date

Same root fact as B1-02 (2 machines live since 2026-04-17); also the circuit-breaker key registry is no longer exhaustive — `ctx/operator-pool.ts` (CF-13) creates per-operator breakers outside the documented 7-key registry, not surfaced via `/metrics`.

### B1-07 [P3] ADR 002 — claimed Go-vs-TS clustering comparison test does not exist

No `.go` files or golden-output fixtures anywhere in the repo; only TS self-consistency tests exist.

### B1-08 [P3] ADR 003 — protobuf test coverage doesn't actually validate the protobuf payload

The Accept-header test accepts either format as a pass; no round-trip test exists on either client or server. Same incident class as a real prior outage (A4-115) with no regression test today.

### B1-09 [P3] ADR 008 — "no cleanup bookkeeping" consequence is now false; native share path untested

`purgeShareImage` (A4-059, PR #1317) was added because share-image PNGs containing redeem code+PIN were accumulating unbounded. ADR never updated. Native `Share.share`/`writeTempShareImage` have zero test coverage.

### B1-10 [P3] ADR 009 — citation drift + stale narrative

Cites `orders/repo.ts` (now `orders/fulfillment.ts`); claims a `DESC` index that's actually ascending; "LOOP stablecoins are a later upgrade" is already superseded in substance by live on-chain LOOP-asset payout in `fulfillment.ts` today (sibling ADRs 010/015).

### B1-11 [P3] No ADR for several shipped Capacitor-ecosystem dependencies

`@capgo/inappbrowser` (security-sensitive in-app-browser redeem-URL flow, hardened against `javascript:`/`file:`/`data:` URLs), `@aparajita/capacitor-biometric-auth` (mentioned only in passing in ADR 006), and `@capacitor/push-notifications` (wired to a no-op boot call, independently flagged as dead code — finding M-04) all lack the ADR AGENTS.md's "new dependency requires an ADR" rule calls for.

### B2-03 [P2, new finding] ADR 010 — documented on-chain refund path does not exist in code

ADR 010 describes an automated `paid → refund_pending → refunded` state machine with on-chain Stellar reversal. No such states exist in `db/schema.ts:622` (enum: `pending_payment, paid, procuring, fulfilled, failed, expired`). The only refund primitive (`credits/refunds.ts`) is an admin-triggered off-chain ledger credit that explicitly does not touch order state — no on-chain reversal exists anywhere. Not flagged in ADR 005 or `roadmap.md` either. This is the most consequential finding in the 010-019 batch and should have disqualified ADR 010 from "fully covered" in the prior audit.

### B2-04 [P3] ADR 010 — deposit watcher mischaracterized as a "Horizon streaming listener"

Actual implementation is a cursor-based polling tick (`payments/watcher.ts`, `runPaymentWatcherTick`), not an SSE stream — affects the ADR's own reconnect/supervisor consequence claim.

### B2-01 [P3] ADR 016 — "Open questions"/retry table contradicted by shipped trustline-probe code

Doc says the trustline probe is skipped and `op_no_trust` is terminal/admin-only; code (`payout-worker-pay-one.ts:68-118`) now probes pre-submit and self-heals — the code's own comment says it closes this exact ADR question. Misleading for on-call use.

### B2-02 [P3] ADR 016 — never documents the CF-14 cross-machine concurrency model

### 4. Headline verdict

**Loop is not honestly "Tranche-1 ready" today as currently documented.** The core purchase flow works end-to-end (real-money e2e validated 2026-05-14), but the launch docs are silent on several live P0/P1 gaps directly in the critical path (CTX-01 operator-breaker permanent-brick, ORD-001/002 silent unrefunded failures, AUTH-01 OTP brute-force-evasion on a fleet already running 2 machines, M-01's likely-dead Apple Sign-In button) and materially mischaracterize Tranche-2 readiness (the wallet work is fully built-and-frozen with its own P0s, not "not started").
