## Findings

### V3-01 [P1 · LIVE] No sanity bounds, cross-source check, or circuit breaker on either price feed — a feed glitch or compromise can mis-price an order/payment by any factor

- File: `apps/backend/src/payments/price-feed.ts:137-222` (XLM oracle, CTX-rates + CoinGecko adapters), `apps/backend/src/payments/price-feed-fx.ts:109-200` (Frankfurter fiat FX)
- Description: Both feeds validate only `price > 0` / `rate > 0` (`price-feed.ts:165`, `price-feed-fx.ts:196`). Neither feed compares the freshly-fetched value against the previous cached value, against a second independent source, or against a hardcoded plausibility band (e.g. "XLM/USD should be between $0.01 and $10"). A 60s TTL cache (`CACHE_TTL_MS` in both files) bounds _staleness_ but not _correctness_ — once the TTL expires, whatever the upstream returns is trusted outright.
- Impact: `requiredStroopsForCharge` / `usdcStroopsPerCent` feed directly into `isAmountSufficient` (`amount-sufficient.ts:141`, `:112`), the watcher's pass/fail gate for marking an order `paid`. If `rates.ctx.com`, the CoinGecko-shape override, or `api.frankfurter.app` ever returns a wildly wrong number — a provider-side bug, a stale/cached-zero-then-recovered response that happens to be 100x off, a compromised DNS/CDN edge for a plain-HTTPS unauthenticated REST endpoint — the watcher computes a `requiredStroops` that is too low, and accepts a payment that is a fraction of the gift card's real value. This is a direct, unbounded fund-loss vector for Loop (the gift card still gets procured from CTX for full price). The reverse glitch (price reported too low) makes Loop systematically overcharge users, a correctness/trust problem in the other direction.
- Evidence: `price-feed.ts:164-172` (`fetchCtxRate` only checks `Number.isFinite(price) && price > 0`), `price-feed-fx.ts:196-198` (`if (rate <= 0) throw …` — no upper bound, no delta-from-last-good check). No test in `price-feed.test.ts` exercises an implausible-but-positive rate (confirmed via grep — only `non-2xx`, `schema drift`, `missing rate`, `non-positive rate` are tested). `amount-sufficient.ts` has no independent secondary check before trusting the oracle result.
- Minimal fix: Reject (and alert) when a freshly-fetched rate differs from the last cached value by more than a configurable percentage (e.g. 20%) within one TTL window; keep serving the last-known-good value and page Discord instead of accepting the outlier blindly. Apply to both `price-feed.ts` and `price-feed-fx.ts`.
- Better fix: Add a second independent feed (e.g. CoinGecko + CTX rates cross-checked for XLM; Frankfurter + a second FX provider) and require the two to agree within a tolerance before accepting either; fail closed (reject order/payment, 503) on divergence. This is the standard oracle-safety pattern (Chainlink-style multi-source median) and directly addresses Part-6 checklist item 35.

### V3-02 [P1 · LIVE] CF-14's `FOR UPDATE SKIP LOCKED` row-claim is not transaction-scoped — the lock is released before the claim it's meant to protect even begins

- File: `apps/backend/src/credits/pending-payouts.ts:116-147` (`listClaimablePayouts`), called from `apps/backend/src/payments/payout-worker.ts:107-145` (`runPayoutTick`)
- Description: `listClaimablePayouts` runs `db.select()...for('update', { skipLocked: true })` as a **standalone Drizzle call, not inside `db.transaction()`**. Every other `.for('update')` site in this codebase (`orders/transitions.ts:92`, `orders/repo-credit-order.ts:74`, `credits/withdrawals.ts:108`, `credits/refunds.ts:175/255`, `credits/adjustments.ts:155`, `credits/accrue-interest.ts:144`, `credits/payout-compensation.ts:107/197`, `users/home-currency-change.ts:103` — confirmed by grep) is wrapped in `db.transaction(async (tx) => {...})` precisely because a `SELECT ... FOR UPDATE` only holds its row locks for the life of the enclosing transaction. Postgres auto-commits a statement that isn't inside an explicit `BEGIN`, so the locks taken by `listClaimablePayouts` are released the instant that one SELECT returns — milliseconds later, long before `payOne` (`payout-worker-pay-one.ts:68`) gets to the trustline read, the idempotency pre-check (two more Horizon round-trips), and finally `markPayoutSubmitted`/`reclaimSubmittedPayout`.
- Impact: The docstring on `listClaimablePayouts` (`pending-payouts.ts:96-105`) explicitly claims "`FOR UPDATE SKIP LOCKED` closes the read→claim window: a row another worker is mid-claim on (locked but not yet committed `submitted`) is SKIPPED here" and that this "stops the two instances from each running the (wasteful) trustline + idempotency Horizon reads on a row the other is about to win." Both claims are false for the actual call pattern used in production. The row is never "locked but not yet committed `submitted`" — by the time any worker reaches the claim UPDATE, the SELECT's implicit transaction has already committed and the lock is gone. The only protection against an actual double-pay remains the orthogonal `WHERE state = 'pending'` (or `attempts = expected`) CAS in `markPayoutSubmitted`/`reclaimSubmittedPayout` (`credits/pending-payouts-transitions.ts:62-72, 76-90`) — which was already sufficient to prevent double-pay on a _single row_ before CF-14 shipped. What CF-14 was actually trying to reduce — two Fly machines both pulling the _same_ candidate batch and then claiming _different_ rows from it, each independently calling `submitPayout`, racing `loadAccount()`'s sequence-number read against the operator account — is **not** meaningfully reduced, because by the time either worker reaches that point the SKIP LOCKED protection has already expired. The residual risk is `tx_bad_seq` churn (burns attempt budget, can drive a legitimate payout to terminal `failed` faster than intended) under concurrent multi-machine load — not a literal double-pay, because Stellar's own sequence-number uniqueness is the real backstop (a resubmitted tx with a stale/colliding sequence is rejected by the network, not double-applied). So this is an availability/cost-of-retries bug, not a P0 fund-loss bug, but the **fix as documented does not do what it says**, and would mislead anyone investigating a `tx_bad_seq` spike during multi-machine operation into thinking CF-14 already covers it.
- Evidence: `pending-payouts.ts:116-147` — no `db.transaction()` wrapper. Compare to every other `.for('update')` call site in the repo (all wrapped). The integration test that exercises this
  (`apps/backend/src/__tests__/integration/payout-worker.test.ts:402-540`, "CF-14 FOR UPDATE SKIP LOCKED claim") only demonstrates the lock works _when artificially held open via an explicit `db.transaction()` with manual `Promise` gates that don't exist in production code_ — "instance A" in that test is `db.transaction(async (tx) => { ...select...for('update', {skipLocked:true})...; await aMayCommit; })`, manually kept open across an `await` the real `runPayoutTick()` never performs. The test is internally correct but is not representative of how `listClaimablePayouts` is actually invoked by `runPayoutTick` (`payout-worker.ts:108`), which calls it as a single bare statement with no enclosing transaction. There is no test exercising the real call pattern (two independent, non-transaction-wrapped `runPayoutTick()` calls racing) to demonstrate the claimed benefit — because it doesn't hold.
- Minimal fix: Document the residual honestly (the per-row CAS, not the row lock, is what prevents double-pay; the row lock only helps for sub-millisecond literal overlap of two SELECTs) and downgrade the docstring's claims accordingly so a future on-call engineer investigating `tx_bad_seq` churn doesn't waste time assuming CF-14 already closed it.
- Better fix: Make the claim atomic with the read. Wrap the candidate SELECT and an immediate state-flip UPDATE (`pending`/stale-`submitted` → a short-lived `claimed` sub-state, or directly → `submitted`) in one `db.transaction()` with no network I/O inside it (move the trustline + idempotency Horizon reads to _after_ the transaction commits, operating on rows the worker now exclusively owns per the DB). This is the standard Postgres job-queue pattern (`SELECT FOR UPDATE SKIP LOCKED` immediately followed by an `UPDATE ... RETURNING` in the same short transaction) and would make the row-claim genuinely race-proof across N machines without holding the lock across slow Horizon calls.

### V3-03 [P2 · LIVE] Path payments to the deposit account are invisible to the watcher — no skip record, no alert, no counter increment

- File: `apps/backend/src/payments/horizon.ts:191-213` (`isMatchingIncomingPayment`, gates on `p.type !== 'payment'`), `apps/backend/src/payments/watcher.ts:141-167` (`processPayment`)
- Description: `isMatchingIncomingPayment` only recognizes Horizon records with `type === 'payment'`. Stellar's `path_payment_strict_send` / `path_payment_strict_receive` operations — which deliver the exact same end-state (the destination account receives an asset) but via the DEX — are structurally excluded, exactly like `create_account`/`account_merge` (which are deliberately and correctly excluded per the schema comment at `horizon.ts:50-56`). Unlike those two, a path payment is a perfectly normal way for a real user's wallet to fund a deposit (e.g. a wallet that only holds XLM auto-routing a payment when the SEP-7 URI requests USDC, or vice versa). When this happens, `processPayment` returns `{ kind: 'no_match' }` (`watcher.ts:167`), which is silently dropped — `TickResult` doesn't even increment `unmatchedMemo` (that's reserved for "memo present but no matching order"), and the comprehensive-audit-2026-06-11-derived skip table (`skipped-payments.ts`) is never engaged because the payment never reaches a point where `recordSkip` would be called.
- Impact: A user who pays via a path payment has their funds land in the deposit account with **zero observability** — no skip row, no Discord alert, no counter that ops would ever look at. The order stays `pending_payment` until the 24h expiry sweep (`watcher-bootstrap.ts:57`) marks it `expired`, and there is no automated reconciliation path back to the user's funds. This is exactly the class of silent-deposit-loss the comprehensive-audit-2026-06-11 CRIT #1/#2 fix (the skip table) was built to close for _processing errors_, but the fix didn't extend to _operation-type mismatches_.
- Evidence: `horizon.ts:199` (`if (p.type !== 'payment') return false;`), `watcher.ts:141-167`, `__tests__/horizon.test.ts:130-181` tests `create_account`/`account_merge` exclusion explicitly but has no equivalent test (positive or negative) for `path_payment_strict_send`/`path_payment_strict_receive`.
- Minimal fix: Extend `isMatchingIncomingPayment` (or add a sibling check) to also match `path_payment_strict_receive`/`path_payment_strict_send` records on the _destination_ asset/amount (Horizon's payment-ops feed exposes the receiving side's `asset_type`/`asset_code`/`amount` for these the same way it does for `payment`), so a path-paid deposit is recognized exactly like a direct payment. At minimum, route non-`payment`-type records that land at the deposit account through `recordSkip` with a new `path_payment_unsupported` reason so ops gets visibility even before the matching logic is extended.
- Better fix: Treat any Horizon payment-op record with `to === account` and a recognized asset/amount as a candidate regardless of `type`, with `path_payment_strict_receive`/`send` handled explicitly (their `amount` field is the _delivered_ amount on the receiving side per Horizon's documented schema, so the existing size-check logic in `amount-sufficient.ts` should apply unchanged once the type-gate is relaxed).

### V3-04 [P2 · LIVE] `onSigned` hash-persist failures (introduced by CF-18) are always classified `terminal_other`, bypassing the attempt-budget retry policy even for a transient DB blip

- File: `apps/backend/src/payments/payout-submit.ts:213-225` (`submitNativePayment`), `:314-326` (`submitPayout`); `apps/backend/src/payments/payout-worker-pay-one.ts:231-239` (the `onSigned` callback)
- Description: CF-18 added an `onSigned` hook that calls `recordPayoutTxHash` (a DB `UPDATE`) before the tx is submitted to Horizon. Both `submitPayout` and `submitNativePayment` wrap _any_ throw from `onSigned` — whether a genuine transient Postgres connectivity blip inside `recordPayoutTxHash`'s own `db.update(...)` call, or the deliberate "row no longer in submitted state" abort thrown by `payout-worker-pay-one.ts:237` when another worker already moved the row — into the exact same `PayoutSubmitError('terminal_other', …)`. `handleSubmitError` (`payout-worker-pay-one.ts:286-321`) only treats `transient_horizon`/`transient_rebuild` as retryable against `maxAttempts`; `terminal_other` skips that check entirely and goes straight to `markPayoutFailed` on the very first occurrence, regardless of how many attempts remain.
- Impact: A routine, recoverable Postgres hiccup (brief connection-pool exhaustion, a failover blip, a statement timeout) during the narrow `recordPayoutTxHash` UPDATE — a class of error every _other_ transient failure mode in this same file gets retried for — instead immediately fails the payout row. For `kind='withdrawal'` rows this triggers `autoCompensateFailedWithdrawal` (so the user isn't shortchanged) but the withdrawal itself now needs manual `resetPayoutToPending` admin intervention for what could have resolved itself on the very next tick. For cashback (`kind='order'`) rows there is no auto-compensation — the row simply sits `failed` until ops notices and resets it. This directly contradicts the file's own stated intent ("fail-closed: better to retry than to submit without having recorded the hash" — `payout-submit.ts:313`): the code aborts the _submit_ correctly, but then denies the _row_ the standard retry budget that every other transient-error path in this same pipeline gets.
- Evidence: `payout-submit.ts:314-326` (unconditional `terminal_other` wrap), `payout-worker-pay-one.ts:297-298` (`isTransient = err.kind === 'transient_horizon' || err.kind === 'transient_rebuild'` — `terminal_other` not included). Test `__tests__/payout-worker.test.ts:515-533` ("CF-18: a hash-persist failure aborts the submit (fail-closed)") explicitly locks in this exact behavior — confirming it's a deliberate-but-unexamined design choice, not an oversight that slipped through review unnoticed, yet the test's own comment conflates "row moved out from under us" (harmless — `markPayoutFailed`'s `WHERE state IN ('pending','submitted')` guard makes this a safe no-op since the row already moved on) with a genuine DB-layer transient failure (not harmless — it burns the row).
- Minimal fix: In the `onSigned` callback (`payout-worker-pay-one.ts:231-239`), catch `recordPayoutTxHash`'s own thrown errors separately from the `stamped === null` case, and surface the former as `PayoutSubmitError('transient_horizon', …)` instead of letting it fall through to the generic catch-all in `payout-submit.ts`.
- Better fix: Add a dedicated `PayoutSubmitErrorKind` (e.g. `'transient_persist'`) so `handleSubmitError` can apply the standard `usedAttempts < maxAttempts` retry logic to persist failures exactly like Horizon transient failures, while keeping the `stamped === null` ("another worker already owns this row") case terminal-but-harmless as today.

### V3-05 [P2 · GATED (manifests once `min_machines_running` scales past 1)] Watcher alert-dedup state is per-process in-memory — multi-machine deployment amplifies the first page of every incident N×

- File: `apps/backend/src/payments/asset-drift-watcher.ts:71` (`assetState` Map), `apps/backend/src/payments/cursor-watchdog.ts:68` (`cursorStaleAlertFired`), `apps/backend/src/payments/stuck-payout-watchdog.ts:4` (`stuckPayoutAlertFired`), `apps/backend/src/payments/interest-pool-watcher.ts` (relies on `discord/monitoring.ts:249` `poolDepletionFired` Set)
- Description: Every one-shot "alert on transition, don't re-fire every tick" dedup gate in this vertical is a bare module-scope variable (`Map`, `Set`, or `boolean`). `apps/backend/src/index.ts` starts `startPaymentWatcher`, `startPayoutWorker`, `startAssetDriftWatcher`, and `startInterestPoolWatcher` unconditionally on boot, with no leader-election or single-instance guard — the same fact CF-14's own docstring states explicitly for the payout worker ("All Loop background workers run in-process on EVERY Fly machine — there is no leader election"). `apps/backend/fly.toml` sets `min_machines_running = 1` (masking this today) but `auto_start_machines = true` scales out under load.
- Impact: The moment a second machine boots, each machine's `asset-drift-watcher`, `cursor-watchdog`, `stuck-payout-watchdog`, and `interest-pool-watcher` ticks **independently** against the same Horizon/DB state but maintains its own private dedup map. On the first tick after a real incident crosses threshold, every machine running the watcher fires its own Discord page for the same event (N pages instead of 1) — not unbounded spam (each machine's own one-shot gate still suppresses its own re-fires), but a real "alert storm on transition" that degrades signal quality exactly when ops needs a clean page. The same applies in reverse on recovery (N "recovered" pages).
- Evidence: `asset-drift-watcher.ts:71` (module-scope `Map`, no DB/shared-store backing), `discord/monitoring.ts:249` (`const poolDepletionFired = new Set<string>();`, module scope), `discord/monitoring-asset-drift.ts` (no internal dedup at all — every call sends a webhook unconditionally; the watcher's own state is the only gate). `apps/backend/src/index.ts` boot wiring confirmed to start all four watchers unconditionally alongside `startPayoutWorker`.
- Minimal fix: Document the N× amplification as a known limitation in the relevant runbooks so on-call doesn't mistake N near-simultaneous identical pages for N distinct incidents.
- Better fix: Move the transition-dedup state to a shared store (a `DISTINCT ON` query against the existing `watcher_cursors`-style table, a dedicated `alert_state` row, or a Postgres advisory lock around the notify call) so only the machine that actually wins the transition pages, mirroring the row-claim pattern this same delta introduced for payouts (once V3-02's fix lands, the same primitive — a short transactional claim — would work here too).

### V3-06 [P3 · GATED (same multi-machine precondition as V3-05)] Payment-watcher cursor write can regress under concurrent multi-machine ticks, causing redundant re-scans (not fund-loss, but wasted work + noisier logs)

- File: `apps/backend/src/payments/watcher.ts:62-93` (`readCursor`/`writeCursor`/`touchCursorUpdatedAt`), `:286-296` (`runPaymentWatcherTick` read-then-process-then-write)
- Description: Two machines' ticks both read the same starting cursor `C0` (if B reads before A has written), both fetch and process the same page, and both write a cursor derived from _their own_ page's last record. If machine A's tick is slow (e.g. blocked on FX-oracle calls inside `isAmountSufficient` for several payments) while machine B's tick — started later but reading the same stale `C0` — finishes first and advances the cursor past A's eventual write target, A's late write of its own (earlier) `last.paging_token` can regress the persisted cursor backward.
- Impact: A regressed cursor causes the next tick (on either machine) to re-scan an already-processed window. This is **not** a money-safety issue: `markOrderPaid` is a state-guarded transition (idempotent — a second call on an already-`paid` order returns `null`/`already_paid`), and `recordSkip`'s `onConflictDoUpdate` keyed on `paymentId` is similarly idempotent. The cost is wasted Horizon/DB round-trips and potentially confusing duplicate "Order transitioned" log lines during an active incident, not incorrect ledger state.
- Evidence: `writeCursor` (`watcher.ts:66-74`) is an unconditional upsert with no compare-and-set against the cursor's current value; nothing prevents a "stale" write from clobbering a more-advanced cursor.
- Minimal fix: None required at current `min_machines_running=1` operation; note as a documented limitation alongside V3-05.
- Better fix: Make `writeCursor` a conditional update (`WHERE cursor < $newCursor` using `paging_token`'s lexicographic/numeric ordering, or store a monotonic ledger-sequence component) so a stale write can never regress the persisted progress.

### V3-07 [P3 · LIVE] Five near-duplicate Horizon-decimal-to-stroops parsers, one with a demonstrated behavioral drift, directly contradicting `stroops.ts`'s stated centralization purpose

- File: `apps/backend/src/payments/stroops.ts:27-38` (canonical, exported `parseStroops`); private byte-identical copies in `horizon-asset-balance.ts:65-74` and `horizon-trustlines.ts:79-90`; a stricter regex variant `amountToStroops` in `horizon-circulation.ts:86-95`; a differently-behaved `decimalToStroops` in `horizon-find-outbound.ts:215-224` (new in this delta)
- Description: `stroops.ts`'s own doc-comment states it was created specifically to stop `watcher.ts` and `horizon-balances.ts` from carrying "byte-identical private copies" of this parser, "to prevent drift the next time someone tweaks the parsing rules in only one of the two call sites." That centralization did not hold: `horizon-asset-balance.ts` and `horizon-trustlines.ts` still carry their own private byte-identical copies (not the prior pair `stroops.ts` already absorbed — these are two _additional_ ones the 06-15 audit also flagged, P3-2, and which remain unfixed). This delta adds a sixth implementation, `horizon-find-outbound.ts:215-224`'s `decimalToStroops`, which is **not** byte-identical: it rejects (`return null`) any amount with more than 7 fractional digits, whereas `stroops.ts`/`horizon-asset-balance.ts`/`horizon-trustlines.ts` all silently truncate (`.slice(0,7)`) excess digits instead. Since a valid Horizon amount is never emitted with more than 7 decimals, this specific divergence can't currently trigger — but it's the exact drift the original consolidation was meant to prevent, now demonstrated rather than hypothetical.
- Impact: Low immediate risk (Stellar's protocol-level 7-decimal precision means the truncate-vs-reject divergence is currently unreachable), but a real maintainability hazard: a future bug fix to amount parsing (e.g. handling a Horizon response quirk) applied to one copy and not the other five will silently diverge financial-math behavior across the payout-idempotency path (`horizon-find-outbound.ts`), the trustline/balance paths, and the drift/circulation paths.
- Evidence: Five separate implementations confirmed by direct read of all five files; `horizon-find-outbound.ts:217-221` regex `^\d+(\.\d+)?$` + `if (frac.length > 7) return null;` vs `stroops.ts:28-37`'s `padEnd(7,'0').slice(0,7)` truncation.
- Minimal fix: Have `horizon-asset-balance.ts` and `horizon-trustlines.ts` import `parseStroops` from `./stroops.js` (per the original 06-15 finding, still open). Document explicitly why `horizon-circulation.ts`'s `amountToStroops` and `horizon-find-outbound.ts`'s `decimalToStroops` are intentionally different (sign support, reject-vs-truncate) rather than leaving the divergence implicit.
- Better fix: Consolidate all five into `stroops.ts` with named variants (`parseStroops` truncating, `parseStroopsStrict` rejecting on overflow, both with optional sign support) so every call site picks an explicit, documented policy instead of accidentally inheriting whatever the original author of that file happened to write.

### V3-09 [P3 · LIVE] Per-asset Horizon balance reads aren't batched against the single-fetch-decode-all-assets pattern already used elsewhere

- File: `apps/backend/src/payments/horizon-asset-balance.ts:83-130` (`getAssetBalance`, one Horizon `/accounts/{id}` fetch per `(account, code, issuer)` tuple), called per-asset in a loop from `asset-drift-watcher.ts:185-198` and `interest-pool-watcher.ts:89-98`

For the asset-drift watcher and interest-pool watcher, each tick issues one Horizon request per configured LOOP asset (today: up to 3) against the _same_ pool account, instead of one.
- Impact: Minor — 30s caching bounds the actual request rate, and 3 configured assets is a small multiplier. Worth fixing opportunistically if/when more LOOP-asset currencies are added (ADR 035's extended markets could eventually motivate more configured assets), since the inefficiency scales linearly with asset count for no benefit.
- Evidence: `horizon-asset-balance.ts:83-130` vs `horizon-trustlines.ts:92-156` — same underlying Horizon resource, different fetch granularity.
- Minimal fix: None required at current scale.
- Better fix: Give `getAssetBalance` (or a sibling) a multi-asset variant that does one `/accounts/{id}` fetch and returns a `Map<"code::issuer", bigint>`, mirroring `horizon-trustlines.ts`'s approach; have the drift/pool watchers call it once per account per tick instead of once per asset.

### V3-10 [P3 · LIVE] No unit-suite test exercises `listClaimablePayouts`'s actual call shape (only the env-gated, real-postgres integration suite touches it, and that suite's scenario doesn't match production usage — see V3-02)

- File: `apps/backend/src/credits/__tests__/pending-payouts.test.ts` (no reference to `listClaimablePayouts` at all — confirmed by grep), `apps/backend/src/__tests__/integration/payout-worker.test.ts:402-540`
- Description: The mocked unit suite for `pending-payouts.ts` covers `insertPayout`, `listPendingPayouts`, `listPayoutsForAdmin`, `markPayoutSubmitted`, `markPayoutConfirmed`, `markPayoutFailed`, `resetPayoutToPending` — but not `listClaimablePayouts`, the CF-14 function itself. The only coverage is the real-postgres integration suite (gated on `LOOP_E2E_DB=1`, not part of the default `npm test` unit run), and as documented in V3-02, that suite's "two concurrent claims" scenario artificially holds "instance A" open via an explicit `db.transaction()` + manual `Promise` gate — a shape that doesn't match how `runPayoutTick` actually calls `listClaimablePayouts` in production (a single, non-transaction-wrapped statement).
- Impact: The CF-14 fix's actual production behavior (two independent, non-overlapping `runPayoutTick()` calls racing) has no test coverage at all — neither confirming nor refuting the gap identified in V3-02.
- Evidence: `grep -n listClaimablePayouts` across `credits/__tests__/*.ts` returns nothing; the only other reference outside production code is the integration suite noted above.
- Minimal fix: Add a real-postgres integration test that calls `listClaimablePayouts()` twice back-to-back with no enclosing transaction (i.e. the actual `runPayoutTick` shape) and asserts whether the second call sees the rows the first call already returned — this would have caught V3-02 directly.

### V3-11 [P3 · LIVE] No cap or refund path for an XLM/USDC overpayment

- File: `apps/backend/src/payments/amount-sufficient.ts:48-147` (`isAmountSufficient`, every branch is a `>=` comparison with no upper bound)
- Description: `isAmountSufficient` validates that a received payment covers the required amount but never checks for (or flags) a payment that substantially _exceeds_ it. A user (or a wallet that mis-estimates the SEP-7 amount, or a user who fat-fingers an extra digit) who sends 10x the required XLM/USDC has the full excess silently absorbed — the order is marked `paid` with no record that an overpayment occurred, and there's no automated refund mechanism visible anywhere in this vertical.
- Impact: Real user funds permanently stuck with no self-service or automated recovery path; for Phase 1 demo-scope volumes this is a low-likelihood/low-blast-radius edge case (acknowledged risk, similar to `sep7.ts`'s documented quote-vs-receipt rate-movement tolerance), but it's a genuine boundary condition with no test coverage and no admin tooling reference found in this vertical for "this order overpaid by X, refund the difference."
- Evidence: Every branch in `amount-sufficient.ts` (`:88`, `:114`, `:142`) is `receivedStroops >= requiredStroops` with no companion upper-bound check or overpayment-delta recording.
- Minimal fix: Log (structured, queryable) when `receivedStroops` exceeds `requiredStroops` by more than a small tolerance, so at minimum ops has visibility to manually refund egregious overpayments.
- Better fix: Persist the overpayment delta on the order/payout row and surface it in the admin order-drill so support can action a refund without needing to manually diff Horizon records against the catalog price.
