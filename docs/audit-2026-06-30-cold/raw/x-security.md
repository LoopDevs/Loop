# Sweep: Security tree-wide route enumeration — raw findings

> Cold adversarial audit, 2026-06-30. Cross-cutting sweep: builds the
> complete backend HTTP route inventory from scratch (independent of the
> 20 vertical deep-dives) and checks for **consistency gaps across the
> whole surface** — "every sibling has X except this one" — rather than
> re-auditing any single route's internal correctness. Vertical raw files
> (`v-auth.md`, `v-admin-writes.md`, `v-admin-reads.md`, `v-ctx.md`, etc.)
> were skimmed for context per the brief and consulted to avoid duplicate
> filings; cross-references are called out explicitly below.

Method: `apps/backend/src/app.ts` + every file in `apps/backend/src/routes/`
parsed with a small paren-balanced script (mirrors
`scripts/check-openapi-parity.mjs`'s own mount collector) to extract every
`app.get/post/put/delete/patch(...)` call, whether it carries `rateLimit(...)`
/ `killSwitch(...)` / `requireAdminStepUp(...)` inline, and its file:line.
Cross-checked against `apps/backend/src/openapi.ts` + `openapi/*.ts`
registrations via the same static parser the `check:openapi-parity` CI gate
uses, then spot-verified a sample of handlers by reading source (Zod
schemas, circuit-breaker / Horizon / CTX call sites, idempotency-key
plumbing) to separate real gaps from correct-by-design omissions.

## Findings

### SEC-01 [P2 · LIVE] `PUT /api/admin/merchant-cashback-configs/:merchantId` is rate-limited at 60/min — 3-6x looser than every other money-affecting admin/auth/order write in the entire route surface

- File: `apps/backend/src/routes/admin-cashback-config.ts:70-74`.
- Evidence: enumerating every non-GET (write) route's rate limit across all 144 mounts (see "Full route inventory" above and the raw extraction), the complete write-route limit distribution is: `2/min` (merchants/resync — uniquely tight, calls upstream), `5/min` (request-otp), `10/min` (verify-otp, social ×2, orders ×2, discord/test, home-currency self-serve, stellar-address self-serve), `20/min` (logout, favorites add/remove, **all five other step-up-gated admin money writes**: credit-adjustments, refunds, withdrawals, payout-retry, payout-compensation, plus admin home-currency), `30/min` (step-up mint, refresh — both non-destructive token operations), and then **`60/min`** for exactly one write: the cashback-config `PUT`. No other state-mutating endpoint in the codebase sits above 30/min, and the admin money-writer cluster — the closest sibling group by both effect (changes how money is split between Loop/merchant/user on every future order) and by missing-step-up status — is uniformly 20/min.
- Impact: This compounds the already-documented ADR-028 gap (cashback-config writes are explicitly excluded from step-up, accepted because the audit trail makes them "reversible"). Reversibility doesn't bound _blast radius before detection_: a captured bearer token (the exact threat ADR-028 itself names — "a 15-minute admin access token, exfiltrated from a compromised laptop / clipboard / browser extension") can fire 60 cashback-config PUTs/min indefinitely (no step-up to expire after 5 minutes, unlike the other writers), each silently shifting `userCashbackPct`/`loopMarginPct`/`wholesalePct` for any merchant — e.g. zeroing user cashback fleet-wide while inflating Loop's margin, or the reverse (margin to zero, cashback to 100%, burning Loop's spread on every subsequent order) — for as long as the token is valid, at 3x the per-minute throughput the project's own convention allows for every comparable write. The per-row Discord diff + audit trigger means it's _detectable after the fact_, not _prevented_ — same gap class step-up exists to close for the other five writers.
- Minimal fix: drop the limit to `20, 60_000` to match its sibling money-writers — purely a config change, no logic risk.
- Better fix: same numeric fix, plus revisit the ADR-028 Phase-1 exclusion now that CF-06/CF-07/CF-08 already expanded the gated set twice this audit cycle (refund, compensation, home-currency all moved from "excluded" to "gated" after initial launch) — cashback-config is the last money-relevant write left in the excluded bucket and arguably has _larger_ per-call blast radius than a single credit-adjustment (it's a standing multiplier applied to every future order on a merchant, not a one-time balance change). At minimum, document in ADR-028 why the rate-limit wasn't also tightened to match the gated cluster when the exclusion decision was made, so a future reader doesn't read 60/min as "this was reconsidered and is fine."

### SEC-02 [P2 · LIVE, cross-referenced — do not double-count] JWT/step-up signing-key entropy validated by length only — confirmed via tree-wide pass to span 4 separate env vars with the identical gap

- File: `apps/backend/src/env.ts:264-299` — `LOOP_JWT_SIGNING_KEY`, `LOOP_JWT_SIGNING_KEY_PREVIOUS`, `LOOP_ADMIN_STEP_UP_SIGNING_KEY`, `LOOP_ADMIN_STEP_UP_SIGNING_KEY_PREVIOUS` all use `z.string().min(32, {...}).optional()` and nothing else.
- Already filed independently by two verticals from different angles: `v-auth.md` AUTH-07 (the JWT keys, user-facing blast radius) and `v-admin-writes.md` ADM-09 (the step-up keys, admin blast radius). This tree-wide pass confirms the **same exact validation gap is duplicated four times** rather than centralized — itself a DRY/consistency finding on top of the entropy gap: there is no shared `secretKeySchema()` helper, so a future fifth signing-key env var (e.g. a Phase-2 webhook HMAC secret) would need the same length-only check copy-pasted a fifth time, with the same blind spot, unless someone remembers to add entropy checking from scratch.
- Minimal fix: per AUTH-07/ADM-09 — add a cheap distinct-character-count or Shannon-entropy floor, logged as a boot warning (not a hard fail).
- Better fix: factor one `secretKeySchema(minLength, minEntropyBits)` Zod helper in `env.ts` and apply it to all four signing-key fields (plus `LOOP_REDEEM_ENCRYPTION_KEY`, which currently has no length/format validation at all beyond the base64/hex decode check noted in its own comment) so the next signing secret added to the schema inherits the check instead of needing to remember it.

### SEC-03 [P2 · LIVE] Inconsistent kill-switch coverage across the six step-up-gated admin money-writers — only 2 of 6 can be instantly killed mid-incident

- Files: `apps/backend/src/routes/admin-credit-writes.ts` (credit-adjustments:48-53, refunds:63-68, withdrawals:75-81), `apps/backend/src/routes/admin-payouts.ts` (retry:90-97, compensate:103-110), `apps/backend/src/routes/admin-user-writes.ts` (home-currency:25-31); kill-switch enum at `apps/backend/src/kill-switches.ts:60-67`.
- Description: All six routes share an identical risk profile by ADR-028's own classification — they're the complete step-up-gated set, each requiring a fresh `X-Admin-Step-Up` token because "a captured bearer token alone is insufficient." But only `withdrawals` (`POST /api/admin/users/:userId/withdrawals`) and `payouts/:id/compensate` carry `killSwitch('withdrawals')`; `credit-adjustments`, `refunds`, `payouts/:id/retry`, and `home-currency` have no kill-switch at all. The `KillSwitch` type (`'orders-legacy' | 'orders-loop' | 'auth' | 'withdrawals'`) has no `'admin-writes'` member to gate the other four under.
- Impact: per the runbook (`docs/runbooks/kill-switch.md`), a kill switch exists precisely for "a known-broken [rail] is causing every [X] to permanently fail" or an active-incident "stop the bleed" scenario. If an incident involves a compromised admin session actively issuing credit-adjustments or refunds (both directly mint positive `credit_transactions` balance — refund and compensation were explicitly hardened in CF-06/CF-07 for exactly this risk class), ops has no instant-off lever for those two paths — only the 5-minute step-up TTL and the 20/min rate limit bound the damage, both of which an in-progress incident (the scenario the kill switch exists for) has already defeated. `payout-retry` and `home-currency` are lower-severity but the same gap applies for completeness.
- Evidence: `grep -rn "killSwitch(" apps/backend/src/routes/*.ts` → exactly 8 hits across `admin-credit-writes.ts:77` (withdrawals only), `admin-payouts.ts:105` (compensate only), `orders.ts:70,76`, `auth.ts:57,66,80,86`; `credit-adjustments` (line 48) and `refunds` (line 63) in the same file as the gated `withdrawals` mount have no `killSwitch(...)` call, nor does `payouts-retry.ts`'s mount or `admin-user-writes.ts`'s mount.
- Minimal fix: add `killSwitch('withdrawals')` to `credit-adjustments` and `refunds` too (broaden the existing switch's scope to "admin money-writes" in the doc comment, since it already covers two of the four credit-mutating writers) — zero new env var, just two more call sites.
- Better fix: introduce a dedicated `'admin-writes'` `KillSwitch` member (`LOOP_KILL_ADMIN_WRITES`) covering all six step-up-gated routes uniformly, document it in `docs/runbooks/kill-switch.md` alongside the existing four, and keep `withdrawals` as a _more specific_ sub-switch an operator can flip independently when only the Stellar payout rail (not the ledger-only writes) is the problem — mirrors the existing `LOOP_KILL_ORDERS` / `LOOP_KILL_ORDERS_LEGACY` precedence-chain pattern already used for orders.

### SEC-05 [P3 · LIVE, confirmation of Part 6 §33, full-surface quantification] Every one of the 141 in-process rate limiters shares the same single-process `Map` — confirmed uniform across the entire route surface, not just the admin/withdrawal path already cited

- File: `apps/backend/src/middleware/rate-limit.ts:39` (`const rateLimitMap = new Map(...)`, module-local) + `apps/backend/fly.toml:65` (`min_machines_running = 1`).
- Description: `v-admin-writes.md` ADM-01 already notes this in passing for the withdrawal cap specifically. This sweep's unique contribution: having enumerated all 144 mounts, **all 141 `/api/*` rate limits — not a subset — key off the identical module-local `rateLimitMap`** keyed `${name}:${ip}`, with no shared store (Redis, Postgres advisory lock, Fly KV) backing it. Today `min_machines_running = 1` in `apps/backend/fly.toml` makes this dormant — there is exactly one process, so the in-process map _is_ the global state. The risk is purely conditional on a future capacity change.
- Impact: if `min_machines_running` is ever raised (a routine scaling response to load, not a deliberate security decision), **every rate limit in the system — OTP requests, order creation, credit-adjustments, all of it — silently multiplies by the machine count** with no code change and no alert, because Fly's load balancer doesn't pin a client to one machine. This is a single shared root cause behind what would otherwise look like N independent findings (one per route). A scaling PR is unlikely to be reviewed by someone who reads `rate-limit.ts`'s docstring (which does disclose the in-process design clearly) and connects it to the security implication.
- Minimal fix: a one-line CI/deploy guard — fail `flyctl deploy` (or a preflight script alongside `preflight-tranche-1.sh`) if `min_machines_running > 1` while the rate-limiter is still in-process, so the silent-multiply class can't ship via routine ops.
- Better fix: migrate `rateLimitMap` to a shared store (Postgres `UNLOGGED` table with `ON CONFLICT` increment-and-check, or Redis if one gets introduced for other reasons) so the configured limits stay accurate at any machine count — the documented direction in the Part 6 §33 checklist item. Out of scope to implement in this sweep; flagging the full-surface blast radius so it's prioritized correctly relative to single-route findings.

### SEC-06 [P3 · LIVE, no fix needed — documented for completeness] Self-serve `POST /api/users/me/home-currency` exists alongside the admin-only, step-up-gated `POST /api/admin/users/:userId/home-currency` — confirms the memory note `feedback_admin_only_user_writes` doesn't (and per ADR 015 isn't meant to) mean "no self-serve path ever"

- Files: `apps/backend/src/routes/users.ts:93-96` (self-serve, `requireAuth` only, 10/min, no step-up — correct, since a user can only mutate their own row) vs. `apps/backend/src/routes/admin-user-writes.ts:25-31` (admin-mediated, step-up-gated, 20/min).
