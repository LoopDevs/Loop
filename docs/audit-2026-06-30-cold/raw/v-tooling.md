## Findings

### TOOL-01 [P1 · LIVE] `archive/ctx-apply.mjs runInfo()` never consumes the review-decisions file — merchant copy bypasses the human-review gate entirely

- File: `tools/ctx-catalog/archive/ctx-apply.mjs:330-370` (`runInfo`), contrast with `:153-167` (`runImages`, which _optionally_ honors `--decisions`)
- Description: `review-server.mjs`'s UI collects per-field approve/reject votes for `description`/`instructions`/`terms`/`intro` (`fv('desc',...)`, `fv('instr',...)`, `fv('terms',...)` — `review-server.mjs:257-263`) into `/tmp/review-decisions.json`. But the script that actually writes that copy to production CTX, `runInfo()`, takes a flat `--info <file>` map and applies every entry unconditionally (gated only by `!FORCE && already-populated`, line 344) — it has **no `--decisions` parameter and never reads the decisions file**. Compare to `runImages()`, which at least has an _optional_ `--decisions` flag (lines 158-167) that nulls out unapproved logo/cover fields when passed — still opt-in, but present. `runInfo()` has no equivalent at all.
- Impact: any operator/agent who builds an `--info` JSON map (e.g. from a templated-text generator, a scrape summary, or an LLM-assisted rewrite of scraped marketing copy) and runs `ctx-apply.mjs --info <file>` writes that text straight to the live, customer-facing merchant `description`/`instructions`/`terms` with **zero enforcement** that it passed the review UI someone may believe is gating it. This is exactly the "human-review gate not actually in the critical path" failure mode Part 6 §34 asks about — except discovered on the apply side, not the review-server side that CF-03 hardened.
- Evidence: `runInfo()` body, `tools/ctx-catalog/archive/ctx-apply.mjs:340-369` — no `val('--decisions')`, no filter against any decisions map, just `Object.entries(map)`.
- Minimal fix: add the same optional `--decisions` plumbing `runImages()` has, filtering `intro`/`description`/`instructions`/`terms` to only the fields marked `'yes'` per merchant.
- Better fix: make `--decisions` **mandatory** (not optional) for both `runImages()` and `runInfo()` — refuse to run without it unless an explicit `--no-review-required` flag is passed and logged. The current "optional" design means correct behavior depends on the operator remembering a flag every time; a mandatory gate with an explicit, loud override is the standard pattern for this class of tooling (cf. how `tillo-allocate.mjs` etc. default-print a plan and require no flag to write — the asymmetry across scripts in this directory is itself the root problem; standardize on "safe by default, explicit unsafe override," consistently, repo-wide for this directory).

### TOOL-02 [P1 · LIVE] Supplier-feed product/brand titles flow unsanitized into production merchant names with no review gate

- File: `tools/ctx-catalog/ezpin-allocate.mjs:240` (`body: JSON.stringify({ name: p.name, country: p.country })`), `tools/ctx-catalog/svs-allocate.mjs:199` (same shape), `tools/ctx-catalog/tillo-allocate.mjs:215` (same shape), and `tools/ctx-catalog/archive/ctx-create.mjs:63` (same pattern, for completeness)
- Description: `p.name`/`b.name`/`p.Name` come directly from a supplier's catalogue feed (Tillo `brands.json`, SVS `products.json`, EzPin `catalogs.json`) with only light normalization for _matching_ (the `canon()`/`base()` helpers used to dedupe), not for the value actually written. The raw supplier string is sent verbatim as the new merchant's `name` to `POST /merchants`. None of the three allocators require `--dry-run`; running any of them with no arguments performs live writes immediately (the README acknowledges this generally — "they predate a uniform `--dry-run` convention" — but doesn't call out the content-injection angle specifically).
- Impact: this is the literal shape of the checklist's worry — "can adversarial content embedded in...a supplier feed manipulate...what gets written" — except for the _new-merchant-creation_ path rather than the _existing-merchant-content_ path the review UI was built for. A mislabeled, misleading, or malformed supplier title (whether from supplier error, a compromised supplier integration, or a deliberately crafted listing — e.g. `"Amazon $500 FREE bonus — claim now"` or a string containing markup) becomes a live, purchasable Loop catalog entry with **no human ever seeing it** before real users can buy it. Whether this is exploitable as literal stored XSS depends on React's default escaping in `apps/web` (likely safe), but the _brand-spoofing / scam-copy_ risk does not depend on XSS at all — it's a direct, unreviewed write of attacker-or-vendor-controlled text into the customer-facing catalog.
- Evidence: see file:line citations above; no sanitization function is applied to `p.name`/`b.name` between feed and `POST /merchants` body construction.
- Minimal fix: cap length, strip control/non-printable characters, and reject names containing HTML-special characters (`<>&"'`) before sending to `POST /merchants`; log+skip rather than silently pass through.
- Better fix: route new-merchant creates (not just content/media) through a lightweight review queue too — even a low-friction one (e.g. append-only `/tmp/new-merchants-pending.json` the operator skims before the next allocator run, or extend `domain-review-server.mjs`'s pattern to a "new merchant name" review lane) — since this is the one write path in the whole pipeline with literally zero human touch between an external supplier feed and the live catalog.

### TOOL-03 [P1 · PARTIALLY-FIXED, re-verification of CF-03 T-03] `demo-seed.mjs`'s production guard is silently defeated by the standard `flyctl proxy` workflow

- File: `tools/ctx-catalog/demo-seed.mjs:58-89`
- Description: the guard's entire safety model hinges on `isLocalDb = LOCAL_DB_HOSTS.has(host)` where `host` is the parsed `DATABASE_URL` hostname. If `isLocalDb` is true, **none** of the NODE*ENV/`--yes-non-prod` checks run at all (`if (!isLocalDb) { ...guard... }`). The standard way Loop operators reach the real (Fly) Postgres instance is `flyctl proxy <local-port>:5432 -a <prod-app>`, which tunnels production Postgres to `127.0.0.1:<local-port>` — i.e. `DATABASE_URL=postgres://...@localhost:<port>/...` against **real production data** is hostname-indistinguishable from a genuine local dev Postgres. The script's own comment shows awareness of "DATABASE_URL exported from a previous deploy/migration session" but only handles the case where the leftover URL has a \_non-local* hostname — it has no defense against the proxy-tunnel case, which is arguably the more common way an operator's shell ends up pointed at prod.
- Impact: an operator who has a `flyctl proxy` tunnel to production open in one terminal (a normal debugging workflow) and, in another terminal/session, runs `demo-seed.mjs` with that `DATABASE_URL` still exported (or pasted from shell history) gets **zero warning** before the script `DELETE`s `credit_transactions`/`pending_payouts`/`user_credits`/`orders` for that email's real account and overwrites their balance with fabricated demo data — exactly the ledger-affecting prod-data-loss scenario T-03 was supposed to close.
- Evidence: `LOCAL_DB_HOSTS = new Set(['localhost', '127.0.0.1', '::1', '0.0.0.0', ''])` short-circuits the entire guard regardless of what's actually listening on that port/host.
- Minimal fix: require an explicit `--local-confirm` flag (or check a distinguishing env var Loop's own tooling sets, e.g. refuse if `FLY_APP_NAME`/a proxy-marker env var is present) even for `localhost` hosts when running outside a recognized dev-compose context; at minimum, print a loud "connecting to `<host>:<port>` — confirm this is your local dev Postgres, not a flyctl-proxy tunnel" prompt requiring interactive confirmation (not just a flag) before the first DELETE.
- Better fix: have the script query something Loop-specific and prod-distinguishing before deleting — e.g. SELECT a row count threshold (refuse if `users` table has more than, say, 500 rows, since a real production user table will dwarf a disposable dev seed DB) or check for a `dev_seed_marker` sentinel row that only ever exists in throwaway dev databases, refusing to proceed if absent. A content-based check is robust to the proxy-tunnel blind spot in a way no hostname heuristic can be.

### TOOL-04 [P1 · LIVE, confirmed NOT fixed by CF-03] `scrape-media-proxied.mjs` still logs the raw residential-proxy URL (credential leak)

- File: `tools/ctx-catalog/scrape-media-proxied.mjs:206`
- Description: `console.log(\`...\nproxy: ${process.env.PROXY_SERVER}\n\`)`prints the unredacted`PROXY_SERVER` value to stdout on every run. Per the script's own docstring, this env var commonly carries embedded credentials (`scheme://user:pass@host`, with a `{session}` token substitution pattern). This is the exact "T-04" finding from the 06-15 cold audit.
- Impact: residential-proxy password ends up in terminal scrollback / any captured CI or session log.
- Evidence: confirmed by direct code read (line 206, unchanged) and by `git show e27063d2 --stat`, which shows CF-03 (commit `e27063d2`, "T-01,02,03,05,06") touched only `demo-seed.mjs`, `domain-review-server.mjs`, and `review-server.mjs` — `scrape-media-proxied.mjs` was **not part of that commit at all**. This vertical's audit brief frames CF-03 as covering "proxy-credential redaction in logs," but the commit's own message explicitly enumerates only T-01/02/03/05/06 — T-04 was never in scope for that fix, and it is demonstrably still open today.
- Minimal fix: redact to `new URL(process.env.PROXY_SERVER).host` (strip userinfo) before logging.
- Better fix: never construct the proxy config from a single combined URL with embedded credentials in the first place — already supported via separate `PROXY_USERNAME`/`PROXY_PASSWORD` env vars (see `proxyForSession()`, lines 25-38); make `PROXY_SERVER` host-only by convention and stop accepting/parsing embedded userinfo at all, removing the leak vector structurally rather than relying on every log site remembering to redact.

### TOOL-05 [P2 · GATED] `review-server.mjs` SSRF defense is resolve-then-fetch-by-hostname — vulnerable to DNS-rebinding TOCTOU

- File: `tools/ctx-catalog/review-server.mjs:84-102` (`safeImageFetch`)
- Description: CF-03's T-02 fix is well-built against the _obvious_ SSRF (arbitrary URL, no allowlist) — it does scheme check, host allowlist, `lookup(hostname, {all:true})` + per-address `isBlockedIp()`, and `redirect: 'error'`. But the actual data-fetching call, `fetch(rawUrl, init)` (line 101), performs **its own independent DNS resolution** internally (via Node/undici's resolver) rather than connecting to the specific IP address that was validated moments earlier. A DNS-rebinding-capable attacker (control of an allowlisted-looking domain's authoritative nameserver, or just a very short TTL + a race) can return a safe public IP to the `lookup()` validation call and a private/loopback/`169.254.169.254` IP to the `fetch()` call's internal resolution, defeating the validation entirely. This is the textbook "validate the hostname, fetch the hostname" SSRF-mitigation antipattern; the standard fix is to validate-then-connect-to-the-validated-IP (pinning), not validate-then-refetch-by-name.
- Impact: bypasses the otherwise-solid T-02 SSRF fix against a moderately sophisticated (not exotic) attacker who controls DNS for a domain an operator/agent chooses to preview via the review UI (a plausible vector: a malicious/seo-poisoned candidate domain surfaced by `ctx-domain-resolve.mjs`'s web-search step, later previewed through the review server). Blast radius is scoped to whatever the operator's machine/network can reach (other localhost services, internal network, or cloud metadata if run on a cloud-hosted box) — meaningful but not as severe as the original unauthenticated/0.0.0.0-bound T-01/T-02 combination CF-03 already closed.
- Evidence: `safeImageFetch()` body — `lookup()` result is checked, then `fetch(rawUrl, init)` is called with the original URL string, not a pinned IP.
- Minimal fix: none cheap — DNS pinning requires either a custom `dns.lookup` override passed to an `http.Agent`/`undici.Agent`, or fetching by IP with an explicit `Host`/SNI override. Document the residual risk if not fixed immediately.
- Better fix: use `undici`'s `Agent` with a custom `lookup` function that returns only the already-validated address(es) (so the actual TCP connection uses the pinned IP), or fetch via `http(s).request({ host: validatedIp, headers: { Host: hostname }, servername: hostname })` for TLS SNI correctness. This closes the TOCTOU gap structurally rather than by re-validating faster.

### TOOL-06 [P2 · LIVE] Vision-QC contact-sheet builders inject unescaped scraped URLs into rendered HTML — the review artifact itself is tamperable

- File: `tools/ctx-catalog/build-cover-sheets.mjs:30` (`<img src="${v.url}" loading="eager">`), `tools/ctx-catalog/build-logo-sheets.mjs:37` (`<img src="${v.logoUrl}" loading="eager">`)
- Description: both scripts interpolate `v.url`/`v.logoUrl` — values sourced from `ctx-media-final.json`, i.e. from the scraping pipeline (site-scraped `og:image`, Tavily results, image-search results) — directly into an HTML `src` attribute with no escaping, then render that HTML in headless Chromium (Playwright) and screenshot it for human/vision-agent review. A crafted URL value containing a `"` followed by an event handler (e.g. `https://evil.example/x.jpg" onerror="...` or, since these are typically wrapped through `images.weserv.nl?url=...`, a crafted _inner_ URL that survives the weserv wrapper) breaks out of the attribute and executes arbitrary script/DOM manipulation inside the very page being screenshotted for review.
- Impact: this is the closest analog in this codebase to "indirect prompt injection into an AI/human review step" — a malicious or compromised merchant website's scraped image URL can manipulate its own representation in the contact sheet a human or vision-QC agent is asked to trust (e.g. overlay a fake "looks correct" caption, hide a competing/correct logo, or simply exfiltrate the local file:// page's reachable content). It does not by itself write to CTX, but it directly undermines the integrity of the one review step the rest of the pipeline leans on. Contrast with `build-logo-montages.mjs` in the same directory, which renders logos from **local files via `sharp` compositing** and uses `esc()` only for the (already-trusted, catalog-internal) name label — the safe pattern already exists next door and simply wasn't applied here.
- Evidence: direct code read; no `esc()`/encoding applied to `v.url`/`v.logoUrl` before interpolation, unlike the consistent `esc()` usage in both review servers' HTML generation.
- Minimal fix: HTML-attribute-escape `v.url`/`v.logoUrl` the same way `review-server.mjs`'s `esc()` does, or use `encodeURI`/strict URL re-validation (reject anything that doesn't round-trip through `new URL()` cleanly) before interpolating.
- Better fix: render contact sheets the same way `build-logo-montages.mjs` does — fetch images server-side with `sharp`, composite locally, never hand a remote/attacker-influenced URL to a browser's `src` attribute at all. This also removes the live network dependency (and rate-limit fragility) from the QC step.

- File: `tools/ctx-catalog/ctx-domain-resolve.mjs:21`, `tools/ctx-catalog/resolve-missing-domains.mjs:11`, `tools/ctx-catalog/fetch-logos.mjs:13`, `tools/ctx-catalog/README.md:14`

### TOOL-09 [P2 · LIVE] `ezpin-availability-sweep.mjs --apply` has no blast-radius cap and mislabels its audit trail

- File: `tools/ctx-catalog/ezpin-availability-sweep.mjs:110-169` (`applyPhase`), `:151` (`statusReason`)
- Description: `applyPhase()` will disable or discount-trim **every** enabled merchant whose cached ezpin SKU shows `delivery_type === 0`, gated only by `--apply` (no second confirmation, no `--yes`, no sanity check on how many merchants would be affected). It also writes `statusReason: 'administrator_error'` (line 151) for what is actually an automated availability-sweep decision, not an administrator error — a misleading audit trail for anyone reviewing why a merchant was disabled later.
- Impact: a regression in EzPin's `/catalogs/{sku}/availability` endpoint (auth failure silently producing `delivery_type: 0` for everything, a malformed response, or an EzPin-side outage that the retry/backoff logic in `checkPhase()` doesn't fully insulate against) would, on the next `--apply` run, mass-disable a large slice of the live, purchasable catalog with no circuit-breaker. This directly matches Part 6 §40's "runaway-cost/runaway-action guard" framing, applied to catalog availability instead of spend.
- Evidence: code read of `applyPhase()`; no `affected.length / all.length` threshold check anywhere in the function.
- Minimal fix: add a sanity cap — abort (or require an explicit `--force-mass` flag) if `affected.length` exceeds some absolute (e.g. 50) or relative (e.g. 5% of enabled catalog) threshold; fix `statusReason` to a dedicated value (e.g. `'supplier_unavailable'`) distinct from `administrator_error`.
- Better fix: same pattern as the allocators' resumable done-files — make the sweep itself two-phase with a written plan file the operator reviews before a second `--confirm` invocation actually applies it, rather than trusting a single `--apply` flag plus operator vigilance.

### TOOL-11 [P3 · LIVE, carried over from 06-15 T-23/T-24] Review-server request handlers: unguarded sync I/O can crash the server; `/save` writes are non-atomic, unvalidated, unbounded

- File: `tools/ctx-catalog/review-server.mjs:112-159` (`loadData`, called from the `/data` handler with no try/catch), `:325-339` (`/save`); `tools/ctx-catalog/domain-review-server.mjs:118-125` (`/data`/`/decisions`, `readFileSync(QUEUE)` not wrapped), `:126-140` (`/save`, wrapped in try/catch for the write itself but still non-atomic/unbounded/unvalidated)
- Description: both servers' request handlers call synchronous `readFileSync`/`JSON.parse` directly inside the `async (req, res) => {...}` `createServer` callback with no try/catch around the read paths; a missing or corrupted `/tmp/*.json` input throws inside an unhandled async callback (Node's default since v15 is to terminate the process on an unhandled rejection), crashing the whole review server mid-session. Separately, both `/save` handlers persist the POSTed body via a raw `writeFileSync` (no temp-file-plus-rename), with no cap on accumulated body size (`body += c` un-bounded) and no `JSON.parse` validation before persisting — a client disconnecting mid-upload or a buggy client could write a truncated/non-JSON decisions file, and the next load of `/decisions` would hand a corrupted blob to the browser (or, more importantly, to whatever downstream apply script reads `/tmp/review-decisions.json` directly).
- Impact: availability/resilience issue for the operator's own tool, not a network-facing security hole (the server is loopback+token-gated post-CF-03). Still relevant because this file is the production-apply trust gate — losing review state mid-session forces re-review, and a crash-prone server makes operators more likely to skip the review step under time pressure.
- Evidence: direct code read, unchanged from the prior audit's T-23/T-24, not in CF-03's fix scope (T-01/02/03/05/06).
- Minimal fix: wrap each handler body in try/catch, returning 500 on read/parse failure instead of crashing the process; cap `/save` body size (reject >1MB) and `JSON.parse`-validate before persisting.
