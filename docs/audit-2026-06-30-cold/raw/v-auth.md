### AUTH-01 [P1 · LIVE] OTP attempts-counter targets the wrong row — both a DoS lockout and a brute-force-cap evasion

- File: `apps/backend/src/auth/otps.ts:138-158` (`incrementOtpAttempts`), call site `apps/backend/src/auth/native.ts:80-87`
- Description: `incrementOtpAttempts({ email })` does not take the guessed code into account at all — it always bumps the **single newest live row** for the email (`ORDER BY created_at DESC LIMIT 1`, A2-561). `nativeVerifyOtpHandler` calls it on _every_ failed `findLiveOtp` lookup, regardless of which row (if any) the submitted code was "aimed at". Two distinct exploit directions fall out of this:
  1. **DoS lockout** (the 06-15 framing, still open): an attacker who knows only the victim's _email_ (no code) can POST 5 garbage codes to `verify-otp`. Each miss bumps the victim's newest live row's `attempts` to `OTP_MAX_ATTEMPTS`. `findLiveOtp`'s `lt(attempts, OTP_MAX_ATTEMPTS)` filter then excludes that row — the victim's just-requested legitimate code now 401s even though they never mistyped anything. The per-IP rate limit (10/min on `verify-otp`) and 5-attempt cap mean a single IP can lock out a fresh code in well under a minute.
  2. **Brute-force-cap evasion** (new angle from this audit): an attacker trying to _guess_ an unknown live code can defeat the "5 attempts then locked" design intent entirely. `request-otp` allows 3 new OTPs/min per email (`OTP_REQUESTS_PER_EMAIL_PER_MINUTE`) and each row stays live for `OTP_TTL_MS` = 10 minutes, so a sustained attacker accumulates ~30 concurrently-live rows for one email. By periodically requesting a fresh "decoy" OTP, the attacker keeps `incrementOtpAttempts` pinned to the decoy while their actual guesses (10/min via the `verify-otp` route limit) probe the full live set. `findLiveOtp` matches a guess against _any_ live row's hash — so guesses against the ~29 non-decoy rows never increment those rows' `attempts` and are never locked out. The OTP_MAX_ATTEMPTS docstring's claim ("At 5 tries × 10⁶ codes, online brute force is not viable") assumes the cap actually bounds guesses against the live set; it does not once more than one row is live.
- Impact: this is the authentication entry point for the entire Loop-native path (`LOOP_AUTH_NATIVE_ENABLED=true` is set in `apps/backend/fly.toml`, i.e. this is the live production auth path, not a future/gated one). Direction 1 is a real, cheap login-denial-of-service against any known email. Direction 2 meaningfully weakens the documented brute-force-infeasibility claim: with ~30 concurrent live codes and a sustained 10/min guess rate (the documented "14,400 guesses/day" ceiling), expected successful guesses are no longer ~0 — they're on the order of `14400 × 30 / 1,000,000 ≈ 0.43/day`, i.e. a handful of expected successes per month against a single targeted, known-active email, entirely within the documented rate limits. (AUTH-06 below shows the per-IP rate limit itself is also diluted under Fly autoscale, which compounds this.)
- Evidence: `apps/backend/src/auth/otps.ts:138-158` — the subselect's `WHERE` clause has no `codeHash` predicate, only `email` + liveness; `apps/backend/src/auth/native.ts:80-87` calls `incrementOtpAttempts({ email })` (no code/hash passed) on every miss. No test in `otps.test.ts` or `native.test.ts` exercises the multi-live-row scenario — `incrementOtpAttempts` tests only assert "an update was issued", not which row.
- This is the **same root cause** flagged P2 in `docs/audit-2026-06-15-cold/raw/v-auth.md` ("Wrong-guess burns the victim's newest live OTP") and explicitly left unfixed across the 22-PR remediation wave between that audit and this one (no PR touches `incrementOtpAttempts`'s targeting logic; CF-26's only otps.ts change was adding `purgeExpiredOtps`). I'm raising it to P1 here given the brute-force-evasion angle in addition to the DoS angle, and the compounding multi-machine rate-limit dilution (AUTH-06).
- Minimal fix: pass the guessed `codeHash` into `incrementOtpAttempts` and target the specific row matching `(email, codeHash)` when one exists; when no row matches any live code (pure garbage guess), fall back to bumping _all_ live rows for the email (closer to pre-A2-561 behaviour) so a content-free guess still costs the attacker progress against the full live set.
- Better fix: decouple "guess accounting" from individual OTP rows entirely. Add a per-email rolling-window guess counter (a small table or a Redis/Postgres-backed counter keyed on `email`, TTL'd to the longest-lived concurrent OTP, e.g. 10 minutes) that increments on _every_ wrong verify attempt regardless of row, and gates `findLiveOtp` lookups once the window's count hits `OTP_MAX_ATTEMPTS`. This bounds total guesses against the _whole_ live set per email per window, independent of how many decoy rows exist, and doesn't lock out a user's legitimately-still-valid second code (the original A2-561 UX complaint) because the lockout is keyed on the wrong-guess budget, not which physical row got "burned".

### AUTH-02 [P2 · LIVE] OTP consume is not atomic — verify-otp can mint two sessions from one code under a race

- File: `apps/backend/src/auth/otps.ts:116-122` (`markOtpConsumed`), call site `apps/backend/src/auth/native.ts:88-90`
- Description: `markOtpConsumed` is an unconditional `UPDATE otps SET consumed_at = now() WHERE id = $1` — no `WHERE consumed_at IS NULL` guard, no compare-and-set, no transaction wrapping `findLiveOtp` → `markOtpConsumed` → `findOrCreateUserByEmail` → `issueTokenPair`. Two concurrent `verify-otp` requests presenting the same still-live, correct code both pass `findLiveOtp` (row not yet consumed) and both proceed to mint independent access/refresh pairs from a single one-time code.
- Impact: breaks the single-use contract the module's own docs assert (see below) and is the exact class of bug the refresh-rotation path was deliberately hardened against via `tryRevokeIfLive` (A4-098, in this same audit's scope — see the contrast in quality). Requires already knowing the valid code, so it's not a privilege-escalation path on its own, but it does mean a code intercepted via a side channel (shared network log, MITM, shoulder-surf) that two parties race to redeem yields two live sessions instead of one rejected attempt, and it silently doubles a refresh-token row's family for one login event.
- Evidence: `apps/backend/src/auth/otps.ts:117` docstring literally claims `/** Marks the OTP row consumed. Called inside the verify-otp txn on success. */` — there is no transaction; `native.ts:80-93` shows the call sequence is bare sequential awaits. No concurrent-verify test exists in `native.test.ts` (the file has a concurrent-rotation test for refresh, A4-098, but nothing analogous for OTP consume).
- Same finding as 06-15's P2 "OTP verify is not transactional / not consume-once under concurrency" — confirmed **still open**, untouched by the 22-PR wave.
- Minimal fix: make the consume a CAS — `UPDATE otps SET consumed_at = now() WHERE id = $1 AND consumed_at IS NULL RETURNING id`; treat zero rows returned as "already consumed" → 401. Correct the misleading docstring either way.
- Better fix: wrap `findLiveOtp` (as a `SELECT ... FOR UPDATE`) + the CAS consume + user upsert + token mint in a single DB transaction so the whole verify-otp success path is atomic end to end, matching the rigor already applied to refresh rotation.

### AUTH-04 [P2 · GATED — safe today by coincidence of defaults] Auth-row purge retention can outrun A2-1608 reuse-detection if misconfigured

- File: `apps/backend/src/auth/refresh-tokens.ts:182-197` (`purgeDeadRefreshTokens`), `apps/backend/src/auth/auth-row-purge.ts:69-81`, `apps/backend/src/auth/tokens.ts:64` (`REFRESH_TTL_SECONDS = 30 days`), `apps/backend/src/env.ts:641` (`LOOP_AUTH_ROW_RETENTION_DAYS` default 30, independently operator-tunable)
- Description: `nativeRefreshHandler`'s A2-1608 reuse-detection (`apps/backend/src/auth/native.ts:128-150`) depends on `findRefreshTokenRecord(jti)` finding a row with `revoked_at != null` when an old, already-rotated refresh token is replayed — that's what triggers `revokeAllRefreshTokensForUser` (the "attacker reused a stolen, already-rotated token" signal). The CF-26 purge sweep (`purgeDeadRefreshTokens`) deletes a row once `revoked_at < now - retentionMs`. By construction, a row can only be revoked at or before its own `expires_at` (the only call sites — `tryRevokeIfLive`, `revokeRefreshToken`, `revokeAllRefreshTokensForUser` — only ever act on rows that passed `verifyLoopToken`'s `exp` check or `findLiveRefreshToken`'s `expiresAt > now` filter). So whether the purge can delete a revoked-but-still-cryptographically-valid row depends entirely on whether `retentionMs < REFRESH_TTL_SECONDS` (30 days, hardcoded in `tokens.ts`, **not** derived from or validated against `LOOP_AUTH_ROW_RETENTION_DAYS`). With the current default (`LOOP_AUTH_ROW_RETENTION_DAYS=30` == 30-day refresh TTL), the algebra works out so the purge can never delete a row before its JWT's own `exp` (`revokedAt ≤ expiresAt` always ⇒ `revokedAt + 30d ≤ expiresAt + 30d`, and the purge's expired-branch only fires after `expiresAt + retention` — the row is purged at the same moment or after natural JWT expiry, never before). But if an operator (or a future change) sets `LOOP_AUTH_ROW_RETENTION_DAYS` to anything less than 30 — entirely plausible for PII-minimization reasons, the env var is a generic positive-int with no cross-validation against the refresh-token TTL — a row revoked early in its life (e.g. day 2 of a 30-day-lived token, very plausible since the access-token TTL is 15 minutes so the app refreshes often) gets purged at `revokedAt + retentionDays`, which can land well before `expiresAt`. In that window, a stolen-and-late-replayed rotated-out token hits `findRefreshTokenRecord(jti) === null` (row gone) instead of "revoked", so the handler falls into the `log.warn('Refresh token not live')` branch instead of `revokeAllRefreshTokensForUser` — a genuine token-theft replay silently degrades to a plain 401 with no family-wide revoke.
- Impact: none today (default config is safe) — this is a latent footgun, not a live bypass, hence GATED. But it's exactly the risk the 06-15 audit flagged hypothetically before this purge job existed ("If that job deletes a revoked-but-expired row, a later replay ... would hit the `record === null` branch → plain 401, no family revoke ... Verify any cleanup job's WHERE clause requires `expires_at < now()`") — and the now-shipped `purgeDeadRefreshTokens` does **not** add that guard; its `revoked_at`-branch has no `expiresAt` floor at all.
- Evidence: `apps/backend/src/auth/refresh-tokens.ts:182-197` — the `or(lt(expiresAt, cutoff), and(isNotNull(revokedAt), lt(revokedAt, cutoff)))` predicate; `tokens.ts:64` `REFRESH_TTL_SECONDS = 30 * 24 * 60 * 60` is a private const, never imported by `env.ts` or `auth-row-purge.ts` for cross-validation; `env.ts:641` `LOOP_AUTH_ROW_RETENTION_DAYS: z.coerce.number().int().positive().default(30)` has no `.min(30)` or any tie to the refresh TTL.
- Minimal fix: in `purgeDeadRefreshTokens`, additionally require `expiresAt < cutoff` in the revoked-branch (i.e. only purge a revoked row once it has _also_ naturally expired) — `or(lt(expiresAt, cutoff), and(isNotNull(revokedAt), lt(revokedAt, cutoff), lt(expiresAt, now)))`, or simpler: just drop the independent revoked-branch cutoff and key purge eligibility on `expiresAt` alone (a row can't be a reuse-detection signal once expired anyway, so there's no PII-retention upside to deleting it earlier via the revoked-branch).
- Better fix: export `DEFAULT_REFRESH_TTL_SECONDS` for use as a boot-time assertion in `env.ts` (or `index.ts`'s boot validation) that `LOOP_AUTH_ROW_RETENTION_DAYS * 86400 >= DEFAULT_REFRESH_TTL_SECONDS`, failing boot loudly if an operator ever sets retention below the refresh-token lifetime — turns the current "safe by coincidence" property into an enforced invariant.

### AUTH-06 [P2 · LIVE] In-memory per-IP rate limiter (and JWKS invalidate-debounce) is per-Fly-machine; effective auth-endpoint caps scale with autoscaled machine count

- File: `apps/backend/src/middleware/rate-limit.ts:36` (`rateLimitMap` — module-local `Map`), `apps/backend/src/auth/jwks.ts:43,59` (`jwksCache`, `lastInvalidatedAtMs` — also module-local), `apps/backend/fly.toml:62-65` (`auto_start_machines = true`, `concurrency.hard_limit = 250`)
- Description: every per-IP rate-limit bucket on the auth surface (`request-otp` 5/min, `verify-otp` 10/min, `social/*` 10/min, `refresh` 30/min, `DELETE /session` 20/min, `POST /api/admin/step-up` 30/min) is keyed in a process-local `Map`. Fly's `http_service` config has `auto_start_machines = true` with a concurrency-based scale trigger (`hard_limit = 250` requests), and `min_machines_running = 1` is only a floor, not a ceiling — under any load spike (including the attacker's own request flood, which is itself "load") Fly can and will start additional machines, each maintaining an independent rate-limit bucket for the same client IP. The documented brute-force-defense math in `AGENTS.md` / route comments ("10 attempts per minute per IP ... caps at ~14,400 guesses/day") implicitly assumes a single shared budget; with N autoscaled machines the real ceiling is `N × 14,400`/day. The same module-local pattern applies to `id-token` JWKS handling: `jwksCache` (1h TTL) and the A4-084 `lastInvalidatedAtMs` debounce (60s) are both per-machine, so an attacker sending unknown-`kid` social id_tokens can trigger up to N independent forced-refetch storms against Google/Apple's JWKS endpoint instead of the single fleet-wide one the A4-084 fix intended.
- Impact: doesn't break any single control outright, but quietly widens every per-IP auth defense's real-world ceiling by the (variable, load-dependent) machine count, and directly compounds AUTH-01's brute-force-evasion math. This is exactly Part 6 checklist item #33's ask ("quantify for every limiter").
- Evidence: `rate-limit.ts:36` `const rateLimitMap = new Map<...>()`; `jwks.ts:43` `const jwksCache = new Map<string, CacheEntry>()`; `jwks.ts:59` `const lastInvalidatedAtMs = new Map<string, number>()`; `fly.toml:62-71`.
- Minimal fix: document the actual ceiling as `per-route-limit × max-machine-count` in the AGENTS.md rate-limit table so the stated numbers aren't misleading; consider lowering `concurrency.hard_limit` or pinning `min_machines_running == max` for the backend if single-bucket accuracy matters more than horizontal scaling headroom.
- Better fix: move rate-limit (and ideally JWKS-cache) state to a shared store (Postgres advisory-row counters, or Redis if one gets introduced elsewhere) so the configured per-route caps are accurate regardless of fleet size — same fix class as CF-14's `FOR UPDATE SKIP LOCKED` payout-worker fix, applied to the rate limiter.

### AUTH-07 [P3 · LIVE] JWT/step-up signing-key entropy is not validated, only length

- File: `apps/backend/src/env.ts:264-299` (`LOOP_JWT_SIGNING_KEY`, `LOOP_JWT_SIGNING_KEY_PREVIOUS`, `LOOP_ADMIN_STEP_UP_SIGNING_KEY`, `LOOP_ADMIN_STEP_UP_SIGNING_KEY_PREVIOUS`)
- Description: every signing-key env var is validated with only `z.string().min(32, ...)`. A 32-character key of `'aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa'` or any other low-entropy/repeated-character string passes boot validation and would be used to HMAC-sign every access/refresh/step-up token in production.
- Impact: low practical likelihood (an operator would have to deliberately or accidentally set a degenerate secret via `flyctl secrets set`), but it's a real gap between the documented security property ("HS256 is a symmetric secret — minimum 32 bytes of entropy", `env.ts:259`) and what's actually enforced (length, not entropy). Flagged explicitly by this audit's own checklist (Part 6 §37).
- Evidence: `env.ts:264-299`.
- Minimal fix: add a cheap entropy heuristic at boot (e.g. reject keys with fewer than ~20 distinct characters, or a Shannon-entropy floor) as a sanity check, logging a boot warning (not a hard fail, to avoid false positives on legitimately-random-but-skewed secrets) when it looks suspicious.
- Better fix: document and enforce that all signing-key secrets MUST be generated via `openssl rand -base64 32` (already the documented pattern for `LOOP_REDEEM_ENCRYPTION_KEY`); add a `scripts/preflight-tranche-1.sh`-style check that fetches the live Fly secret and runs the entropy heuristic against it as part of deploy preflight, rather than only at process boot.

### AUTH-10 [P3 · LIVE] `verifyLoopToken` has zero clock-skew leeway on `exp`, unlike the social id_token verifier

- File: `apps/backend/src/auth/tokens.ts:200` (`if (obj['exp'] < Math.floor(Date.now() / 1000))`)
- Description: Loop's own access/refresh JWT verification rejects on exact `exp < now`, with no leeway. Contrast with `id-token-verify-with-key.ts` (social id_tokens), which deliberately added a 60s leeway (A2-569) specifically because Loop and the provider's clocks can't be assumed perfectly synced. For Loop-signed tokens the signer and verifier can run on different Fly machines (mint on one, verify on another, or a deploy mid-flight running mixed binaries) — multi-machine wall-clock drift, while normally sub-second with NTP, is not theoretically bounded to zero.
- Impact: very low (would only manifest as a rare false-401 right at a token's expiry boundary under clock drift, not a security weakening — tightening rather than loosening would be the wrong direction here). Noted for completeness per checklist §26 (clock-skew tolerance).
- Evidence: `tokens.ts:200` vs `id-token-verify-with-key.ts:82` (`if (exp + leeway < now)`).
- Minimal fix: none necessary; if false-401s are ever observed near token boundaries in production logs, add a small (≤5s) leeway matching typical Fly inter-machine NTP drift.
