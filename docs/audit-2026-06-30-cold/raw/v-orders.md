### ORD-001 [P1 · LIVE] `procureOne` treats every `payCtxOrder` failure as definitively-unpaid, even `PayoutSubmitError(kind='transient_horizon')` whose own type contract says "ambiguous, safe to retry"

- File: `apps/backend/src/orders/procure-one.ts:272-306` (catch block around `payCtxOrder`); `apps/backend/src/payments/payout-submit.ts:75-87` (`PayoutSubmitErrorKind`)
- Description: `payCtxOrder` (the principal-switch hop that pays CTX from the operator wallet) can throw `PayoutSubmitError` with one of: `transient_horizon` ("Network / 5xx / timeout — safe to retry on next tick"), `transient_rebuild`, or three `terminal_*` kinds. `procureOne`'s catch treats **all** of them identically — `markOrderFailed` + `return 'failed'` — with no distinction from the `terminal_*` kinds, and crucially with none of the `OperatorRateLimitedError`/`OperatorPoolUnavailableError` revert-to-`paid`-for-retry treatment that CF-12/CF-13 correctly apply one layer up (around the CTX gift-card-create call). Since `markOrderFailed` is a terminal state and `runProcurementTick` only ever selects `state='paid'` rows, an order that hits `transient_horizon` here can **never** be retried.
  This throws away the exact safety net `pay-ctx.ts` was built for: `payCtxOrder`'s own idempotency check (`findOutboundPaymentByMemo`, CF-18-widened to ~1600 records) exists precisely so a _retried_ `payCtxOrder` call can detect "we already paid this" and skip re-submitting. By failing the order outright instead of reverting `procuring → paid`, that idempotent-retry machinery is built but never exercised on this path.
  Also: when this branch fires, the CTX gift-card id obtained from the (successful) create call (`parsed.data.id`, line 260's sibling read) is **not** included in the error log (`log.error({ orderId: order.id, kind: err.kind, resultCodes: err.resultCodes }, ...)`, line ~298) and is **not** persisted to the order row (only `markOrderFulfilled` writes `ctx_order_id`). If a human needs to manually reconcile against CTX's dashboard, the CTX-side id is gone from both the DB and this log line — only recoverable via the `Idempotency-Key: order.id` header CTX itself may have stored.
- Impact: A transient Horizon hiccup (network blip, 5xx, timeout) while submitting the CTX payment — genuinely common at the tail of any third-party RPC dependency — permanently fails a real, already-paid-by-the-user order with no automatic recovery path, even though the codebase has the exact mechanism (idempotent memo-scan retry) needed to safely resolve it. Worst case: the Stellar tx actually landed (broadcast succeeded, ack timed out) — Loop has now paid CTX for an order it's telling the user failed, with no record correlating the spend, and (per ORD-002) no refund and no alert either.
- Evidence: `procure-one.ts:297-303` — the `PayoutSubmitError` branch is unconditional on `err.kind`. `procure-one-resilience.test.ts:247-269` explicitly asserts `revertToPaid` is **not** called for `PayoutSubmitError('transient_horizon', ...)`, i.e. the current "always terminal" behavior is locked in by a passing test, not an oversight that tests would catch. `payout-submit.ts:75-87` defines `transient_horizon`/`transient_rebuild` as retry-safe in their own doc comments — that contract is honored by the payout _worker_ (which does retry via the authoritative `getOutboundPaymentByTxHash` + memo-scan fallback, per CF-18) but not by `procureOne`.
- Minimal fix: in the `PayoutSubmitError` branch, special-case `kind === 'transient_horizon' || kind === 'transient_rebuild'` to call `revertOrderProcuringToPaid(order.id)` and `return 'skipped'` (mirroring the CF-12/13 treatment one layer up), leaving only the `terminal_*` kinds as immediate `markOrderFailed`. Add `ctxOrderId: parsed.data.id` to the log line unconditionally (it's already in scope at that point).
- Better fix (if different): persist a `pay_ctx_tx_hash`/`pay_ctx_ctx_order_id` column on `orders` at the moment `payCtxOrder` is about to submit (mirroring the payout-worker's CF-18 authoritative-hash pattern), so a retried `procureOne` can use `getOutboundPaymentByTxHash` instead of relying solely on the depth-limited memo scan — closing the same class of double-pay-to-CTX risk CF-18 already closed for the payout worker, but never extended to the orders principal-switch hop. Also bound the retry count (e.g. cap revert-to-paid retries per order, falling through to `markOrderFailed` + alert after N) so a permanently-misbehaving Horizon endpoint doesn't spin an order forever between `paid`/`procuring`.

### ORD-002 [P1 · LIVE] Orders failing for any reason _before_ `payCtxOrder` succeeds leave the user's already-deposited funds un-refunded with zero ops visibility

- File: `apps/backend/src/orders/procure-one.ts:214-219, 226-228, 239-244, 248-253, 276-303` (every `markOrderFailed` call site prior to `ctxPaid = true` at line 259); `apps/backend/src/admin/stuck-orders.ts:14` ("Fulfilled / failed / expired are terminal from Loop's POV; they never appear here")
- Description: `procureOne` only ever runs against orders already in `state='paid'` — meaning the user's on-chain XLM/USDC/LOOP-asset deposit has _already landed and been accepted_ by the time procurement starts (the inbound watcher gates the `pending_payment → paid` transition on a confirmed, amount-sufficient deposit). Every failure path that occurs before `payCtxOrder` returns successfully — CTX non-ok create response, CTX response schema drift, missing `paymentUrls` entry, a SEP-7 parse failure on CTX's own URI, `PayCtxConfigError` (operator secret unset), `PayCtxReconcileError` (memo-collision fail-closed), and every `PayoutSubmitError` kind (see ORD-001) — calls `markOrderFailed` and returns, with **no** call to `applyOrderAutoRefund` and **no** Discord notification of any kind (only a `log.error`).
  The only two notifiers that ever fire for a `failed` order are `notifyStuckProcurementSwept` (the _ambiguous_-outcome 15-minute timeout sweep, which has a runbook prescribing manual reconciliation) and `notifyOrderFailedAfterCtxPaid` (CF-20, fires only when `ctxPaid === true`, i.e. strictly _after_ `payCtxOrder` resolves successfully and a _later_ step then fails). Every other `failed` transition is silent.
  The only admin "needs attention" surface (`GET /api/admin/stuck-orders`) explicitly excludes terminal states by design ("Fulfilled / failed / expired are terminal from Loop's POV; they never appear here") — so there is no dashboard, alert, or runbook step that would ever surface these rows to a human. The only recourse is the customer filing a support ticket, or an operator manually running an ad-hoc `state='failed'` query.
- Impact: this is the **most common, least exotic** order-failure shape (CTX returning a clean error — out-of-stock, merchant temporarily disabled, a validation rule Loop's pre-checks don't mirror, or a CTX API contract change causing schema drift) and it is the one with the _weakest_ compensation/observability story in the whole vertical, weaker even than the genuinely-ambiguous timeout-sweep case. A paying user gets nothing (no gift card, no refund, no visible path to one) and Loop has no automatic signal that this happened. At any non-trivial CTX-side error rate this silently erodes user trust and accumulates an invisible "owed refunds" backlog.
- Evidence: `procure-one.ts:217, 227, 243, 252, 279, 288, 302` — seven distinct `markOrderFailed` call sites, none paired with a refund or notify call. `admin/stuck-orders.ts:14` confirms `failed` rows are out of scope for the only triage view. `grep` across `docs/runbooks/*.md` finds exactly one runbook touching order refunds (`stuck-procurement-swept.md`), scoped only to the timeout-sweep alert. The prior audit's P2-02 (now CF-20) only ever scoped the _post_-`ctxPaid` case — this finding is the un-addressed mirror image (and the test suite's CF-28 block, `procure-one-resilience.test.ts:240-301`, explicitly _asserts_ `applyOrderAutoRefundMock` is **not** called for any of these branches, confirming the gap is intentional-by-omission rather than accidental).
- Minimal fix: add a Discord alert (new notifier, e.g. `notifyOrderFailedNeedsReview`, modeled on `notifyStuckProcurementSwept`'s per-row embed) at every `markOrderFailed` call site in `procure-one.ts` that isn't already covered by `notifyOrderFailedAfterCtxPaid`, carrying orderId/userId/merchantId/chargeMinor/chargeCurrency/reason — restoring at least the same visibility the timeout-sweep case already has, so ops can manually refund via the existing `applyAdminRefund` API.
- Better fix (if different): classify each failure as provably-CTX-never-charged (CTX non-ok create, schema drift, missing paymentUrls, bad SEP-7, `PayCtxConfigError`, `PayCtxReconcileError` — none of these ever reach a Stellar submit) vs genuinely-ambiguous (`PayoutSubmitError('transient_horizon'/'transient_rebuild')`, see ORD-001). For the provably-unpaid set, auto-refund via `applyOrderAutoRefund` immediately (safe — CTX was never charged, no double-pay risk) and still alert for visibility. For the ambiguous set, apply ORD-001's revert-to-paid retry first; only after retries are exhausted, alert (don't auto-refund — a real CTX payment may have landed) so ops can reconcile, exactly like the timeout-sweep runbook already prescribes.

### ORD-003 [P2 · LIVE] Cross-currency FX conversion (CF-19 order path) has no sanity bound beyond `rate > 0` — a feed glitch can silently mis-price an order by orders of magnitude

- File: `apps/backend/src/payments/price-feed-fx.ts:183-200` (`usdRateFor`), `:227-268` (`convertMinorUnits`); called from `apps/backend/src/orders/loop-handler.ts:288-317` (the CF-19 FX-pin at order creation)
- Description: every loop-native order whose catalog currency differs from the user's home currency (any GBP/EUR user buying a USD-denominated card, or any ADR-035 extended-market order) gets its `chargeMinor` pinned via `convertMinorUnits`, which ultimately trusts a single third-party Frankfurter rate per currency for up to 60 seconds (process-wide shared cache). The only validation on a fetched rate is `rate <= 0` rejection (`price-feed-fx.ts:196-198`) and the upstream `FxFeedResponse` Zod shape (`base==='USD'`, `rates` is a number record). There is no check that a rate is _plausible_ relative to its known real-world band (GBP/EUR ≈ 0.5–1.5, AED ≈ 3.6, INR ≈ 80–90, SAR ≈ 3.75, MXN ≈ 17–20, AUD ≈ 1.3–1.7), and no check against the _previously cached_ rate for an implausible jump.
- Impact: a Frankfurter-side glitch (decimal-place error, base-currency confusion, a stale/corrupted snapshot — all documented real-world failure modes for third-party FX feeds) would silently produce a `chargeMinor` wrong by whatever factor the bad rate introduces, in either direction: too high (user massively overcharged for a gift card — direct consumer-harm + chargeback/reputational risk) or too low (user pays a pittance while Loop still settles the full wholesale face value to CTX via the operator wallet — direct Loop-side financial loss with no cap other than the unrelated `ORDER_MAX_FACE_VALUE_MINOR` $50k ceiling, which bounds the _catalog_-currency face value, not the post-conversion charge). Because the rate is cached process-wide for 60s, a single bad fetch can mis-price every cross-currency order placed in that window, not just one.
- Evidence: `price-feed-fx.ts:196-198` (`if (rate <= 0) throw ...` — the only guard); `:227-268` shows no jump/band check anywhere in `convertMinorUnits`'s math. This matches the audit brief's explicit Part 6 §35 ask verbatim ("sanity bounds on rate jumps... shouldn't let an order clear at a 100x-wrong rate").
- Minimal fix: add a static plausible-band check per currency in `usdRateFor` (reject + throw a typed `FxRateImplausibleError` outside e.g. `[0.3, 3.0]` for GBP/EUR, wider bands per extended currency) so a wildly-wrong rate fails closed (503) instead of silently pricing an order. Surface a Discord alert on rejection so ops knows the feed degraded rather than just seeing a spike in 503s.
- Better fix (if different): also reject on a large relative jump from the last successfully-cached rate (e.g. >15% change tick-to-tick triggers a re-fetch-and-confirm or a fail-closed + alert rather than an instant accept) — catches a glitch even if it happens to still fall within the static plausible band. Consider a secondary feed (or at minimum a staleness-aware "last known good" with an explicit alert when falling back to it) so a `convertMinorUnits` outage doesn't 503 every cross-currency order during a feed incident.

### ORD-004 [P2 · LIVE] `CtxGiftCardDetailResponse`'s strict `.url()` validation discards a valid code+PIN when only the redeem URL fails to parse

- File: `apps/backend/src/orders/procurement-redemption.ts:43-50` (schema), `:75-83` (`safeParse` → all-null on failure); contrast `apps/backend/src/orders/get-handler.ts:45` (`redeemUrl: z.string().optional()`, no `.url()`)
- Description: `CtxGiftCardDetailResponse` requires `redeemUrl`/`url` to satisfy Zod's `.url()` (strict WHATWG-absolute-URL) constraint. `fetchRedemption`'s `safeParse` call validates the **whole** response object at once — if CTX returns a perfectly good `redeemCode`/`redeemPin` alongside a `redeemUrl` that isn't a strict absolute URL (a relative path, an unusual scheme, a URL CTX itself didn't validate before sending), the entire parse fails and `fetchRedemption` returns `{code: null, pin: null, url: null}`, discarding the usable code+PIN along with the unusable URL. The sibling schema in `get-handler.ts` (used by the legacy CTX-proxy single-order GET, same upstream field) has **no** `.url()` constraint on the same field — confirming the strict constraint here is most likely an oversight, not a deliberate tightening.
- Impact: a CTX response shape that's perfectly usable for the user-facing redemption (code+PIN present) gets treated as completely empty. `waitForRedemption` then persists nulls, the order still transitions to `fulfilled` (per design), and the only recovery is the `redemption-backfill.ts` sweeper — which calls the _same_ `fetchRedemption`/schema, so it fails identically on every retry until the 10-attempt/~17h cap is hit and `notifyRedemptionBackfillExhausted` finally pages ops. A user who paid for a gift card could wait up to ~17 hours (or until a human investigates) for a code that was available on the very first fetch.
- Evidence: `procurement-redemption.ts:46,49` vs `get-handler.ts:45`. `redemption-backfill.test.ts` and `redemption.test.ts` only exercise the "fields genuinely absent" case, not "fields present but the URL field fails `.url()`" — so this gap has no regression-test coverage either.
- Minimal fix: drop `.url()` from `redeemUrl`/`url` in `CtxGiftCardDetailResponse`, matching `get-handler.ts`'s permissiveness — the WebView/redemption flow can deal with a malformed URL at render time; the backend's job here is to capture what CTX sent, not gatekeep its format.
- Better fix (if different): validate each field independently (e.g. `z.string().url().optional().catch(undefined)` per-field, or move to three separate `safeParse` calls) so a single malformed field never voids the other two — applies the same robustness CTX's own "various field aliases" tolerance already reflects for `code`/`redeemCode`.

### ORD-005 [P2 · GATED (active once `LOOP_REDEEM_ENCRYPTION_KEY` is set)] No key-rotation story for the redeem-secret envelope key

- File: `apps/backend/src/orders/redeem-crypto.ts:78-116` (`resolveRedeemKey`/`decodeKey`); contrast `LOOP_JWT_SIGNING_KEY` / `LOOP_JWT_SIGNING_KEY_PREVIOUS` rotation pattern documented in `AGENTS.md`
- Description: `resolveRedeemKey()` reads exactly one key from `LOOP_REDEEM_ENCRYPTION_KEY`, memoized for the process lifetime. Unlike the JWT signing key (which has a documented `_PREVIOUS` companion var so a rotation window can verify tokens signed under either key), there is no `LOOP_REDEEM_ENCRYPTION_KEY_PREVIOUS`, no multi-key decrypt-attempt fallback, and no backfill/re-encrypt script anywhere in `scripts/` or `tools/`.
- Impact: if the key is ever rotated (planned hygiene, or in response to a suspected leak — the exact scenario this control exists to mitigate), every row encrypted under the old key becomes permanently undecryptable the moment the new key is deployed: `decryptRedeemField` throws `RedeemDecryptError` for any `enc:v1:`-tagged value under the new key, and the read handler (`loop-read-handlers.ts:readRedeemField`) serves `null` ("redemption unavailable") for every historical order's code/PIN. There is no way to recover those values short of restoring the old key. This makes key rotation — the standard incident response to a suspected secret leak — actively destructive to historical user data, which in practice means operators will be reluctant to rotate even when they should.
- Evidence: `redeem-crypto.ts:80-90` (`resolveRedeemKey`, single key source, no fallback list); repo-wide `grep` for `LOOP_REDEEM_ENCRYPTION_KEY` finds zero hits outside `env.ts`/`redeem-crypto.ts`/its own tests — no rotation runbook, no re-encrypt CLI.
- Minimal fix: add `LOOP_REDEEM_ENCRYPTION_KEY_PREVIOUS` (mirroring the JWT pattern) and have `decryptRedeemField` try the current key, then the previous key, before failing — giving operators a rotation window during which both keys decrypt.
- Better fix (if different): ship a one-shot `scripts/rotate-redeem-key.mjs` that reads every `enc:v1:` row under the previous key and re-writes it under the new key (batched, idempotent, resumable — same posture as the other operator scripts in this repo), then document the rotation procedure in a runbook. This removes the need to keep two keys live indefinitely and closes the gap properly rather than just widening the decrypt window. (Minor, same area: `isEncryptedRedeemField` matches on a literal `enc:v1:` prefix with no escaping — a legacy plaintext redeem code that coincidentally started with that exact string would be misidentified as ciphertext and fail to decrypt. Vanishingly unlikely for real gift-card codes, but worth a one-line comment acknowledging the edge case.)

### ORD-006 [P3 · LIVE] Legacy `POST /api/orders` still doesn't reject disabled merchants (prior audit P3-01 — confirmed still open)

- File: `apps/backend/src/orders/handler.ts:47-51`
- Description: `createOrderHandler` only checks `merchant === undefined` (404 unknown merchant); it does not check `merchant.enabled`. The loop-native handler correctly rejects `enabled === false` (`loop-handler.ts:220`). Independently re-discovered during this cold read before consulting the prior audit; confirmed unchanged since 06-15's P3-01.

- Minimal fix: add `|| merchant.enabled === false` to the legacy reject (404, to avoid leaking disabled-merchant existence), matching loop-handler.
- Better fix (if different): same as minimal — this is a one-line consistency fix, no design decision needed.

### ORD-007 [P3 · LIVE] `loopListOrdersHandler` pagination has no tiebreaker on `createdAt` ties (prior audit P3-03 — confirmed still open)

- File: `apps/backend/src/orders/loop-read-handlers.ts:198-209`
- Description: the list query orders by `desc(orders.createdAt)` only and pages via `lt(orders.createdAt, beforeDate)`. Two rows sharing the exact same `createdAt` (timestamptz; plausible under burst order creation or an idempotent-replay double-submit landing in the same instant) would have one silently skipped by a client paging with the tied timestamp as `before` — strict `lt` excludes both ties as soon as the first is seen.
- Impact: low — bounded to same-instant ties, owner-scoped read only, no cross-user leak — but a silent data-completeness gap in a paginated financial list (a user's order history could appear to be missing a row while paging).
- Evidence: `loop-read-handlers.ts:205` (`lt(orders.createdAt, beforeDate)`), `:208` (`orderBy(desc(orders.createdAt))`) — no secondary key anywhere in the query.
- Minimal fix: accept as a documented known limitation (probability is low and impact is cosmetic) if not worth the migration.
- Better fix (if different): add `id` as a tiebreaker — keyset pagination on `(createdAt, id)` for both the `ORDER BY` and the cursor predicate (`(createdAt, id) < (beforeDate, beforeId)`), requiring the client to also round-trip the last row's `id`.

### ORD-009 [P3 · LIVE/GATED] In-process Discord dedup/throttle state is per-Fly-machine, not shared — duplicate notifications once the backend scales past one machine

- File: `apps/backend/src/orders/get-handler.ts:84-93` (`notifiedFulfilled` Set), `apps/backend/src/orders/procurement-asset-picker.ts:36-42` (`lastBelowFloorAlertAt`)
- Description: both dedup structures are module-level mutable state, scoped to a single Node process. `apps/backend/fly.toml` sets `min_machines_running = 1` but also `auto_start_machines = true` with request-concurrency-based autoscaling (`hard_limit = 250`), so the backend can and will run more than one machine under load. A user's `GET /api/orders/:id` polls can land on different machines (no session affinity declared), and the procurement worker itself could run on multiple machines per CF-14's own cross-instance-safety reasoning elsewhere in this vertical.
- Impact: with N machines live, a single order's "Order Fulfilled" Discord notification could fire up to N times (once per machine whose `notifiedFulfilled` Set hasn't seen that order id yet), and the USDC-below-floor alert's 15-minute cooldown is enforced independently per machine rather than fleet-wide. Pure alert-noise / Discord-channel-signal-quality issue, not a money or correctness bug — explicitly called out as a check in this audit round's Part 6 §33.
- Evidence: `get-handler.ts:84` (`const notifiedFulfilled = new Set<string>();`), `procurement-asset-picker.ts:37` (`let lastBelowFloorAlertAt = 0;`) — both plain in-memory, no shared store (Postgres/Redis) backing either. `fly.toml:60-68` confirms autoscaling is configured beyond the floor of 1.
- Minimal fix: accept as low-priority noise given current traffic volume (Phase 1); revisit if/when `min_machines_running` is raised or sustained autoscaling beyond 1 machine is observed.
- Better fix (if different): move both dedup keys into Postgres (a small `notified_fulfilled_orders` table with a unique constraint, or reuse the existing `pendingPayouts.orderId`-style `onConflictDoNothing` pattern; the below-floor throttle could be a single advisory-locked row with a `last_alert_at` timestamp) so the dedup/cooldown window is accurate fleet-wide regardless of machine count.

### ORD-010 [P3 · LIVE] `Idempotency-Key` replay does not verify the replayed request matches the original

- File: `apps/backend/src/orders/loop-handler.ts:178-200` (lookup-first short-circuit), `apps/backend/src/orders/repo-idempotency.ts:54-62` (`findOrderByIdempotencyKey`)
- Description: a repeat `POST /api/orders/loop` with a previously-used `Idempotency-Key` (scoped to `(userId, key)`) returns the _original_ order's response unconditionally — there is no check that the new request's `merchantId`/`amountMinor`/`currency`/`paymentMethod` match what was originally submitted under that key.
- Impact: low — scoped to a single user's own bearer, so this is not cross-user/IDOR — but it diverges from the stricter idempotency-key contract used elsewhere (e.g. Stripe-style implementations reject a key reused with a different request body as a 422/409 conflict). A client bug that reuses a stale key after the user changed the amount/merchant in the UI would silently return the _original_ order rather than surfacing the mismatch, which could confuse both the client and a support investigation ("why did my $50 order get created when I changed it to $30").
- Evidence: `loop-handler.ts:196-199` — `findOrderByIdempotencyKey(auth.userId, idempotencyKey)` ignores the rest of `parsed.data` entirely when a prior row exists.
- Minimal fix: when a prior order is found for the key, compare `merchantId`/`amountMinor`/`currency`/`paymentMethod` against the request; on mismatch return a 409 `IDEMPOTENCY_KEY_CONFLICT` instead of silently replaying.
- Better fix (if different): same as minimal, plus document the comparison fields in the OpenAPI description (`openapi/orders-loop.ts:129`) so client authors know reuse-with-different-body is rejected, not silently coerced.

### ORD-011 [P3 · LIVE] Fail-open zero-amount SEP-7 fallback when the XLM/USDC oracle is unavailable at order-creation/replay time

- File: `apps/backend/src/orders/loop-create-response.ts:160-170`, `apps/backend/src/orders/loop-replay-response.ts:123-136`
- Description: when `xlmAmountFor`/`usdcAmountFor` throws (oracle unreachable), both the create and replay response builders catch the error and substitute `{ stroops: 0n, formatted: '0.0000000' }`, then build a SEP-7 payment URI advertising a **zero** payment amount, returned to the client with a `200`. The comment frames this as intentional fail-open ("the watcher's amount validation will still hold").
- Impact: not a money-loss issue (a zero-amount payment can never satisfy the watcher's amount-sufficiency check, so the order simply can't be paid this way until the oracle recovers) but it is a confusing-UX / weak-resilience choice: the client renders a payment screen asking the user to send `0.0000000 XLM`/USDC, which most wallets will refuse to build a transaction for, leaving the user stuck with no actionable error message and no retry guidance. A clean `503 SERVICE_UNAVAILABLE` (consistent with the deposit-address-unset / FX-feed-outage failure modes the same handler already uses elsewhere) would let the client show a proper "try again shortly" state instead.
- Evidence: `loop-create-response.ts:160-169` (`catch (err) { ...; assetAmount = { stroops: 0n, formatted: '0.0000000' }; }`), identical pattern in `loop-replay-response.ts:130-136`.
- Minimal fix: return `503 SERVICE_UNAVAILABLE` (matching the sibling failure modes already in these same functions) instead of synthesizing a zero-amount payment instruction.
- Better fix (if different): same as minimal — there's no UX or correctness benefit to the current fail-open shape since the order can't be paid at zero anyway; failing closed is strictly better here for both the user (clear error) and Loop (no orphaned, perpetually-unpayable `pending_payment` rows accumulating in the 24h expiry window for no reason).

### ORD-012 [P3 · n/a] Minor REST-convention inconsistency: legacy create returns `201`, loop-native create returns `200`

- File: `apps/backend/src/orders/handler.ts:147-157` (`201`); `apps/backend/src/orders/loop-handler.ts:399-405` / `apps/backend/src/orders/loop-create-response.ts` (all `c.json(...)` calls default to `200`)
