## Findings

### PLAT-30-01 [P1 · LIVE] In-memory per-machine rate limiting silently multiplies every configured cap by the live machine count

- File: `apps/backend/src/middleware/rate-limit.ts:39` (`rateLimitMap`), `apps/backend/fly.toml:55-66` (`auto_start_machines=true`, `min_machines_running=1`, `concurrency.soft_limit=200`/`hard_limit=250`)
- Description: `rateLimitMap` is a module-level `Map`, meaning each Fly machine running the backend image maintains a fully independent counter for every `(routeName, ip)` key. Loop's own rate-limit design (A4-001, documented in the file header) assumes the configured `maxRequests` is _the_ effective per-IP-per-route budget. It is not — it is the effective budget _per machine_. `fly.toml` does not pin the backend to a single machine: `min_machines_running = 1` is a floor, not a ceiling, and `auto_start_machines = true` plus `[http_service.concurrency]` (`soft_limit = 200`, `hard_limit = 250`) means Fly will start additional machines once a running machine's concurrent in-flight requests cross the soft limit. This is not theoretical: `flyctl machines list -a loopfinance-api` (run live during this audit) shows **2 machines already provisioned** for the app (`autumn-thunder-2616` stopped, `nameless-fog-5019` started) — i.e. the fleet is already sized for ≥2 well before any abuse scenario.
- Impact: Every documented per-route limit is actually `limit × N` where N is whatever the Fly scheduler currently has running, which the limiter itself cannot see or account for. The two highest-value targets: (1) `POST /api/auth/request-otp` is rate-limited at 5/min specifically as an OTP-spam / email-bombing / enumeration defense — at N=2 that's already 10/min, and Fly's autoscaler does not require sticky sessions, so an attacker spraying parallel connections has no guarantee of hitting the same machine twice. (2) `POST /api/orders` + `POST /api/orders/loop` at 10/min is the primary flat defense the checklist's own §38 (fraud/abuse) calls out for a gift-card cash-out vector — also silently doubled-or-more. The mechanism is self-defeating under a genuine attack: a high-volume abuse burst is exactly the condition that pushes a machine's concurrency past `soft_limit=200` and causes Fly to start more machines, which _further_ loosens the per-IP cap precisely while the attack is in progress.
- Evidence:
  ```
  $ flyctl machines list -a loopfinance-api
  loopfinance-api
   ID             NAME                 STATE    REGION  ...
   48e1342a1e7918 autumn-thunder-2616  stopped  iad     ...
   7846645ad24268 nameless-fog-5019    started  iad     ...
  ```
  `apps/backend/fly.toml`:
  ```
  auto_stop_machines = "stop"
  auto_start_machines = true
  min_machines_running = 1
    [http_service.concurrency]
      type = "requests"
      hard_limit = 250
      soft_limit = 200
  ```
  `rateLimitMap` is declared at module scope in `rate-limit.ts:39` with no external store.
- Minimal fix: Document the real effective ceiling (limit × current machine count) in `AGENTS.md`'s rate-limit table and the on-call runbook so an incident responder isn't surprised mid-incident; as a stronger stopgap, lower the per-route `maxRequests` values to budget for the realistic max fleet size (e.g. configure as if N=3–5), or pin the deployment to a single machine (`auto_start_machines=false` / remove the concurrency autoscale block) if the simplicity of a correct single-instance limiter is worth the availability trade-off for Phase 1's traffic level.
- Better fix: Move the rate-limit counter to a store shared across machines. Given the project's own architecture (Postgres is already a hard dependency, ADR 012; no Redis/Upstash dependency exists anywhere in the repo today and adding one would need its own ADR per the project's dependency policy), the natural fit is a small Postgres-backed fixed-window counter table — e.g. `rate_limit_buckets(name, ip, window_start, count)` with an atomic `INSERT ... ON CONFLICT (name, ip, window_start) DO UPDATE SET count = count + 1 RETURNING count`, checked against `maxRequests`, reaped by the existing hourly-cleanup pattern. This mirrors the `FOR UPDATE SKIP LOCKED` / CAS coordination this codebase already trusts for payout-worker and procurement-worker cross-instance safety, and needs no new infrastructure. Trade-off: a DB round-trip on every rate-limited request (acceptable on `/api/auth/*` and `/api/orders*`'s low caps; would need to be cheap/indexed for `/api/clusters` at 60/min and `/api/image` at 300/min — consider a hybrid where each machine batches local counts and flushes to Postgres every 1-2s, bounding the cross-machine inaccuracy window instead of eliminating sync overhead entirely).

### PLAT-30-02 [P2 · LIVE] Circuit-breaker state is also per-machine — an upstream incident is only half-contained

- File: `apps/backend/src/circuit-breaker-registry.ts:21` (`circuitsByKey` module-level Map), `apps/backend/src/circuit-breaker.ts` (state machine instantiated per key, per process)
- Description: Same class of issue as PLAT-30-01 but for circuit breakers rather than the rate limiter. `getUpstreamCircuit(key)` lazily creates and caches a `CircuitBreaker` per process. With N machines, each upstream category (`login`, `merchants`, `gift-cards`, the Horizon-call sites, etc.) effectively has N independent breakers. If one machine accumulates 5 consecutive failures against a struggling upstream and opens its breaker (or CF-13's `forceOpen()` trips on a dead operator credential), sibling machines that haven't yet seen 5 failures of their own keep sending live traffic to the same unhealthy upstream at full cadence.
- Impact: The breaker's stated purpose — stop hammering a failing dependency — is only achieved at `1/N` of the fleet's request volume during an incident, which both prolongs the upstream's recovery and means each machine independently re-discovers the failure (and re-fires its own Discord circuit-open notification — `notifyCircuitBreaker` dedups per-key but the key namespace is also per-process unless the notifier's dedup store is itself shared; not independently verified here, flag for the observability vertical).
- Evidence: `circuit-breaker-registry.ts:21,30-37` (`const circuitsByKey = new Map(); ... if (cb === undefined) { cb = createCircuitBreaker(...); circuitsByKey.set(key, cb); }`) — no external state, no shared store.
- Minimal fix: Document the per-machine scope of breaker state next to the existing "Independent so a failing `/locations` doesn't trip auth" doc claim in `AGENTS.md`'s middleware-stack section, so operators reading `/metrics`' `loop_circuit_state` gauge during an incident know it reflects one machine only.
- Better fix: Lower priority than the rate-limiter fix (a circuit breaker's failure mode here is "slower fleet-wide containment," not "security control silently weakened") — if pursued, the same Postgres-backed shared-counter approach from PLAT-30-01 could carry a `circuit_state` row per upstream key with a short TTL/heartbeat, letting one machine's OPEN decision propagate to siblings on their next check. Given the modest payoff vs. added complexity, this is reasonable to defer behind PLAT-30-01.

### PLAT-30-03 [P1 · LIVE] DSR self-delete has no precondition on a non-zero off-chain cashback balance — money becomes permanently unreachable

_(Surfaced by the users-_.ts openapi sub-agent; included here because it's a genuine P1-adjacent platform/ledger-integrity finding worth flagging at top severity even though the file itself, `users/dsr-delete.ts`, sits in the V8/credits vertical's primary territory.)\*

- File: `apps/backend/src/users/dsr-delete.ts:115-183` (handler: `apps/backend/src/users/dsr-handler.ts:92-147`, route: `apps/backend/src/routes/users.ts:114-118`)
- Description: `deleteUserViaAnonymisation` blocks self-delete on three preconditions — in-flight `pending_payouts`, `failed_uncompensated_withdrawals`, in-flight `orders` — but never checks `user_credits.balanceMinor`. Cashback is credited to `user_credits` on every fulfilled order regardless of whether the user has ever linked a Stellar wallet (`orders/fulfillment.ts:114-130`); a `pending_payouts` row only exists when a wallet address was on file at fulfillment time. A user who never links a wallet accumulates 100% of their cashback as a bare `user_credits` balance with zero `pending_payouts` rows ever created, so the existing preconditions are trivially satisfied regardless of balance size.
- Impact: A user (or anyone holding a leaked/stolen access token) can self-delete via `POST /api/users/me/dsr/delete` with an arbitrary unredeemed cashback balance still on the books. Anonymisation nulls `email`/`stellarAddress`/`ctxUserId` and revokes refresh tokens; there is no self-serve withdrawal path (withdrawals are admin-only, `credits/withdrawals.ts`), so the balance becomes permanently unreachable — yet `liabilities.ts`'s `sumOutstandingLiability` keeps counting it as outstanding company liability forever. This is the same risk class the team already guarded against for `pending_payouts`/`failed_uncompensated_withdrawals` but missed for at-rest credit. No test exercises a non-zero-balance DSR delete; the privacy policy / deletion-confirmation UI never discloses balance forfeiture.
- Evidence: `dsr-delete.ts` has zero references to `userCredits`/`balanceMinor` (confirmed via grep), vs. the analogous admin-mediated `home-currency-change.ts:113-128` which explicitly blocks on `liveBalance.balanceMinor !== 0n`.
- Minimal fix: Add a fourth precondition mirroring `home-currency-change.ts`'s pattern — block with a typed `blockedBy: 'non_zero_credit_balance'` 409 when any `user_credits` row has `balanceMinor !== 0n`; surface in `dsr-handler.ts` and the `users-dsr-orders.ts` openapi registration.
- Better fix: Decide and implement an explicit policy — either require zero balance before self-delete (spend it or have support withdraw it first), or build a real "re-link on re-signup" recovery path — and disclose the chosen policy in the privacy policy and deletion-confirmation UI.

### PLAT-30-04 [P2 · LIVE] `env.test.ts` has zero coverage for any of the four new env vars added in this delta, including the new boot-time encryption-key length validator

- File: `apps/backend/src/env.ts:786-805` (new `LOOP_REDEEM_ENCRYPTION_KEY` 32-byte boot validator), `apps/backend/src/__tests__/env.test.ts` (305 lines, no reference to `ADMIN_EMAILS`, `LOOP_REDEEM_ENCRYPTION_KEY`, `LOOP_AUTH_ROW_PURGE_INTERVAL_HOURS`, or `LOOP_AUTH_ROW_RETENTION_DAYS` anywhere)
- Description: This delta added 4 new env vars to `env.ts` (`ADMIN_EMAILS`, `LOOP_REDEEM_ENCRYPTION_KEY`, `LOOP_AUTH_ROW_PURGE_INTERVAL_HOURS`, `LOOP_AUTH_ROW_RETENTION_DAYS`) plus a new ~20-line boot-time validator that decodes `LOOP_REDEEM_ENCRYPTION_KEY` (hex-or-base64 sniff, then exact-32-bytes check) and throws a descriptive error if the key is the wrong length. None of the four vars, and specifically not this new validator, appear anywhere in `env.test.ts`. There IS a test for logically-identical decode logic — `orders/__tests__/redeem-crypto.test.ts:183-185` ("throws when the key does not decode to 32 bytes") — but that exercises a _separate_, intentionally-duplicated function (`orders/redeem-crypto.ts`'s `decodeKey`, whose own comment says "env.ts already validated... but guard anyway for the direct-call path"). The boot-time `parseEnv` path itself — i.e., does the application actually refuse to start when this env var is malformed — is untested.
- Impact: A regression in the new boot validator (e.g., a typo in the hex-detection regex, an off-by-one in the length check, or the validator accidentally being skipped under some `NODE_ENV`) would not be caught by CI; the only safety net is the separately-implemented, separately-tested twin in `redeem-crypto.ts`, which only runs at first encrypt/decrypt call, not at boot. Given this var protects spendable gift-card bearer secrets at rest (CF-25), the boot-fail-fast guarantee is exactly the kind of security-relevant behavior this audit's dimension §12 ("coverage of security controls") flags as needing direct tests, not indirect/coincidental coverage via a sibling module.
- Evidence: `grep -n "ADMIN_EMAILS\|LOOP_REDEEM_ENCRYPTION_KEY\|LOOP_AUTH_ROW_PURGE\|LOOP_AUTH_ROW_RETENTION" apps/backend/src/__tests__/env.test.ts` → zero matches.
- Minimal fix: Add a `parseEnv`-level test asserting a wrong-length `LOOP_REDEEM_ENCRYPTION_KEY` throws at parse time (mirroring the existing `LOOP_STELLAR_USDC_ISSUER mainnet tripwire` / `DISABLE_RATE_LIMITING production guard` `describe` blocks already in the file), plus basic default/coercion tests for the other three new vars.
- Better fix: Same, plus consider having `orders/redeem-crypto.ts`'s `decodeKey` import/reuse the exact validation routine from `env.ts` (or a shared helper) instead of an independently-maintained copy of the same regex + decode logic, so the two can't silently drift.

### PLAT-30-06 [P2 · LIVE] `probeGateAllows` (the `/metrics` + `/openapi.json` bearer gate) has zero direct test coverage, and the only indirect coverage never exercises the production-closed branch

### PLAT-30-14 [P2 · LIVE, carried forward + severity raised, code unchanged] Image proxy `mode=private` lets any unauthenticated caller deterministically defeat the LRU cache and force CPU-bound re-encode work — and this compounds with the per-machine rate-limit multiplier (PLAT-30-01)

- Description: Any caller — no auth required, `/api/image` is a public route — can append `?mode=private` to force a cold fetch + full `sharp` decode/resize/re-encode on every single request, with both the cache read and the cache write skipped. ### PLAT-30-27 [P1 · LIVE] `POST /api/admin/users/{userId}/withdrawals` — the OpenAPI contract entirely omits the ADR-028 step-up gate that actually protects Loop's highest-risk admin write

- File: `apps/backend/src/openapi/admin-withdrawal-write.ts:104-115` (request headers), `:126-129` (401 description), `:148-152` (503 description); route: `apps/backend/src/routes/admin-credit-writes.ts:75-81`
- Description: The route mount wraps `adminWithdrawalHandler` in both `killSwitch('withdrawals')` and `requireAdminStepUp('withdrawal')` — i.e. issuing a real outbound Stellar payout from this endpoint genuinely requires a valid `X-Admin-Step-Up` token in addition to the bearer (ADR 028 / CF-07's entire point: "a captured bearer alone must not be able to issue an outbound on-chain payout"). The OpenAPI registration for this exact endpoint never declares `x-admin-step-up` as a request header, its `401` description only says "Missing or invalid bearer" (no mention of `STEP_UP_REQUIRED`/`STEP_UP_INVALID`/`STEP_UP_SUBJECT_MISMATCH`), and its `503` description only covers `NOT_CONFIGURED` (LOOP issuer missing), not `STEP_UP_UNAVAILABLE`. Every sibling money-write registration does this correctly — `admin-credit-writes.ts` (credit-adjust/refund), `admin-payouts-cluster-writes.ts` (payout-retry/compensate), `admin-user-writes.ts` (home-currency) — confirmed via grep that `admin-withdrawal-write.ts` is the only one of the four ADR-028-gated write files with zero occurrences of `step-up`/`STEP_UP` anywhere in the file.
- Impact: This is the single highest-risk write in the entire admin surface — it debits a user's off-chain cashback ledger and queues a real outbound Stellar payout to an admin-supplied destination address. The published API contract for this exact endpoint currently reads as bearer-token-only, which directly misrepresents the security control that's the whole reason ADR 028 exists. Anyone relying on the OpenAPI spec for a security review, a generated client, or an automated LLM-assisted PR review (per ADR 025) would conclude no second factor is required, when one is strictly enforced server-side. (The server-side enforcement itself is correct — this is a documentation/contract-integrity gap, not an auth bypass — but it's exactly the class of gap that turns into a real bypass if anyone ever builds tooling that trusts the spec over the code.)
- Evidence: `apps/backend/src/routes/admin-credit-writes.ts:75-81` mounts `requireAdminStepUp('withdrawal')`; `grep -n "step-up\|STEP_UP" apps/backend/src/openapi/admin-withdrawal-write.ts` → no matches, vs. the same grep against `admin-credit-writes.ts`/`admin-payouts-cluster-writes.ts`/`admin-user-writes.ts` → multiple matches each.
- Minimal fix: Add `'x-admin-step-up': z.string()` to the registration's `headers` object, extend the `401` description to cover the step-up-invalid cases, and extend `503` to also cover `STEP_UP_UNAVAILABLE` — copy the pattern verbatim from `admin-credit-writes.ts`'s refund registration.
- Better fix: Same as minimal, plus extend the `503` description to also name the `withdrawals` kill switch (`SUBSYSTEM_DISABLED`) as a third cause (PLAT-30-28 below covers this same gap for the sibling compensate endpoint) — this route sits behind both `killSwitch('withdrawals')` and step-up, and the spec should enumerate all three 503 causes, not just one.

### PLAT-30-28 [P2 · LIVE] Several admin-write/admin-read gaps the OpenAPI-batch sub-agents found beyond the automated parity script's coverage
