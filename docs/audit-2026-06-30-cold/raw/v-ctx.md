### CTX-01 [P0 · LIVE] Operator circuit breaker can never self-heal — `pickHealthyOperator` permanently strands any operator it opens, defeating CF-13's "rotation path"

- File: `apps/backend/src/ctx/operator-pool.ts:226-237` (`pickHealthyOperator`), `apps/backend/src/ctx/operator-pool.ts:364` (`forceOpen()` call site), `apps/backend/src/circuit-breaker.ts:134-142` (the only OPEN→HALF_OPEN transition), `apps/backend/src/circuit-breaker.ts:207-209` (`getState()` is a pure read)
- Description: `pickHealthyOperator()` selects the next operator in round-robin order but **filters out any operator whose `op.breaker.getState() === 'open'`** before ever calling `op.breaker.fetch()` on it again (line 231: `if (op !== undefined && op.breaker.getState() !== 'open')`). The _only_ code path that transitions a breaker from `open` back to `half_open` is the cooldown check at the top of `wrappedFetch` in `circuit-breaker.ts:136-141` (`if (Date.now() - openedAt >= cooldownMs) transitionTo('half_open')`) — and that check only runs _inside_ a `.fetch()` call. `getState()` (`circuit-breaker.ts:207-209`) is a bare getter with zero side effects; it never advances the state machine. Because `pickHealthyOperator` is the _only_ gate that decides whether `.fetch()` is ever invoked on a given operator again, once a breaker opens — whether via 5 consecutive real failures, or via CF-13's `forceOpen()` on a single 401 (`operator-pool.ts:364`) — that operator is excluded from every future `pickHealthyOperator()` call for the remaining lifetime of the process. The cooldown-based recovery the breaker implements (and the module's own docstring promises: `operator-pool.ts:13-16`, "an OPEN operator is skipped until its cooldown elapses") is structurally unreachable through this call path. `pickOperatorCredentials()` (`operator-pool.ts:208-217`, used by the SSE stream) shares the same `pickHealthyOperator()` and the same bug.
- Impact: CF-13 made this dramatically easier to trigger — pre-CF-13 an operator needed 5 _consecutive_ failed fetches before opening; post-CF-13 a single CTX 401 (even a spurious/transient one — e.g. a load-balancer edge serving a stale auth check, a brief CTX-side auth-service blip) is sufficient via `forceOpen()`. In a typical 2-operator pool (the shape every test in `operator-pool.test.ts` exercises), one bad 401 on operator A permanently routes 100% of procurement traffic to operator B for the rest of the process's life; a second independent bad response on B (a 5xx burst is enough — CF-12 also credits 429s toward the same threshold, see CTX-02) **permanently bricks the entire operator pool** — every `operatorFetch` call throws `OperatorPoolUnavailableError`/`OperatorRateLimitedError` forever, even though CTX itself may have recovered seconds later. This degrades the documented "transient → caller defers" framing (`operator-pool.ts:267-269`, `:271-275`) into a real, manual-intervention-only outage of the entire loop-native purchase flow. `health.ts:142-176`'s `/health` probe deliberately bypasses circuit breakers (its own comment: "Deliberately bare `fetch`, NOT `getUpstreamCircuit('status').fetch`") and never inspects `getOperatorHealth()`, so **Fly's health check cannot detect or restart a machine stuck in this state** (`apps/backend/fly.toml:71-79`) — recovery requires a human to notice the Discord alert / `admin/treasury.ts:100`'s `operators` snapshot and manually redeploy or restart, which is the _only_ thing that resets the in-memory `operators` array (`ensureInitialised()`, `operator-pool.ts:137-177`, only ever runs its parse-and-build step once per process).
- Evidence: Traced by hand and confirmed against both test suites. `operator-pool.test.ts` cannot catch this because it fully replaces `createCircuitBreaker` with a stub (`vi.mock('../../circuit-breaker.js', ...)`, lines 39-68 of the test file) whose `getState`/`fetch`/`forceOpen` have no time-based recovery logic at all — the integration between the pool's selection filter and the breaker's real cooldown timer is never exercised. `circuit-breaker.test.ts` proves recovery works (`'recovers via a HALF_OPEN probe after the cooldown elapses'`, lines 492-501) but only by calling `cb.fetch(...)` _directly_ and _unconditionally_ after the cooldown — exactly what every other consumer in the codebase does (`auth/handler.ts`, `merchants/sync.ts`, `images/proxy.ts`, `clustering/data-store.ts` all call `getUpstreamCircuit(key).fetch(...)` with no `getState()` pre-filter — confirmed via `grep -rn "getState()" apps/backend/src`, which returns only `circuit-breaker-registry.ts:46` (read-only `/metrics` snapshot) and the two `operator-pool.ts` sites). Operator-pool.ts is the _only_ breaker consumer in the repo that pre-filters by `getState()`, and it is therefore the only place this defect can manifest.
- Minimal fix: Ship an immediate operator stop-gap so ops has a manual lever without a full fix: export `resetOperatorBreaker(id: string): boolean` from `operator-pool.ts` (it's a one-line wrapper around the already-existing-but-unexposed `breaker.reset()`, `circuit-breaker.ts:211-217`/`237`), wire it behind a new or existing step-up-gated admin action so a stuck operator can be un-stranded without a redeploy while the root-cause fix below ships.
- Better fix (current practice, root cause): Add a non-mutating `isAvailable(): boolean` to the `CircuitBreaker` interface in `circuit-breaker.ts` that mirrors the existing cooldown arithmetic already computed in `wrappedFetch` (`state !== 'open' || Date.now() - openedAt >= cooldownMs`), and change `pickHealthyOperator()` (`operator-pool.ts:231`) to filter on `op.breaker.isAvailable()` instead of `op.breaker.getState() !== 'open'`. This requires zero changes to the actual state-machine mutation logic (the half-open single-probe-in-flight guard and the probe-timeout failsafe keep governing the real transition once `.fetch()` is finally called), it makes operator-pool.ts behave like every other breaker consumer in the codebase, and it makes the docstring's "skipped until its cooldown elapses" claim true. Pair with a new integration test that uses the _real_ `createCircuitBreaker` (not the mocked one) to assert an operator forced open via a 401 actually becomes selectable again after `cooldownMs` elapses — this is the test gap that let the bug ship silently across two PRs (#1364 SSE-first redemption, #1444 CF-12/13).

### CTX-02 [P1 · LIVE] CF-12's `Retry-After` is parsed correctly but never enforced — no real backoff, only "don't fail the order"

- File: `apps/backend/src/ctx/operator-pool.ts:69-77` (`OperatorRateLimitedError`), `apps/backend/src/orders/procure-one.ts:354-361` (catch handler), `apps/backend/src/orders/transitions.ts:202-213` (`revertOrderProcuringToPaid`, no delay parameter), `apps/backend/src/env.ts:599` (`LOOP_PROCUREMENT_INTERVAL_SECONDS` default `5`)
- Description: `parseRetryAfterMs` (`operator-pool.ts:88-103`) is a careful, RFC-9110-correct implementation (delta-seconds + HTTP-date forms, negative-clamp, 5-minute cap) and the parsed value is threaded onto `OperatorRateLimitedError.retryAfterMs`. But the only consumer, `procure-one.ts:354-361`, does this with it: `log.warn({ orderId: order.id, retryAfterMs: err.retryAfterMs }, ...)` and then `await revertOrderProcuringToPaid(order.id)` — a plain state-flip back to `paid` with **no delay, no "not-before" timestamp persisted anywhere**. The next `runProcurementTick` (`orders/procurement.ts:50-73`) re-selects oldest-`paid`-first with no awareness of any back-off window, and that tick fires every `LOOP_PROCUREMENT_INTERVAL_SECONDS` — **5 seconds by default** (`env.ts:599`). So a CTX-requested back-off of, say, 60–300 seconds (the realistic range for a real rate-limit event; `RETRY_AFTER_MAX_MS` caps parsing at 5 minutes) is not honored at all: the same order, and every other order in the paid queue, gets re-attempted roughly every 5 seconds regardless of what CTX asked for.
- Impact: This is precisely the "self-sustaining hot loop" failure mode the code's own comments say CF-12 prevents (`operator-pool.ts:271-275`: "a 429 storm... degrades into a self-sustaining hot loop that fails real paid orders... rather than marking the order failed and re-hammering CTX at full cadence" — the order-failure half is fixed, the re-hammering half is not). The only actual throttle that engages is the generic circuit-breaker mechanism: 429s count toward `consecutiveFailures` (`circuit-breaker.ts:193`, correctly not treated as success), and after 5 consecutive 429s an operator's breaker opens with a **fixed 30s default `cooldownMs`** — unrelated to CTX's actual requested `Retry-After` window, and (per CTX-01) that cooldown can never even elapse in practice through `pickHealthyOperator`'s selection path, so in the worst case the "backoff" is actually permanent exclusion, not a 30s pause. With up to `limit ?? 10` orders processed sequentially per tick (`procurement.ts:53,68-71`), a sustained CTX 429 storm could push 10 orders × 2 operator attempts = up to 20 requests against CTX every 5 seconds across the paid-order backlog, compounding whatever rate-limit condition triggered the 429s in the first place and accelerating CTX-01's pool-bricking scenario.
- Evidence: `grep -n "retryAfterMs" apps/backend/src/orders/procure-one.ts` → appears only inside a `log.warn` call; `transitions.ts:202-213`'s `revertOrderProcuringToPaid(orderId: string)` signature takes no second argument; no `next_attempt_at` / backoff column exists on the `orders` table for this purpose (confirmed by reading the full `revertOrderProcuringToPaid` body — it's a 4-field `UPDATE`).
- Minimal fix: Track a single module-level (or Discord-alert-adjacent) "global CTX backoff until" timestamp in `procure-one.ts`/`procurement.ts`, set to `Date.now() + err.retryAfterMs` on `OperatorRateLimitedError`, and have `runProcurementTick` skip picking any orders (or `procureOne` no-op and re-skip) until that timestamp passes. Cheap, in-process, good enough for a single-machine deploy.
- Better fix: Push the back-off into the operator-pool itself, parallel to the breaker: track a per-operator `backoffUntilMs` set from the parsed `Retry-After` on a 429, and have `pickHealthyOperator` skip an operator until both its breaker is non-open _and_ `Date.now() >= backoffUntilMs`. This converges every caller (procurement, redemption polling, any future CTX-proxy path) toward CTX's actual requested window without needing per-order persistence, and composes cleanly with the CTX-01 fix (`isAvailable()` can fold in the same check).

### CTX-03 [P2 · LIVE] All operator-pool state is in-process — inconsistent across Fly machines, and CTX-01's stranding bug becomes a per-machine landmine

- File: `apps/backend/src/ctx/operator-pool.ts:105-107` (`operators`, `nextIndex`, `initialised` are module-level `let`s), `apps/backend/src/ctx/operator-pool.ts:117` (`lastExhaustedAlertAt`), `apps/backend/src/discord/monitoring.ts:476` (`operatorCredentialLastNotified` Map), `apps/backend/fly.toml:63-65` (`auto_start_machines = true`, `min_machines_running = 1`)
- Description: Operator list, round-robin pointer, per-operator breaker state, the pool-exhausted alert throttle, and the per-operator-credential alert dedup map are all plain in-process module state — nothing is shared across Fly machines. `fly.toml` permits scaling above the `min_machines_running = 1` floor under load (`auto_start_machines = true`, concurrency soft/hard limits at `:67-69`). Per checklist Part 6 §33, this needs explicit derivation: under N machines, (a) `admin/treasury.ts:100`'s `getOperatorHealth()` snapshot reflects only whichever machine answers that particular admin request, with no machine identifier in the response — an operator showing `closed` on one machine and (per CTX-01) permanently `open` on another is invisible/misleading to an operator reading the dashboard; (b) `notifyOperatorPoolExhausted`/`notifyOperatorCredentialExpired`'s dedup windows are per-machine, so a fleet-wide CTX incident could fire up to N× the intended alert volume (bounded but noisier than the 15-min/10-min windows imply); (c) CTX-01's permanent-stranding bug becomes a **per-machine** landmine — machine 1 might brick operator A from one bad 401 while machine 2 never sees that 401 and keeps using A fine, so the fleet _degrades unevenly and silently_ rather than failing uniformly, making the failure mode harder to diagnose from aggregate metrics.
- Impact: Reduced effective redundancy and confusing observability under any deploy that scales past one machine; not an immediate correctness bug (each machine's procurement state-guard via `markOrderProcuring`'s `WHERE state='paid'` still prevents double-procurement, per the existing CF-14 cross-machine note in `procure-one.ts:103-112`), but it compounds CTX-01/CTX-02 in ways that are invisible from a single machine's logs.
- Evidence: `grep -n "^let \|^const.*= new Map" apps/backend/src/ctx/operator-pool.ts apps/backend/src/discord/monitoring.ts` shows every piece of pool/alert state is a bare module-level binding.
- Minimal fix: Tag the admin `getOperatorHealth()` response and Discord alert payloads with the responding machine's `FLY_ALLOC_ID`/`FLY_MACHINE_ID` so operators reading either surface know they're seeing one machine's partial view, not the fleet's.
- Better fix: Once CTX-01 is fixed (so a stuck-open state should be rare/self-healing again), this is lower priority; if/when the pool grows beyond 2 static operators or the fleet routinely runs >1 machine, move breaker state to a shared store (Redis/Postgres-backed) the way CF-14 did for payout-worker row claims — out of scope for a quick patch, flag as a Tranche-2 consideration.

### CTX-04 [P2 · LIVE] SSE stream bypasses the circuit breaker entirely — a dead operator picked for the stream never gets marked unhealthy or alerts via that path

- File: `apps/backend/src/ctx/stream.ts:104` (`const res = await fetch(url, init);` — bare global `fetch`, not `op.breaker.fetch()`), `apps/backend/src/ctx/operator-pool.ts:208-217` (`pickOperatorCredentials` returns raw credentials with no breaker handle attached)
- Description: `streamGiftCardStatus` calls the global `fetch` directly, never going through any operator's `CircuitBreaker.fetch()`. This is consistent with the module's own framing ("the stream is opportunistic, polling is the safety net" — `stream.ts:30-32`) and is fine for _transient_ network blips. But it means an operator whose bearer is genuinely dead (expired/revoked) and gets picked via `pickOperatorCredentials()` for the SSE attempt will 401 there _without_ `forceOpen()`-ing its breaker, _without_ `notifyOperatorCredentialExpired` firing, and _without_ the failure counting toward `consecutiveFailures` — `waitForRedemption` (`orders/procurement-redemption.ts:170-202`) just catches the generic thrown `Error`, logs a `log.warn`, and falls through to the polling path, which _does_ go through `operatorFetch` and _does_ correctly detect/alert the 401. Net effect: every redemption-wait against a dead operator wastes one full SSE connect-attempt round-trip before the polling fallback (which has its own pre-filter problem, but at least drives detection) ever gets a chance to flag it.
- Impact: Slower detection of a dead operator specifically through the stream path (bounded — the polling fallback still catches it on the very next `fetchRedemption` call), and a missed opportunity to `forceOpen()` the breaker one fetch-attempt earlier. Not independently severe, but it means SSE traffic contributes zero signal toward the (broken, see CTX-01) health-tracking system, undermining the "every upstream call should be a health signal" design intent stated throughout `operator-pool.ts`'s comments.
- Evidence: `stream.ts:104` uses bare `fetch`; confirmed no breaker import in `stream.ts` at all (`grep -n "circuit-breaker" apps/backend/src/ctx/stream.ts` → no matches).
- Minimal fix: Document the gap explicitly in `stream.ts`'s docstring (currently silent on this) so a future reader doesn't assume SSE failures feed the same health signal polling does.
- Better fix: Route the SSE connect attempt through the picked operator's breaker too — e.g. have `pickOperatorCredentials()` return the `Operator` (or a thin handle exposing `recordSuccess()`/`recordFailure()`) alongside the credentials, and have `streamGiftCardStatus`'s caller credit/debit it on connect success/failure (not on stream-content errors, which are expected/benign per the docstring) so a 401 on the stream path gets exactly the same `forceOpen()` + alert treatment a 401 on `operatorFetch` gets.

### CTX-05 [P3 · LIVE] SSE "ended without terminal status" error embeds the raw last frame verbatim — latent PII/credential-adjacent log-leak surface if CTX's frame contract ever changes

- File: `apps/backend/src/ctx/stream.ts:180-184`
- Description: `throw new Error(\`CTX SSE stream ended without terminal status (last: ${last === null ? 'null' : JSON.stringify(last)})\`)` embeds the full last-received frame into the error message with no field whitelisting/scrubbing. The module docstring (`stream.ts:21-23`) currently asserts frames "may or may not include the redemption fields (`redeemUrl`, `redeemCode`, `redeemPin`)" — implying CTX's *current* SSE contract sometimes does carry exactly the sensitive fields the rest of the codebase goes out of its way to encrypt at rest (CF-25) and never log (`procurement-redemption.ts:94`explicitly stops body-preview logging the instant any redemption field is non-null). If a frame containing those fields ever fails to reach a terminal status before the budget exhausts, this thrown message — caught and logged at`procurement-redemption.ts:197,201` (`waitForRedemption`'s `log.warn({ ctxOrderId, err: msg }, ...)`) — would write the redeem code/PIN into application logs in plaintext.
- Impact: Currently speculative (no confirmed observation of a frame carrying redemption fields before terminal status), but the blast radius if it does fire is real PII/financial-instrument leakage into logs, which is exactly the class `upstream-body-scrub.ts` exists to prevent for the regular HTTP error paths — this one code path is unscrubbed.
- Evidence: `stream.ts:180-184`; compare with the deliberate restraint already present at `procurement-redemption.ts:94-102`.
- Minimal fix: Whitelist `{fulfilmentStatus, paymentStatus, status}` (the three documented fields, `StreamFrame` interface at `stream.ts:44-49`) into the error message instead of `JSON.stringify(last)` wholesale.
- Better fix (same effort, more durable): Route this error message through `scrubUpstreamBody` (already imported elsewhere in the orders module) before embedding it, so any future field CTX adds to the frame shape gets the same redaction guarantee as the rest of the upstream-facing surface, without requiring this file to track the whitelist by hand.

### CTX-06 [P2 · LIVE] `streamGiftCardStatus` has no connect/idle timeout of its own — a wedged-but-connected CTX socket can park a redemption wait for the full 5-minute budget

- File: `apps/backend/src/ctx/stream.ts:100-104`, `apps/backend/src/orders/procurement-redemption.ts:170-189`
- Description: `streamGiftCardStatus` only sets `init.signal` when the caller supplies one (`stream.ts:103`: `if (opts.signal !== undefined) init.signal = opts.signal;`) — unlike `operatorFetch`, which applies a 30s default (`operator-pool.ts:285,312-313`) when the caller doesn't pass one. Its only caller, `waitForRedemption`, passes a signal that aborts at the _full_ `totalTimeoutMs` (5 minutes by default, `procurement-redemption.ts:155-156,171-172`). So a CTX SSE endpoint that accepts the TCP/TLS connection but then sends no bytes (a half-open socket, a slow-loris-style upstream hang, or a misbehaving proxy in between) parks the entire redemption wait for the full 5-minute budget before the polling fallback ever gets a chance to run — at which point `deadline` has already been reached (it's computed once, up front, as `Date.now() + totalTimeoutMs`, `procurement-redemption.ts:159`), so the polling `while (Date.now() < deadline)` loop (`:215`) executes zero times and the order fulfills with null redemption fields.
- Impact: Worst-case 5-minute stall per order against a wedged-but-open CTX SSE connection, with zero polling fallback time left over once it finally times out — the user's "Ready" screen sits without a code for the full window, then never gets one from this attempt (only a later backfill sweep could recover it). This is the same gap independently re-confirmed from the 06-15 audit pass (tracked there as CTX-P2-02); `git log` confirms `stream.ts` has not changed since its introducing commit (`543b047e`, PR #1364) and is untouched by the CF-12/13 delta — still open.
- Evidence: `stream.ts:100-104` (no default `AbortSignal.timeout`); `procurement-redemption.ts:159,171-172,215` (single up-front deadline shared between stream attempt and polling fallback, leaving the fallback with zero remaining budget on a full-timeout stream failure).
- Minimal fix: Add a connect/first-byte timeout inside `streamGiftCardStatus` itself (e.g. `AbortSignal.any([opts.signal, AbortSignal.timeout(10_000)].filter(Boolean))` if the caller's signal is absent or longer than 10s), so a wedged stream fails fast and leaves real time for the polling fallback.
- Better fix: Split the budget explicitly — give the stream attempt a fixed slice (e.g. 15-30s to prove it's making progress) independent of the caller's overall deadline, and reserve the remainder for polling, rather than letting one transport attempt consume the entire wait budget.

## Delta re-verification

### CF-12 (429 → defer + Retry-After, not counted as breaker-success)

**Verdict: partially closed.** The two defects this was meant to fix are split:

- **"429 counted as breaker success" (closed, correct):** `circuit-breaker.ts:193` (`if (response.status >= 500 || response.status === 429) { onFailure(); }`) now correctly credits a 429 toward `consecutiveFailures` instead of resetting it via `onSuccess()`. Confirmed by `circuit-breaker.test.ts` is silent on this specific case directly, but `operator-pool.test.ts`'s "CF-12: fails over to the next operator on a 429" and "throws OperatorRateLimitedError... when every operator is rate-limited" tests exercise it end-to-end and pass against real `operatorFetch` logic (only the breaker factory is mocked, not the 429-handling branch in `operator-pool.ts` itself).
- **"429 fails real paid orders" (closed, correct):** `operator-pool.ts:386-397` fails over to a sibling and on exhaustion throws `OperatorRateLimitedError` (carrying a correctly-parsed `retryAfterMs`, `operator-pool.ts:88-103`); `procure-one.ts:354-361` catches it, reverts `procuring→paid` via `revertOrderProcuringToPaid`, and returns `'skipped'` — **not** `markOrderFailed`. This is the headline fix and it is solid.
- **"defer ... and back off" (NOT closed — new finding CTX-02):** `retryAfterMs` is parsed correctly but is logged-and-discarded; nothing reads it to delay the next attempt. The procurement tick re-picks the same order (and the rest of the paid queue) on the very next tick — 5 seconds later by default (`env.ts:599`) — regardless of what CTX's `Retry-After` requested (up to 5 minutes per the `RETRY_AFTER_MAX_MS` cap). The only real throttle is the generic 30s breaker cooldown after 5 consecutive 429s, which is unrelated to CTX's actual requested window and (per CTX-01) may never actually elapse through `pickHealthyOperator`'s selection path. The "self-sustaining hot loop" the code comments claim to prevent is only half-prevented: orders stop failing, but CTX does not actually get backed off.

### CF-13 (expired-operator-401 → mark unhealthy + failover + alert + rotation path)

**Verdict: partially closed, and the part that's broken is the most consequential one — see CTX-01.**

- **"mark unhealthy" (closed, correct):** `operator-pool.ts:362-364` detects 401, drains the body, and calls `op.breaker.forceOpen()`.
- **"failover" (closed, correct within one request):** the same request retries against the next healthy operator when one exists, confirmed by `operator-pool.test.ts`'s "CF-13: marks the 401 operator unhealthy, fails over, and serves from a healthy sibling".
- **"alert" (closed, correct, with sane dedup):** `notifyOperatorCredentialExpired` fires per 401 with a 10-minute per-operator-id dedup (`discord/monitoring.ts:467-523`), confirmed by `operator-pool.test.ts`'s "alerts via notifyOperatorCredentialExpired on a 401" and "alerts per attempt but defers" tests.
- **"rotation path" (NOT closed — this is CTX-01, the audit's top finding):** the docstring's own framing — "pulling it from rotation," skipped "until its cooldown elapses" (`operator-pool.ts:13-16`, `:266`) — is false in practice. `pickHealthyOperator` filters by `getState() !== 'open'` (`operator-pool.ts:231`) before the breaker is ever `.fetch()`-ed again, and the cooldown-driven OPEN→HALF_OPEN transition lives exclusively inside `wrappedFetch` (`circuit-breaker.ts:136-141`), which that filter prevents from ever running. A forced-open operator is excluded for the remaining process lifetime, not 30 seconds. **CF-13's `forceOpen()` is what turns a pre-existing-but-rarely-triggered latent bug (needed 5 consecutive real failures pre-CF-13) into an easily-triggered one (needs exactly 1 bad 401 post-CF-13).** `/health` does not probe this state (`health.ts:142-176`, deliberate bare-fetch bypass) so Fly will not auto-restart a machine stuck this way. This is a regression introduced in effect (dramatically higher trigger probability + higher blast radius) by the CF-13 commit, even though the root defect predates it.
