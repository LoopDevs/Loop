## Findings

### PRIV-01 [P0 · LIVE] DSR data-export payload leaks to Sentry as an unredacted breadcrumb — compounds the native "console.log only" export bug into a real third-party PII/financial-data disclosure

- File: `apps/web/app/routes/settings.privacy.tsx:97-101` (the native export path, already filed as **W30-02** by v-web-routes.md) × `apps/web/app/utils/sentry-scrubber.ts:30-40` (the web `beforeSend` scrubber) × `node_modules/@sentry/browser/build/npm/esm/prod/integrations/breadcrumbs.js:10-20` (confirms `console: true` is Sentry's **default** integration option — verified against the actually-installed package, not assumed)
- Description: On native platforms, `handleExport` does `const payload = await getMyDataExport(); console.log('[loop] your data export', payload);` — the full `DsrExport` envelope (email, Stellar address, every credit-transaction row, every order including memos/ctxOrderId, every pending-payout row including `toAddress`/`txHash`) goes through `console.log`. `@sentry/react`'s default `Breadcrumbs` integration (confirmed live in the installed `node_modules` copy, version 10.51.0, matching `package.json`) instruments `console.*` with `console: true` by default — `apps/web/app/utils/sentry-lazy.ts`'s `runInit` never passes `defaultIntegrations: false` or overrides the `Breadcrumbs` integration, so this instrumentation is active. The resulting breadcrumb (`category: 'console'`, `message` built from `safeJoin(args)`, capped at 1024 chars but that's still enough to carry the email, Stellar address, and a meaningful slice of ledger/order data) sits in the Sentry scope's breadcrumb buffer for the rest of the SPA session (React Router doesn't reload the page on navigation, so the buffer persists across routes). The web `scrubSentryEvent` in `sentry-scrubber.ts` — the only `beforeSend` hook configured — only walks `event.request`, `event.extra`, `event.contexts`, `event.tags`. It has **no `breadcrumbs` handling at all** (contrast the backend's `sentry-scrubber.ts`, which explicitly walks `event.breadcrumbs[].message` and `.data` — see PRIV-02). So the next time _any_ error fires in that session — an unrelated network blip, a failed query, anything caught by the `QueryCache`/`MutationCache` `onError` or the route `ErrorBoundary` — the captured Sentry event ships with this breadcrumb attached, completely unredacted, to Sentry's servers.
- Impact: a native user who taps "Download my data" then hits any subsequent error in the same session has their full Loop financial/PII record (the single most sensitive payload the company holds on them — explicitly called out as such in `dsr-export.ts`'s own module header) sent to a third-party SaaS (Sentry) the company has 30-day retention on and that the DSR export's own "excluded" notes claim is _not_ where this data lives. This directly contradicts the design intent recorded in two separate docblocks (`dsr-export.ts`'s "Sentry events... manual operator process" and `sentry-lazy.ts`'s "Mirror of the backend Sentry init"). No attacker action is required — this is a self-inflicted, automatic leak triggerable by normal use plus ordinary bad luck (any transient error). It is also a GDPR Art. 5(1)(f) integrity/confidentiality and Art. 28 (processor) problem: Sentry wasn't disclosed as a recipient of this data class.
- Minimal fix: land the rest of W30-02's recommended native-export fix (write to `Directory.Cache` + native share sheet — never `console.log`) — that alone closes this specific trigger. Independently, since the underlying scrubber gap is broader than this one bug, also disable console breadcrumbs in the web Sentry init (`Sentry.init({ integrations: [Sentry.breadcrumbsIntegration({ console: false }), ...] })`) as defense-in-depth against the _next_ stray `console.log` of sensitive data (there will be one; see `api-client.ts:191-200` for two more console calls in auth-adjacent code that happen to be safe today but aren't structurally prevented from becoming unsafe).
- Better fix: port the backend's `event.breadcrumbs` / `event.message` / `event.exception.values[]` scrubbing into the web scrubber (see PRIV-02 — this is the root-cause fix, not just the symptom) AND fix the native export AND disable console breadcrumbs. Add a regression test that asserts a `console.log` containing a fake DSR payload, run through a captured Sentry event with `scrubSentryEvent`, never round-trips the sensitive substrings — the kind of test the backend scrubber already has (`apps/backend/src/__tests__/sentry-scrubber.test.ts`) but the web one's breadcrumb path does not.

### PRIV-02 [P1 · LIVE] Web Sentry scrubber has silently drifted from the backend's — missing breadcrumbs/message/exception scrubbing and a missing key (`idempotencyKey`), despite an explicit "keep these two files in step" invariant in its own docblock

- File: `apps/web/app/utils/sentry-scrubber.ts:1-29` vs `apps/backend/src/sentry-scrubber.ts:1-79`
- Description: The web file's docblock says, verbatim, "Mirror of apps/backend/src/sentry-scrubber.ts... Keep these two files in step: any sensitive key added there must be added here." They are not in step. Diffing the two:
  - Backend's `SentryEventLike` covers `message`, `request`, `extra`, `contexts`, `tags`, `exception.values[]`, `breadcrumbs[]`. Web's covers only `request`, `extra`, `contexts`, `tags` — **`message`, `exception.values[]`, and `breadcrumbs[]` are entirely unhandled** on the web side (root cause of PRIV-01).
  - Backend's `SENSITIVE_KEY_RE` includes `idempotencykey`/`idempotency-key` (added per its own A4-039 comment specifically because "the web UI mints them via `crypto.randomUUID()`" — i.e. the comment explicitly anticipates the web side needing this too) and `resend_api_key`. Web's regex has neither. The admin web UI's idempotency keys (used to gate credit-adjust/withdrawal/payout-retry replay, ADR 017) can therefore appear unredacted in a web-side Sentry event's `extra`/`tags`/headers if ever logged or attached — the exact attacker scenario the backend's own comment describes ("an attacker with leaked keys + an admin session can replay a stored response or fabricate an `audit.replayed: true` envelope").
  - Backend also scrubs free-text PII shapes (email/bearer/Stellar-secret/long-hex regexes) inside `event.message` and `event.exception.values[].value`; web relies entirely on the separate pre-capture `scrubErrorForSentry` (`sentry-error-scrubber.ts`) for this, which only runs on values explicitly passed to `captureExceptionLazily` — it does **not** run on breadcrumbs accumulated before that call, and the regex-based `event`-level backstop the backend has doesn't exist on web at all.
- Impact: P1 on its own (idempotency-key admin-replay exposure + general drift from a documented invariant), and it's the structural root cause that makes PRIV-01 possible. Going forward, any new sensitive key added to the backend's redaction list per the "keep in step" instruction is silently _not_ applied to web unless someone remembers to hand-edit a second file — this is exactly the kind of invariant that should be enforced by a shared module or a parity test, not a comment.
- Minimal fix: copy the backend's `message`/`exception.values[]`/`breadcrumbs[]` walking logic and the missing regex keys (`idempotencykey`, `idempotency-key`) into the web scrubber now.
- Better fix: extract one shared scrubber into `packages/shared/` (per ADR 019's three-part test — this is genuinely identical logic needed by both apps, the canonical case the policy exists for) so "keep in step" becomes structurally true instead of a comment, and add a `check:*`-style parity test (mirroring `shared-type-parity`/`openapi-parity`'s pattern) that fails CI if the two `SENSITIVE_KEY_RE` patterns or `SentryEventLike` shapes ever diverge again.

### PRIV-03 [P2 · LIVE] DSR export omits `user_favorite_merchants` — a GDPR Art. 15/20 portability gap

- File: `apps/backend/src/users/dsr-export.ts` (no import of `userFavoriteMerchants`), `apps/backend/src/db/schema.ts:1025-1044`
- Description: `user_favorite_merchants` is a real per-user table (`(user_id, merchant_id)` composite PK, `onDelete: 'cascade'` on `users.id`) holding a user's pinned/favourited merchants — personal data reflecting shopping preference, squarely the kind of thing Art. 15/20 covers ("data concerning the data subject"). `buildDsrExport` never queries it; the favourites a user has saved are invisible to both the export and the "excluded — request via privacy@" notes list (so the export's own completeness claim — "every database row Loop holds keyed to the calling user" — is currently false).
- Impact: low sensitivity per-row (merchant IDs), but it's a portability-completeness defect, the kind of small omission a regulator's spot-check or a determined user's "this export is incomplete" complaint would catch. Also currently silently survives account deletion (see DSR matrix) since the `users` row is never hard-deleted, so old favourites linger attached to an anonymised account indefinitely with no path to view or clear them.
- Minimal fix: add `userFavoriteMerchants` to the `Promise.all` in `buildDsrExport`, add a `favorites: Array<{ merchantId: string; createdAt: string }>` field to `DsrExport`, bump `DSR_EXPORT_SCHEMA_VERSION`. Also add an explicit `DELETE FROM user_favorite_merchants WHERE user_id = ?` to `deleteUserViaAnonymisation`'s transaction (or document why retaining them post-anonymisation is the deliberate choice — currently undocumented either way).
- Better fix: same, plus register a concrete (non-passthrough) schema for the DSR export response in OpenAPI — this dovetails with **PLAT-30-24** (v-platform.md), which already flagged the export's `z.object({}).passthrough()` schema as defeating contract-drift detection on exactly this kind of silent field gap.

### PRIV-04 [P2 · LIVE] DSR delete never touches the `otps` table — the deleted user's real email persists in plaintext for up to 30 days post-"deletion" on a clock the deletion request doesn't start

- File: `apps/backend/src/users/dsr-delete.ts` (no reference to `otps`/`email`-keyed cleanup), `apps/backend/src/db/schema.ts:358-375` (`otps` keyed on `email` text, no `userId` FK), env var `LOOP_AUTH_ROW_RETENTION_DAYS` (default 30, per `AGENTS.md`)
- Description: `otps` rows are keyed purely by the literal `email` string typed at request-otp time — there is no `userId` foreign key linking them to the `users` row at all, so `deleteUserViaAnonymisation`'s `users.email` rewrite to `deleted-{uuid}@deleted.loopfinance.io` has zero effect on any pre-existing `otps` rows for the user's real email (their last successful login OTP, any abandoned/expired OTP attempts, etc.). Those rows — real email + `codeHash` + `attempts` + timestamps — are only ever cleaned up by the generic, deletion-request-independent `auth-row-purge` sweep, which runs on `LOOP_AUTH_ROW_PURGE_INTERVAL_HOURS` and only deletes rows older than `LOOP_AUTH_ROW_RETENTION_DAYS` (30 by default). A user who submits a DSR delete request has no guarantee their real email is erased from this table for up to 30 days afterward — a different, longer clock than the deletion request itself, and one the user is never told about (it's not in the deletion-confirmation copy in `settings.privacy.tsx`, nor in the DSR export's "excluded" notes, nor in `/privacy`).
- Impact: low-to-moderate. The data at rest is an email address + a SHA-256-style code hash (not directly exploitable on its own) + timestamps, so this isn't a high-severity secret leak — but it is a literal, real PII anchor (the thing DSR deletion exists to remove) surviving the "your personal data is removed" promise the deletion-confirmation UI makes ("This permanently removes your personal data... This can't be undone"). Under GDPR's "without undue delay" standard for Art. 17 erasure, a fixed 30-day residual window on an unrelated retention sweep is a defensible-but-undisclosed gap, not obviously compliant by default.
- Minimal fix: have `deleteUserViaAnonymisation` also `DELETE FROM otps WHERE email = <pre-anonymisation email>` inside the same transaction (capture the original email before the UPDATE overwrites it). Cheap, no schema change.
- Better fix: same, plus add a `userId` FK to `otps` (nullable, since CTX-proxy and pre-auth flows may not have a resolved user yet) so future deletion/export logic doesn't have to special-case an email-keyed table, and document the otps-table exclusion/inclusion explicitly in the DSR export's "excluded" notes either way (currently silent).

### PRIV-05 [P0 · LAUNCH-GATE] Zero terms-of-service acceptance or age-gate capture at signup — `/terms` asserts 18+ and "use implies acceptance," but nothing enforces, prompts, or records either

- File: `apps/web/app/components/features/onboarding/signup-tail.tsx` (full signup form — email entry + 6-digit OTP entry, confirmed via read: no terms/age UI element of any kind), `apps/web/app/routes/onboarding.tsx`, `OnboardingDesktop.tsx`, `Onboarding.tsx`, `screens-trust.tsx` (native walkthrough — zero references to `/terms` or `/privacy` anywhere in the onboarding component tree, confirmed via grep), `apps/web/app/routes/terms.tsx:62-67` ("You must be at least 18... Use of the service implies acceptance of these terms.")
- Description: the entire web/native signup flow is exactly two screens: type an email, type a 6-digit OTP. There is no terms/privacy link, no checkbox, no "I confirm I am 18+" prompt, and no `auth.tsx`/`signup-tail.tsx` reference to `/terms` or `/privacy` at all (the only `/settings/privacy` reference in `auth.tsx` is an unrelated account-settings link, not a signup-time consent flow). `/terms` itself is explicitly a "structural shell... pending legal copy" (its own placeholder banner) whose Eligibility section relies entirely on passive "use implies acceptance" — legally the weakest form of consent, and not paired with any actual gate even at that low bar. No `termsAcceptedAt` / `ageConfirmedAt`-shaped column exists anywhere in `schema.ts` (grepped) — there is no way to even retroactively prove a given user ever saw or accepted anything.
- Impact: this is the same gap CF-26 (06-15 audit, X-PRIV-02) already flagged and it remains fully open — re-verified independently this round with fresh evidence (file/line citations above) rather than just re-asserted. It blocks public launch on two independent grounds: (1) Apple App Store Guideline 5.1.1/1.2 and Play Store policy both expect an enforceable terms-acceptance gate for an app handling payments; (2) without any recorded consent or age confirmation, Loop has no evidentiary basis that a given user agreed to the (still-placeholder) terms or confirmed eligibility, which matters a great deal for a product that moves real money via Stellar and is actively expanding into new geographies (ADR 035).
- Minimal fix: add a single non-dismissable checkbox to the email-entry screen ("I'm 18+ and agree to the Terms and Privacy Policy") wired to `requestOtp`'s call site, persist `termsAcceptedAt: timestamp` (+ a `termsVersion` string so future legal-copy changes are attributable) on the `users` row at account-creation time, and block `verifyOtp`/account creation server-side if the flag wasn't sent — client-side-only gating is trivially bypassable via direct API calls and doesn't produce a record.
- Better fix: same, plus version the terms/privacy documents explicitly (a `legal_documents` table or simple versioned-slug scheme) so re-acceptance can be required when the placeholder copy is replaced with the real, counsel-reviewed text before App Store submission — don't ship the "real" terms without a forced re-consent flow for already-onboarded users.

### PRIV-06 [P0 · LAUNCH-GATE] Zero sanctions/OFAC/geo-eligibility screening anywhere — not for signup, not for orders, not for Stellar payout destinations — and the gap's urgency just increased with ADR 035's AE/IN/SA markets

- File: confirmed via exhaustive grep (`sanction|ofac|denylist|blocklist|embargo|export.control` across `apps/backend/src` and `docs/`) — zero implementation hits; only doc mentions are `env.ts` matching unrelated config names and `docs/adr/005-known-limitations.md` (a known-gap acknowledgement, not a control). `apps/backend/src/public/geo.ts` (MaxMind GeoLite2 IP→country lookup) is used **exclusively** for UX (seed the locale/region selector, ADR 033/034) — confirmed by reading the full handler: the resolved country is returned to the client for display purposes and never referenced by any auth, order, or payout code path. `docs/adr/035-extended-supplier-currency-markets.md` (read in full) discusses KYC/licensing only as a _future, separate-decision_ prerequisite for adding cashback to a region — not as a precondition for the order-path (gift-card purchase via XLM) it ships today.
- Description: signup (email OTP) has no IP-geofencing, no SDN/OFAC list check, no jurisdiction allowlist/denylist. Order creation has no destination/jurisdiction check. Stellar payout submission (cashback payouts to a user-linked wallet address) has no address-level screening against sanctioned-entity wallet lists (a standard control at crypto-adjacent fintechs — Chainalysis/TRM/Elliptic-style screening, or even a static OFAC SDN Stellar-address list, neither of which exists here). This matches the brief's prior expectation ("Expected: none yet") — confirmed precisely, not assumed. What's new this round: ADR 035 (merged, live) added AE (UAE), IN (India), SA (Saudi Arabia), AU, MX as **display + order-path-orderable** markets. UAE in particular is a recognised higher-risk jurisdiction for crypto/gift-card-adjacent sanctions-evasion typologies (transshipment risk re: Iran/Russia-adjacent flows is a standing OFAC/FinCEN concern, independent of UAE itself being sanctioned) — surfacing and actively promoting (SEO-indexed locale pages, per ADR 034's sitemap model) a real-money order path into that market without even a baseline IP/country gate raises the stakes on a gap that was previously more abstract (US/GB/CA/Eurozone all being lower-risk by comparison).
- Impact: LAUNCH-GATE for any jurisdiction-conscious counsel sign-off, and the kind of gap that gets materially worse the more markets/volume ship before it's closed (today: zero users, zero history; every day post-launch without it is exposure accruing against a clean slate). Not a code defect in the sense of "broken code" — characterizing precisely per the brief: this is a missing control, full stop, not a bug in an existing one.
- Minimal fix: a static, periodically-refreshed OFAC SDN / comprehensively-sanctioned-country IP-geofence at signup (block `request-otp` for IPs resolving to Cuba/Iran/North Korea/Syria/sanctioned-Russia-regions via the MaxMind DB already wired for `public/geo.ts` — re-purposing infrastructure that already exists) plus a denylist check against a maintained OFAC SDN address list before submitting any Stellar payout (`payments/payout-submit*`). Both are bolt-on checks, not architecture changes.
- Better fix: integrate a real sanctions-screening provider (Chainalysis/TRM/Elliptic for the Stellar-address side; a maintained KYC/geo provider for the user side) gated behind a feature flag so it can be enabled per-market as licensing work completes per ADR 035's own stated phasing — and treat AE/IN/SA specifically as needing this _before_ (or concurrent with, at minimum) their order-path going live, not after.

### PRIV-07 [P1 · LIVE] No per-user order/spend velocity controls beyond a flat, IP-keyed, in-process, bypassable rate limit — the real gate against gift-card cash-out abuse is "did a real on-chain payment land," not any anti-fraud signal

- File: `apps/backend/src/routes/orders.ts:71-86` (`rateLimit('POST /api/orders', 10, 60_000)`, `rateLimit('POST /api/orders/loop', 10, 60_000)`), `apps/backend/src/orders/request-schemas.ts:42-50` (`amount: z.number().min(0.01).max(10_000)` — the only per-order cap), `apps/backend/src/orders/loop-create-checks.ts` (the only DB-backed pre-flight checks: sufficient-credit balance and "is this the user's first loop_asset order" — neither is a fraud/velocity signal), `apps/backend/src/middleware/rate-limit.ts:35-37` (module-local `Map`, per-(route,IP) key)
- Description: order creation has exactly three numeric gates and nothing else: a $10,000-equivalent per-order cap, a 10-requests/minute-per-IP-per-route rate limit, and (for credit-funded orders) a balance check. There is no per-user daily/weekly/monthly cumulative spend cap, no anomaly signal (e.g. "N orders to N different merchants within M minutes from a brand-new account"), no manual-review queue or threshold for high-value/high-velocity activity, and no automated SAR-style flagging — `admin/top-users.ts`'s own docblock literally frames its only fraud-adjacent feature as a _human eyeballing a leaderboard_ ("fraud / concentration signal — one user accounts for 70%..."), not an automated control. The rate limit itself is the weakest possible velocity control for this purpose: it's keyed on IP (trivially rotated — VPN/proxy/mobile-network-NAT), it's in-process per Fly machine (so N machines ≈ N× the configured budget, per this audit's own Part 6 §33 framing — not independently re-verified here but the mechanism is the same one that vertical describes), and most importantly it isn't _account_-scoped at all, so it does nothing against the textbook gift-card-cash-out pattern of "many cheap accounts, each under its own IP/rate budget."
- Impact: gift cards are a recognised money-laundering/stolen-payment-instrument cash-out vector specifically because they convert a payment into a liquid, resellable, hard-to-trace instrument. Loop's only real friction today is that each order requires an actual on-chain XLM/USDC/LOOP-asset payment to clear before procurement — that's a genuine and non-trivial barrier (you can't manufacture gift cards from nothing), but it's a payment-rail property, not a Loop-side control, and it does nothing to slow down someone who _does_ have a supply of crypto they want to launder through gift-card resale: nothing stops one funded account from placing $10,000 orders back-to-back up to whatever the 10/min-per-IP budget allows (≈$6M/hour theoretical ceiling against a single IP, ignoring funding-side friction entirely), and nothing stops splitting that across IPs/accounts to remove even that ceiling.
- Minimal fix: add a per-user (not just per-IP) cumulative daily order-value cap, enforced server-side in `loop-create-checks.ts` alongside the existing balance check — e.g. sum `orders.faceValueMinor` for the user over a rolling 24h window and reject/flag above a threshold. Cheap, uses data already in the `orders` table, no new infra.
- Better fix: real velocity/anomaly tooling — per-user AND per-device/per-payment-source cumulative tracking (shared cross-machine store, not in-process, per Part 6 §33), an admin-facing review queue for orders crossing a value/velocity threshold (manual hold before procurement, not just post-hoc dashboard visibility), and — once Loop is processing real volume — a real KYC tier for users above a cumulative-spend threshold, consistent with standard MSB/money-transmitter practice for this product category.

### PRIV-08 [P2 · LIVE / forward-looking] Multi-accounting is structurally trivial (email-OTP-only, no device/IP correlation) — live, narrow benefit today (rate-limit evasion for structuring); the larger cashback/referral-farming risk is correctly Phase-2-gated but undocumented as a launch consideration

- File: `apps/web/app/components/features/onboarding/signup-tail.tsx` (full signup surface — email + OTP, no device fingerprint/attestation of any kind ever collected), `apps/backend/src/auth/` (OTP issuance has no device/IP binding beyond the flat per-IP rate limit already covered in PRIV-07)
- Description: creating a new Loop account costs nothing but a fresh email address (and the OTP-request rate limit, 5/min per IP — easily satisfied by spacing requests or rotating IP). There is no device fingerprinting, no phone-number verification, no CAPTCHA, no IP-reputation check. Characterizing precisely per the brief's ask:
  - **Phase 1 (discount mode, today):** multi-accounting buys an attacker almost nothing on the _discount_ side — the gift-card discount is identical regardless of account, and (per PRIV-07) there's no per-account spend cap to bypass by spreading across accounts. The one live, narrow benefit: bypassing the per-IP rate limit (PRIV-07) to structure a larger cash-out across many freshly-OTP'd accounts than one account/IP could push through alone. This is a real, live concern, but it's a _rate-limit-evasion_ concern, not a _discount-arbitrage_ one.
  - **Phase 2 (cashback mode, not yet live — gated by `LOOP_PHASE_1_ONLY`/`LOOP_WORKERS_ENABLED`/`LOOP_AUTH_NATIVE_ENABLED` per `AGENTS.md`):** this is where multi-accounting becomes a real abuse vector in its own right — farming repeated "new user" cashback incentives (none exist yet in code, but the product shape strongly implies they will) and inflating recycling/cashback-volume metrics. Correctly out of scope for a Phase-1 code fix, but worth flagging now since identity/device correlation is exactly the kind of control that's much cheaper to add _before_ cashback goes live than to retrofit once farmed accounts already hold balances.
- Impact: P2 — not urgent for Phase 1 beyond its overlap with PRIV-07, but a named gap that should be tracked so it isn't rediscovered cold at the Phase-2 cashback-launch review.
- Minimal fix: none needed for Phase 1 specifically beyond PRIV-07's per-user velocity fix (which also raises the cost of the rate-limit-evasion use case). Document this finding (or CF-26's existing X-PRIV reference) in the Phase-2 cashback launch checklist explicitly so it's evaluated before cashback flips on.
- Better fix: before Phase 2, add lightweight device/IP correlation at signup (e.g. a device-id cookie/fingerprint correlated across accounts, flagged — not necessarily blocked — for manual review) and design any referral/promo mechanism (none exists in code today — `docs/roadmap.md:237` lists "Referral program" as a still-unbuilt roadmap item, so there is currently no referral-abuse surface to audit) with self-referral and cap-enforcement built in from spec, not retrofitted.
