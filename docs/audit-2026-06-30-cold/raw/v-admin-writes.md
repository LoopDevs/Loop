## Findings

### ADM-01 [P1 · LIVE] No daily/fleet aggregate cap on admin withdrawals — the one money-mutating write type the CF-06/A4-020 cap hardening didn't reach

- File: `apps/backend/src/credits/withdrawals.ts` (whole file); contrast `apps/backend/src/credits/adjustments.ts:110-145`, `apps/backend/src/credits/refunds.ts:211-247`, `apps/backend/src/credits/payout-compensation.ts:150-191`.
- Description: `credit_transactions.type='adjustment'` (per-admin cap), `'refund'` (fleet-wide cap, CF-06) and the compensation `'adjustment'`-referencing-`'payout'` subtype (fleet-wide cap, A4-020) all check `env.ADMIN_DAILY_ADJUSTMENT_CAP_MINOR` before committing. `applyAdminWithdrawal` performs **no analogous check** — the only bounds are the per-call body-schema cap (`amountMinor ≤ 10_000_000n` minor, same regex in `admin/withdrawals.ts`'s `BodySchema`) and `InsufficientBalanceError` (can't withdraw more than the user's existing balance). ADR-024 §"Deferred (Phase 2b)" documents this explicitly as a known gap ("Min / max withdrawal caps... Phase 2a uses the same 10,000,000 minor cap as refund/adjustment — enough for manual operator use"), but the daily-aggregate dimension specifically was never revisited even as every sibling write got it during this delta.
- Impact: Withdrawal is the **only** admin money-write where real value leaves the system end-to-end (a Stellar payment to an admin-specified `destinationAddress`, queued via `pending_payouts` and submitted by the payout worker) rather than just shifting an internal ledger balance. ADR-028's own threat model is "a stolen admin session... can issue unlimited credit adjustments... the per-day cap is a circuit-breaker"; for withdrawal specifically, that circuit-breaker doesn't exist. An attacker who has both a captured bearer token AND a step-up token (full account compromise, e.g. email/OTP access) can issue repeated $100k-equivalent (10M minor) withdrawals — bounded only by the 20/min rate limit (`routes/admin-credit-writes.ts:78`) and the existence of users with sufficient balance — with no per-day ceiling. At 20/min for a full 5-minute step-up TTL window that's up to 100 calls × $100k = $10M in a single window, replayable again after re-minting a fresh step-up token. Note the rate limiter itself is in-process/per-IP (`middleware/rate-limit.ts`, keyed `${name}:${ip}`), so if `min_machines_running`is ever raised above the current`fly.toml` value of 1, the effective cap multiplies by machine count too (Part 6 §33 concern, compounding this one).
- Evidence: `applyAdminWithdrawal` in `apps/backend/src/credits/withdrawals.ts:87-218` has no `adjustmentCapLockKey`/`DailyAdjustmentLimitError` reference (confirmed via `grep -rn "ADMIN_DAILY" apps/backend/src` — zero hits in `credits/withdrawals.ts` or `admin/withdrawals.ts`, vs. one hit each in `adjustments.ts`, `refunds.ts`, `payout-compensation.ts`).
- Minimal fix: Add the same fleet-wide (or per-admin) `ADMIN_DAILY_ADJUSTMENT_CAP_MINOR` check to `applyAdminWithdrawal`, mirroring the refund/compensation pattern (advisory lock on `adjustmentCapLockKey('withdrawal', currency, dayStart)`, sum `type='withdrawal'` magnitudes since UTC start-of-day).
- Better fix: A dedicated, lower `ADMIN_DAILY_WITHDRAWAL_CAP_MINOR` (withdrawals are real on-chain egress, not a same-system ledger shuffle — the bar should plausibly be tighter than the adjustment/refund cap), plus surfacing the day's cumulative withdrawal total on the `/admin/payouts` or `/admin/treasury` page so ops sees the budget burn in real time, not just on a 429.

### ADM-02 [P1 · LIVE] `admin/step-up-handler.ts` — the endpoint that mints the token gating every destructive admin write — has zero test coverage

- File: `apps/backend/src/admin/step-up-handler.ts` (whole file, no `__tests__/step-up-handler.test.ts` exists).
- Description: `adminStepUpHandler` (`POST /api/admin/step-up`) contains all of: the 503-when-unconfigured branch, the `auth.kind !== 'loop'` 401, body validation, `normalizeEmail`/`NonAsciiEmailError` handling, `findLiveOtp` miss → `incrementOtpAttempts` → 401, the happy path (`markOtpConsumed` → `signAdminStepUpToken`), and the catch-all 500. None of this is exercised anywhere in the test tree. `auth/__tests__/admin-step-up.test.ts` only tests the lower-level `signAdminStepUpToken`/`verifyAdminStepUpToken`/`isAdminStepUpConfigured` crypto helpers in `auth/admin-step-up.ts` — a different file. The integration suite (`__tests__/integration/admin-writes.test.ts`) deliberately bypasses the handler by minting tokens directly via `signAdminStepUpToken(...)` (its own comment at line 87-89 says "the integration value here is [gating], not the step-up minting flow itself (covered by unit tests)" — but no such unit tests exist).
- Impact: A regression here is invisible to CI. Concretely: a future refactor that reorders `markOtpConsumed`/`signAdminStepUpToken` (enabling OTP replay to mint multiple step-up tokens from one code), drops the `auth.kind !== 'loop'` check, or changes the 503-vs-401 branching, would ship with all tests green. This is the single most security-critical new code path in the ADR-028 surface (it's the thing that turns "I have a bearer token" into "I can destroy money"), and it's the one piece with no direct test.
- Evidence: `grep -rln "step-up\|stepUp\|StepUp" apps/backend/src --include="*.test.ts"` returns only `__tests__/integration/admin-writes.test.ts`, `auth/__tests__/admin-step-up-middleware.test.ts`, `auth/__tests__/admin-step-up.test.ts` — no hit for `admin/step-up-handler.ts`. `grep -rln "/api/admin/step-up'" apps/backend/src --include="*.test.ts"` returns nothing.
- Minimal fix: Add `apps/backend/src/admin/__tests__/step-up-handler.test.ts` covering: 503 unconfigured, 401 non-loop auth, 400 malformed body, `NonAsciiEmailError` → 401, OTP-miss → `incrementOtpAttempts` called + 401, happy path → `markOtpConsumed` called once + token minted with correct `scope` (both omitted-defaults-to-wildcard and explicit-narrow cases), and the catch-all 500.
- Better fix: Same, plus an integration-level test that drives the real `/api/admin/step-up` HTTP path end-to-end (seed a live OTP row via `auth/otps.ts` primitives, POST the code, assert the OTP is consumed and a second POST with the same code 401s) — the current integration suite's claim that this is "covered by unit tests" should either become true or the comment should be corrected.

### ADM-03 [P2 · LIVE] Credit-adjustment daily cap is per-admin while refund/compensation caps are fleet-wide — same threat, inconsistent scope

- File: `apps/backend/src/credits/adjustments.ts:110-145` (per-admin: `adjustmentCapLockKey(args.adminUserId, ...)`, query filters `referenceId = args.adminUserId`) vs. `apps/backend/src/credits/refunds.ts:211-247` and `apps/backend/src/credits/payout-compensation.ts:150-191` (both fleet-wide: scope is a fixed string `'refund'` / `'payout-compensation'`, no admin-id filter).
- Description: The refund/compensation caps' own doc comments explain WHY they're fleet-wide: "there is no per-admin attribution to key on" (refunds.ts:33-40) and "until a schema-level per-admin attribution lands" (payout-compensation.ts:150-158) — both implicitly acknowledging that a per-admin cap is weaker against a multi-admin-compromise scenario than a fleet-wide one. The original `applyAdminCreditAdjustment` cap (A2-1610, predates this delta) was never revisited to match.
- Impact: ADR-028's stated threat is a stolen/coerced admin session; if an attacker compromises **N** admin accounts (phishing, credential-stuffing, malware across N laptops), the _adjustment_ primitive gives them N independent daily budgets (N × `ADMIN_DAILY_ADJUSTMENT_CAP_MINOR`), while the _refund_/_compensation_ primitives correctly cap the combined total at one. This is a real, if narrower, gap in "the daily cap bounds the blast radius of a compromised admin surface."
- Evidence: `adjustmentCapLockKey(args.adminUserId, args.currency, dayStart)` at `adjustments.ts:115` vs. the fixed-string scopes `REFUND_CAP_LOCK_SCOPE = 'refund'` (`refunds.ts:42`) and `COMPENSATION_CAP_LOCK_SCOPE = 'payout-compensation'` (`payout-compensation.ts:45`).
- Minimal fix: Document the asymmetry as an accepted trade-off (adjustments are bidirectional/legitimate-correction-heavy so a tighter fleet cap would hurt normal ops more) if that's the intent — currently no comment explains why adjustment stayed per-admin while its siblings moved to fleet-wide.
- Better fix: Move the adjustment cap to fleet-wide too (or add a secondary fleet-wide ceiling alongside the existing per-admin one, so neither a single admin nor a swarm of compromised admins can exceed the fleet-wide figure in a day) — consistent with the refund/compensation precedent already shipped in this delta.

### ADM-04 [P2 · LIVE] OpenAPI doc/contract drift: withdrawal write doesn't document its own step-up gate

- File: `apps/backend/src/openapi/admin-withdrawal-write.ts` (whole file); contrast `apps/backend/src/openapi/admin-credit-writes.ts:139-142,179-182`, `apps/backend/src/openapi/admin-payouts-cluster-writes.ts:62-65,96-99,147-150,188-191`, `apps/backend/src/openapi/admin-user-writes.ts:72-75,111-114`.
- Description: `routes/admin-credit-writes.ts:75-81` mounts `requireAdminStepUp('withdrawal')` on `POST /api/admin/users/:userId/withdrawals` — the route genuinely is step-up-gated. But `openapi/admin-withdrawal-write.ts`'s registration for that exact path: (a) doesn't declare `'x-admin-step-up'` in `request.headers` (every sibling write does); (b) doesn't declare a `503` response for `STEP_UP_UNAVAILABLE` (every sibling write does); (c) its `401` description says only "Missing or invalid bearer" (siblings say "... / missing or invalid step-up token"); (d) the path `description` text never mentions ADR-028 or step-up at all, unlike every sibling's description which explicitly calls it out (e.g. credit-adjustments: "ADR-028 step-up gate enforced at the route"). This is the exact bug class the 2026-06-15 audit flagged as P2-7 ("credit-adjustments and payouts/:id/retry omit the reachable 503 STEP_UP_UNAVAILABLE") — that finding was fixed for those two endpoints in this delta, but the same gap on withdrawal (which has been step-up-gated since the original Phase-1 ADR-028 rollout, predating this delta) was never closed.
- Impact: Documentation/contract integrity only — the actual runtime behavior is correct (confirmed the web client `apps/web/app/services/admin-user-credits.ts:133` sends `withStepUp: true` for `applyAdminWithdrawal`). But this violates the project's own `CLAUDE.md` doc-update rule ("declare every status code the handler can return... 503 for circuit-open") and `scripts/check-openapi-parity.mjs`'s own stated purpose of catching "spec/handler drift" — its rules (`missing-registration`, `missing-429`, `admin-missing-404`, `admin-403`) don't check header/503 correctness for step-up-gated routes, so this drift is structurally invisible to the automated gate. Anyone generating an API client or onboarding doc from `openapi.ts` for this one endpoint will be surprised by an undocumented `401 STEP_UP_REQUIRED`/`503 STEP_UP_UNAVAILABLE`.
- Evidence: Side-by-side diff of `openapi/admin-withdrawal-write.ts:104-153` (no step-up header/503/description) against any of its four siblings, all of which have all three.
- Minimal fix: Add the `'x-admin-step-up'` header schema, a `503 STEP_UP_UNAVAILABLE` response entry, and the step-up clause in the `401` description and the path `description`, mirroring `admin-user-writes.ts`'s home-currency registration verbatim.
- Better fix: Same, plus extend `check-openapi-parity.mjs` with a new rule (`step-up-mount-undocumented`): cross-reference every route mount wrapped in `requireAdminStepUp(...)` against its openapi registration and fail if the registration is missing the step-up header/503 — turns this class of drift into a CI-caught one instead of relying on per-PR vigilance.

### ADM-05 [P2 · LIVE] `admin/payout-compensation.ts`'s own pre-checks get permanently cached in the idempotency snapshot, unlike the sibling pattern established by `payouts-retry.ts`

- File: `apps/backend/src/admin/payout-compensation.ts:108-191` (handler-level pre-checks for payout-not-found / wrong-kind / wrong-state / non-LOOP-asset, all `return`ed from inside the `withIdempotencyGuard` `doWrite()` callback).
- Description: `withIdempotencyGuard`'s contract (`admin/idempotency.ts:97-131`) stores **whatever `doWrite()` returns**, regardless of status code — `200`, `404`, `400`, `409`, `500` alike get persisted to `admin_idempotency_keys` for the 24h TTL and replayed verbatim on the next call with the same `(adminUserId, key)`. `admin/payouts-retry.ts` explicitly engineered around this: its analogous "payout not retryable" condition is raised as a **thrown** `PayoutNotRetryableError` (not returned), with an inline comment explaining exactly why — "Throwing... rolls the guard txn back, so the 404 is NOT snapshot-stored... a replay with the same key stays free to try again once the row is back in a failed state." `admin/payout-compensation.ts`'s four pre-checks (payout missing → 404, `kind!=='withdrawal'` → 400, `state!=='failed'` → 409, non-LOOP asset → 500) do the opposite: they `return { status, body }` from inside `doWrite()`, which gets stored. The underlying `credits/payout-compensation.ts` primitive (`applyAdminPayoutCompensation`) re-checks the SAME `kind`/`state` conditions under `FOR UPDATE` and correctly **throws** `PayoutNotCompensableError` for them (not cached) — so the redundant handler-level pre-check is the only place this poisoning can occur, and it exists purely to get the right HTTP status code (400 vs. 409) before the row lock, since the primitive's single exception class can't distinguish them by type.
- Impact: If a caller (automation, a future "compensate" web form, or a retried request after a client crash) reuses the same `Idempotency-Key` for a payout whose state changes between attempts — e.g. first attempt hits the 409 "payout is in state 'pending'" pre-check before the worker has marked it `failed` — a second attempt with the _same key_, made after the payout legitimately becomes `failed`, replays the stale cached 409 for up to 24h instead of re-evaluating and succeeding. The blast radius is currently bounded by the web UI minting a fresh `Idempotency-Key` per click (`apps/web/app/routes/admin.payouts.$id.tsx`'s `handleRetryReasonResolve`-equivalent), but there is no service function in the web app that even calls `/payouts/:id/compensate` (see ADM-06) — so today this can only be hit via direct API automation, where exactly this kind of key-stability is the more natural pattern to reach for.
- Evidence: `admin/idempotency.ts:204-229` stores `doWrite()`'s return unconditionally; `admin/payouts-retry.ts:36-48,176-182` documents and implements the throw-to-avoid-caching pattern for the equivalent 404 case; `admin/payout-compensation.ts:124-159` returns (doesn't throw) for all four pre-check failures. The unit test suite's mock of `withIdempotencyGuard` (`admin/__tests__/payout-compensation.test.ts:69-93`) faithfully replays this behavior (every `doWrite()` result is passed to `storeMock`), but no test asserts `storeMock` is/isn't called for the negative-status cases, nor exercises a second call with the same key after a state change — the gap is unexercised, not just unfixed.
- Minimal fix: Convert the four pre-checks in `admin/payout-compensation.ts` to throw typed errors (mirroring `PayoutNotRetryableError`) instead of returning, so none of them get snapshot-cached — exactly the pattern `payouts-retry.ts` already uses next door.
- Better fix: Same, plus a regression test: seed a "wrong state" 409 with key K, then flip the payout to `failed` and re-POST with the same key K, asserting success (not a replayed 409) — would have caught this on either side of the fix.

### ADM-06 [P2 · LIVE] Admin UI has no surface to call refund or payout-compensation — backend primitives are fully built (step-up + idempotent + audited) but unreachable from `/admin/*`

- File: `apps/web/app/services/*` (absence) — `apps/web/app/services/admin-user-credits.ts` exports `applyCreditAdjustment` and `applyAdminWithdrawal` only; there is no `applyAdminRefund`/`applyRefund`-equivalent function anywhere in `apps/web/app/services/`, and no service function calling `POST /api/admin/payouts/:id/compensate`.
- Description: `grep -rli "refund" apps/web/app --include="*.tsx" --include="*.ts"` (excluding tests) hits only `utils/error-messages.ts` (an error-code-to-copy map) and `CreditTransactionsTable.tsx` (a read-only label for the `'refund'` transaction type in a history table) — no form, no mutation call. Same for "compensat(e/ion)" — only unrelated hits (`settings.privacy.tsx`, `services/user.ts`). ADR-017's own motivating context for building these primitives was "Currently this requires a DB console" — the goal was explicitly to give support a tool that doesn't require raw DB/API access, but for these two of the six step-up-gated money writers, that goal isn't met.
- Impact: Per Part 1 §17 ("Usefulness: can support resolve common tickets end-to-end?") — refund and payout-compensation are exactly the kind of support-mediated actions ADR-017/024 were built for, yet ops must curl the admin API (bearer token + manually-minted step-up token + hand-built idempotency key) to issue either. That's both a productivity gap and a quieter audit-trail risk (curl-based ops actions are more likely to be run from a personal laptop/terminal history than a logged admin-panel session, though the backend-side Discord/audit-tail trail is identical either way).
- Evidence: `grep -rln "StepUpModal" apps/web/app --include="*.tsx"` lists `CreditAdjustmentForm.tsx`, `AdminWithdrawalForm.tsx`, `HomeCurrencyForm.tsx`, `admin.payouts.tsx`, `admin.payouts.$id.tsx` — five surfaces, none of which is refund or compensation.
- Minimal fix: Ticket it explicitly (it was already flagged in the 2026-06-15 audit's "Tooling utility / coverage gaps" #2 and is still open 22 commits later) so it doesn't silently fall off the backlog again.
- Better fix: Build `RefundForm`/`AdminPayoutCompensateButton` components mirroring `CreditAdjustmentForm.tsx`'s shape (step-up wrap, stable per-attempt idempotency key, `ReasonDialog`), wired into the user-drill page (refund, bound to an order picker) and the payout-detail page (compensate, shown only when `state==='failed' && kind==='withdrawal'`).

### ADM-07 [P2 · LIVE incident-interaction] The fleet-wide refund daily cap is shared between the admin-fraud circuit-breaker and the CF-20 automatic-refund reliability safety-net

- File: `apps/backend/src/credits/refunds.ts:211-247` (cap check inside `applyAdminRefund`); `apps/backend/src/orders/procure-one.ts:398-449` (`autoRefundAfterCtxPaid`, calls `applyOrderAutoRefund` → `applyAdminRefund`).
- Description: `applyOrderAutoRefund` (CF-20: refunds a user automatically when an order fails _after_ Loop already paid CTX) is a thin wrapper around the exact same `applyAdminRefund` primitive admin-issued refunds use, including the same fleet-wide-per-currency-per-UTC-day cap added by CF-06. The cap was designed as a circuit-breaker against a compromised admin session (A2-1610 lineage); CF-20's worker-driven auto-refund consumes the identical budget with no separate accounting.
- Impact: A systemic incident that fails many orders _after_ CTX payment in a single UTC day (a CTX-side outage mid-procurement window, an operator-pool exhaustion event, etc.) would burn down the shared refund cap through legitimate auto-refunds. Once exhausted, **both** further auto-refunds and any admin-issued refunds for the rest of the day throw `DailyAdjustmentLimitError` — turning a procurement-availability incident into a wave of "debited with no gift card and refund blocked" users, which is precisely the failure mode CF-20 exists to prevent. The failure is handled gracefully at the call site (`autoRefundAfterCtxPaid`'s `catch` logs + always pages ops — not a silent drop), but recovery requires either the UTC-midnight reset or a manual env-var bump + redeploy (`ADMIN_DAILY_ADJUSTMENT_CAP_MINOR` is boot-validated in `env.ts`, not one of the `LOOP_KILL_*` runtime-read switches).
- Evidence: `applyOrderAutoRefund` (`refunds.ts:118-137`) calls `applyAdminRefund` directly with no cap bypass; the cap query at `refunds.ts:218-247` has no actor/source discriminator (it sums all `type='refund', currency=X` rows for the day, system- and admin-issued alike).
- Minimal fix: Document the interaction in `docs/runbooks/` (whichever runbook covers CF-20/stranded-order recovery) so an on-call engineer investigating a cap-exhaustion 429 storm knows to check for a concurrent order-failure incident, not just assume fraud.
- Better fix: Give system-issued auto-refunds (`AUTO_REFUND_SYSTEM_ACTOR`) either a separate budget or exemption from the cap (they're driven by Loop's own worker re-checking real order/charge state under a row lock, not an external actor — the fraud threat model doesn't apply to them), and/or make `ADMIN_DAILY_ADJUSTMENT_CAP_MINOR` a live-read value (same pattern as the `LOOP_KILL_*` switches) so ops can raise it without a redeploy during an incident.

### ADM-08 [P3 · LIVE, dead code] `requireAdminStepUp`'s CTX-proxy exemption branch is unreachable given `requireAdmin`'s stricter gate

- File: `apps/backend/src/auth/admin-step-up-middleware.ts:80-86`; `apps/backend/src/auth/require-admin.ts:37-42`.
- Description: `admin-step-up-middleware.ts` carries a documented exemption: `if (auth !== undefined && auth.kind === 'ctx') { return next(); }`, framed as "ADR-028 §Excluded" for legacy CTX-proxy admins. But every `/api/admin/*` route — including every step-up-gated write — is mounted behind `app.use('/api/admin/*', requireAdmin)` (`routes/admin.ts:130`) **before** any route-specific `requireAdminStepUp(...)` runs, and `requireAdmin` itself unconditionally rejects `auth.kind !== 'loop'` with `401 UNAUTHORIZED` (`require-admin.ts:37-42`). So no request with `auth.kind === 'ctx'` can ever reach `requireAdminStepUp` in the first place — the branch is dead in the current architecture, confirmed by `auth/__tests__/require-admin.test.ts`'s own "401 when the auth context is legacy ctx pass-through" test.
- Impact: None functionally (the branch being unreachable is harmless — it can't be exploited since it never executes). But it's misleading: a future refactor that loosens `requireAdmin` (e.g. to allow `staff`-tier CTX-proxy accounts per the ADR-037 staff-roles branch) would silently re-activate this exemption and re-introduce the exact gap ADR-028 says is intentionally excluded only for a narrow, already-closed window ("When ADR-013 Phase C lands and the CTX-proxy path is removed, this exemption goes with it" — implying it's currently load-bearing, which it is not).
- Evidence: `routes/admin.ts:129-130` mount order; `require-admin.ts:37-42`; `auth/__tests__/require-admin.test.ts` ("401 when the auth context is legacy ctx pass-through").
- Minimal fix: Add a one-line comment in `admin-step-up-middleware.ts` noting the branch is currently unreachable given `requireAdmin`'s gate, so a future reader doesn't mistake it for load-bearing.
- Better fix: Remove the branch (or convert it to an assertion that throws/logs if ever hit, as a tripwire for the "requireAdmin loosened" regression class) and correct ADR-028's "Excluded" prose to reflect that CTX-proxy admins cannot reach `/api/admin/*` at all today, not just that they're exempted from step-up specifically.

### ADM-09 [P3 · LIVE, cross-cutting] Step-up signing key validated for length only, not entropy

- File: `apps/backend/src/env.ts:290-298` (`LOOP_ADMIN_STEP_UP_SIGNING_KEY` / `..._PREVIOUS`).
- Description: Both env vars are validated with `.min(32, {...})` and nothing else. A boot-time secret of `'a'.repeat(32)` or any other low-entropy 32-char string passes validation and would let an attacker who guesses/leaks it forge step-up tokens directly (no need to compromise OTP delivery at all).
- Impact: Same class as the Part-6 §37 cross-cutting item for `LOOP_JWT_SIGNING_KEY`; directly applicable here since `auth/admin-step-up.ts` HMAC-signs with this exact key. Low likelihood (requires a careless operator generating the secret by hand rather than via `openssl rand`, which the ADR's own runbook prose recommends), but the validator doesn't defend against it.
- Evidence: `env.ts:290-298`; no entropy/charset-diversity check anywhere in the schema.
- Minimal fix: None required if operational discipline (`openssl rand -base64 48`, per ADR-028's own rotation guidance) is trusted; otherwise, add a cheap charset-diversity or Shannon-entropy floor to the zod refinement.
- Better fix: Shared across every signing-key env var (`LOOP_JWT_SIGNING_KEY`, `LOOP_ADMIN_STEP_UP_SIGNING_KEY`, and their `_PREVIOUS` siblings) — factor a single `secretKeySchema(minLength, minEntropyBits)` helper in `env.ts` rather than fixing this one field in isolation.

### ADM-10 [P2 · LIVE, carried forward from 06-15 P2-9, still open] `pendingResolve` race in `use-admin-step-up.ts`

- File: `apps/web/app/hooks/use-admin-step-up.ts:56-61`.
- Description: The hook's queued-mutation slot is plain `useState<{fn,resolve,reject}|null>`. If `runWithStepUp` is called a second time (e.g. a user double-fires two different destructive actions back-to-back) before the first's step-up modal flow resolves, the second call's `setPendingResolve(...)` overwrites the first entry. The first mutation's `resolve`/`reject` are never invoked — its promise hangs forever (a stuck spinner / disabled button with no error surfaced).
- Impact: Confirmed still present — identical code structure to the 06-15 finding (`useState`, not `useRef`/queue). Low likelihood in normal single-action-at-a-time admin UX, but a real lost-update class bug, not merely cosmetic (a hung `useMutation` never calls `onSettled`, so a `disabled={retrying}` button can get stuck permanently for that session).
- Evidence: `apps/web/app/hooks/use-admin-step-up.ts:24,56-61` — `import { useCallback, useState } from 'react';` and `const [pendingResolve, setPendingResolve] = useState<...>(null);` unchanged from the prior audit's citation.
- Minimal fix: Hold the pending entry in a `useRef`, or reject the previous pending entry with an explicit "superseded" error before overwriting it in `runWithStepUp`.
- Better fix: Same, plus a small integration test firing two `runWithStepUp` calls in sequence before either resolves, asserting the first's promise settles (reject, not hang).

### ADM-11 [P2 · LIVE, carried forward from 06-15 P2-6, still open] Step-up token store is never cleared on logout

- File: `apps/web/app/stores/admin-step-up.store.ts:26-31` (docstring) vs. actual call sites.
- Description: The store's `clear()` docstring says it's "called explicitly on admin logout, on STEP_UP_INVALID / STEP_UP_SUBJECT_MISMATCH responses, and when the UI detects an already-expired token." Confirmed `clearStepUp` is only ever invoked from inside `use-admin-step-up.ts`'s own `runWithStepUp` (on a step-up failure code) — `grep -rn "useAdminStepUpStore" apps/web/app` shows no caller in any logout/session-clear path (`stores/auth.store.ts`, `services/auth.ts`, `hooks/use-auth.ts`, `hooks/use-session-restore.ts`).
- Impact: Bounded — the JWT's own 5-minute `exp` and the backend's `STEP_UP_SUBJECT_MISMATCH` check (different `sub` after a re-login as a different admin on the same tab) are both real backstops. But a logout that doesn't clear the in-memory store leaves a live step-up token sitting in a Zustand store for up to 5 minutes after the admin believes they've logged out — on a shared/kiosk machine that's a real (if short) window where the next person to open the same tab inherits the prior admin's step-up freshness alongside whatever bearer-token gap already exists.
- Evidence: `apps/web/app/stores/admin-step-up.store.ts:26-31` docstring; no logout-path import of `useAdminStepUpStore` anywhere outside `use-admin-step-up.ts` / `api-client.ts` (which only reads it).
- Minimal fix: Call `useAdminStepUpStore.getState().clear()` from whatever the admin logout / cross-tab-logout handler is (the auth-store equivalent action).
- Better fix: Same, plus drive the store's `expiresAtMs` off a single timer-clear effect so a tab left open also self-clears in the UI (not just refuses on the next request).

### ADM-12 [P3 · LIVE, carried forward from 06-15 P2-8, still open] Step-up middleware's `STEP_UP_UNAVAILABLE` (503) branch is untested

- File: `apps/backend/src/auth/__tests__/admin-step-up-middleware.test.ts` (absence of a 503 case).
- Description: `requireAdminStepUp`'s first branch (`admin-step-up-middleware.ts:68-78`) returns 503 when `LOOP_ADMIN_STEP_UP_SIGNING_KEY` is unset. The test file pins the key via `vi.hoisted`/module mock for every test, so this branch is never exercised — confirmed via `grep -n "503\|STEP_UP_UNAVAILABLE\|isAdminStepUpConfigured" admin-step-up-middleware.test.ts` returning no hits.
- Impact: Low-likelihood regression surface (an accidental flip of the configured-check's truthiness, or a future change to `isAdminStepUpConfigured()`'s key-resolution logic) but it is the fail-closed path for the entire surface ("a deployment that never provisions [the key] boots cleanly while every gated endpoint fails closed with 503" per ADR-028's own 2026-06-11 operator-visibility note) — exactly the kind of branch worth a tripwire test given it's been called out twice now (once in ADR-028's own doc, once in the 06-15 audit) without landing a test.
- Evidence: As above.
- Minimal fix: Add one test: unset the env var (or stub `isAdminStepUpConfigured` to return false) and assert 503 `STEP_UP_UNAVAILABLE` with no downstream `next()` call.
- Better fix: Same, plus a matching test in `admin/__tests__/step-up-handler.test.ts` once ADM-02 lands, since `adminStepUpHandler` has the identical 503 branch and needs the same coverage.
