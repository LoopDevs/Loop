### A4-018: Cashback-split rounding residual lands in wholesale (Loop pays CTX more), contradicting `repo.ts` docstring | Medium | 12 | apps/backend/src/orders/cashback-split.ts:119; apps/backend/src/orders/repo.ts:14-18

`wholesaleMinor = faceValueMinor - userCashbackMinor - loopMarginMinor` (cashback-split.ts:119) so the rounding residual after flooring `userCashbackMinor` and `loopMarginMinor` falls into `wholesaleMinor` — i.e. into what Loop pays CTX, not what Loop keeps as margin. The cashback-split.ts comment correctly describes this ("Loop overpays CTX"). The repo.ts comment claims the opposite ("rounding residual lands in Loop's margin — errs toward Loop"). Net effect over millions of orders: Loop's effective wholesale spend is 0–3 minor units per order higher than the configured `wholesale_pct` would imply; Loop's margin is exact, user's cashback is exact-floored. Whether this is intentional (wholesale-as-residual is a defensible policy) or accidental (margin should absorb residual) is an admin decision — but the docs disagree with the code.

### A4-019: Admin refund handler skips `withIdempotencyGuard`; lookup→write→store has a race window vs concurrent retries | High | 07 | apps/backend/src/admin/refunds.ts:111-188

`adminRefundHandler` calls `lookupIdempotencyKey` → `applyAdminRefund` → `storeIdempotencyKey` without the advisory lock that `withIdempotencyGuard` (used by adjustments + withdrawals) provides. Two concurrent retries with the same `(adminUserId, key)` both miss the lookup, both reach `applyAdminRefund`. Bounded by the partial unique index on `(type='refund', reference_type, reference_id)` (schema.ts:205-209) — the second insert hits the constraint and surfaces 409 — but the same key against TWO DIFFERENT order ids would write two rows. Operator misuse that re-uses keys across distinct refund intents passes both. Same shape as A4-003 (payouts retry).

### A4-020: Admin payout-compensation primitive bypasses the daily admin adjustment cap | Medium | 12 | apps/backend/src/credits/payout-compensation.ts:122-133; apps/backend/src/credits/adjustments.ts:108-135

`applyAdminPayoutCompensation` writes `type='adjustment'` with `referenceType='payout'`, `referenceId=payoutId`. The `ADMIN_DAILY_ADJUSTMENT_CAP_MINOR` enforcement query in `applyAdminCreditAdjustment` filters on `referenceType='admin_adjustment'` AND `referenceId=adminUserId`, so compensation rows neither contribute to nor are bounded by the cap. A compromised admin token can drain treasury via compensation beyond the per-day cap.

### A4-021: `applyAdminPayoutCompensation` does not verify `amountMinor` matches the locked payout's stroop value | Medium | 12 | apps/backend/src/credits/payout-compensation.ts:80-167

### A4-072: Production web bundle ships WITHOUT Sentry — `VITE_SENTRY_DSN` is never plumbed | High | 19 | apps/web/Dockerfile:20-21; apps/web/fly.toml:5-6

`apps/web/app/root.tsx:42` gates `Sentry.init()` on `import.meta.env.VITE_SENTRY_DSN`. Vite bakes `import.meta.env.VITE_*` at BUILD time. The web Dockerfile only declares one `ARG VITE_API_URL=...`; the web fly.toml `[build.args]` only sets `VITE_API_URL`. Neither `VITE_SENTRY_DSN`, `VITE_SENTRY_RELEASE`, nor `VITE_LOOP_ENV` is plumbed through. So `import.meta.env.VITE_SENTRY_DSN` is undefined in every Fly-built bundle, the if-guard fails, and `Sentry.init` is never called. Every subsequent `Sentry.captureException(...)` call on the web side becomes a no-op — the TanStack Query → Sentry forward (root.tsx:91), the React Router ErrorBoundary capture (root.tsx:580), the `ClusterMap` capture (ClusterMap.tsx:143). All elaborate frontend Sentry scrubbing is dead code in production. Frontend errors do not reach observability at all. Pair with A4-051 — even if Sentry were initialised, the scrubber wouldn't walk the exception message — but the more serious failure is that NOTHING reaches Sentry. **Severity: High** (every frontend error invisible to ops; the operator believes the SDK is working because dev environment with `.env.local` ships fine). Fix: add `ARG VITE_SENTRY_DSN` / `VITE_SENTRY_RELEASE` / `VITE_LOOP_ENV` to the Dockerfile, plumb `ENV` lines for each, and add `[build.args]` entries to `apps/web/fly.toml` (the secrets must be wired at build, not deploy, because Vite bakes them into the static bundle).

### A4-073: Backend Docker `HEALTHCHECK` + Fly `/health` probe both treat HTTP 200 as healthy, but `/health` returns 200 even when degraded | High | 19 | apps/backend/Dockerfile:60-62; apps/backend/fly.toml:[[http_service.checks]]; apps/backend/src/health.ts:206

`apps/backend/Dockerfile:61-62` HEALTHCHECK runs `node -e "require('http').get(...)/health, r => process.exit(r.statusCode === 200 ? 0 : 1))"`. Fly's `[[http_service.checks]]` does the same shape. Both treat HTTP 200 as healthy. But `healthHandler` in health.ts:206 ALWAYS returns `c.json({...status: 'degraded' or 'healthy', ...})` with default status 200 — the `degraded` field is in the body, not the HTTP status. So a backend container with stale merchants, stale locations, unreachable upstream, OR degraded workers (per `runtime-health.ts`) STILL returns HTTP 200 to the orchestrator. Fly never cycles the machine; Docker never reports unhealthy. Compounds A4-034 (no DB probe) and A4-035: a backend container with a dead DB connection pool can stay in rotation indefinitely until manually noticed. Combined risk on the financial-ops surface is significant. **Severity: High**. Fix: return 503 from `healthHandler` when `degraded` is true; verify Docker HEALTHCHECK + Fly probe both flip on the new status.

### A4-074: Sentry scrubber does NOT walk `event.exception.values[].value`; PII in error messages reaches Sentry uncensored regardless of capture path | High | 17 | apps/backend/src/sentry-scrubber.ts:45-67; apps/web/app/utils/sentry-error-scrubber.ts (web equivalent only runs on the ErrorBoundary path)

`scrubSentryEvent` walks `event.request`, `event.extra`, `event.contexts`, `event.tags`, `event.cookies`. It does NOT walk `event.exception.values[].value` — which is where `Error.message` lands when `Sentry.captureException(err)` is called. Any error message containing email, OTP, JWT, refresh token, Stellar secret, idempotency key, or other PII reaches Sentry uncensored on EVERY capture path on the backend (where the scrubber is the only defense — there is no per-call argument scrubber on the backend). On the web side, `scrubErrorForSentry` runs only on the ErrorBoundary path (root.tsx:580); the TanStack Query forward, the ClusterMap fallback, and any future `Sentry.captureException(err)` call site bypass it. **Severity: High**. Fix: extend `scrubSentryEvent` to walk the exception-value list and apply the same regex-based string scrubbing the web's `scrubStringForSentry` already uses for emails / bearer tokens / Stellar secrets / hex tokens. Also fix at the call-site level: route every `Sentry.captureException(err)` through `scrubErrorForSentry(err)` first (apply the lint-docs guard).

### A4-110: Cashback double-spend — `loop_asset` order payment doesn't debit `user_credits`, AND `paymentMethod='credit'` drains cashback-source credit without requiring on-chain return | Critical | 12 | apps/backend/src/orders/fulfillment.ts:86-110,143-160; apps/backend/src/orders/transitions.ts:29-44 (markOrderPaid); apps/backend/src/orders/repo-credit-order.ts:56-122; apps/backend/src/payments/watcher.ts:175 (markOrderPaid call)

**Intended financial model** (per operator clarification 2026-05-03): cashback is denominated in LOOP-asset (USDLOOP/GBPLOOP/EURLOOP). At fulfilment, both halves are written intentionally — `user_credits.balanceMinor +X` is Loop's internal liability ledger, and the on-chain LOOP-asset sent to the user's Stellar wallet is the user-held proof. Two halves exist for reconciliation. The redemption rule: **to spend their cashback, the user must send their on-chain LOOP-asset back to Loop**, which then debits `user_credits` (extinguishes the liability) and provides the value (gift card or fiat). Both halves should be debited together; neither should be drainable independently.

**The code violates the rule on two paths:**

**(a) `loop_asset`-method gift-card purchase:** the user sends LOOP-asset on-chain to Loop's deposit address (correct under the model). The watcher detects the inbound payment via `findPendingOrderByMemo` and calls `markOrderPaid` (transitions.ts:29-44), which only updates `state`, `paid_at`, and `payment_received_at`. **`user_credits` is NOT debited** anywhere on this path. The user has burned X LOOP-asset on-chain but their `user_credits.balanceMinor` still reads the pre-purchase value. They can subsequently use `paymentMethod='credit'` for another gift card OR be granted a withdrawal that pays out the same X again.

**(b) `paymentMethod='credit'` purchase:** `repo-credit-order.ts:56-122` debits `user_credits.balanceMinor` directly via SQL with NO on-chain operation required. This contradicts the "must send on-chain to spend" rule entirely. `user_credits` is fungible across all positive sources (cashback, refund, adjustment, interest), so the credit method drains the cashback-tagged portion of the balance even though the user is still holding the matching on-chain LOOP-asset in their wallet, which they can then spend separately via `loop_asset` OR present for a withdrawal.

**Either path lets a user redeem the same cashback X twice.** Path (a) is the more visible exploit because it requires no special tooling — any user with a linked wallet who places an order via `loop_asset` and later places another via `credit` (or requests a withdrawal) gets 2X for one cashback event. Path (b) is the same root cause from the other end.

**Severity: Critical** (every fulfilled cashback with a linked-wallet user is exposed; no race or privileged access; only requires payout workers + issuers to be configured).

**Remediation under the operator's model:**

1. **`markOrderPaid` for `loop_asset`-method orders** (or the watcher path) must atomically write a negative `credit_transactions` row (`type='spend'`, amount = `-order.chargeMinor`, currency = chargeCurrency, referenceType='order', referenceId=orderId) AND decrement `user_credits.balanceMinor` by the same amount, inside the same transaction as the state flip. This extinguishes the off-chain liability when the on-chain proof returns.
2. **`paymentMethod='credit'` must either be removed from the user-facing flow** (reserve it for refund/adjustment-funded balances that have no on-chain mirror — though this only works if a way exists to tag credit by source), **or every cashback-source credit must be tagged separately** so that credit-method spend can only drain non-cashback credit (refunds, admin adjustments, interest). Today `user_credits` has no source bucket; this requires schema work or a "spend-from" enum on the credit-method handler.

Pair with A4-111 (Phase 2b withdrawal-direction inversion) — the symmetric fix on the withdrawal path.

**Verification after remediation:**

- Add an integration test: user has X LOOP-asset on-chain + X user_credits. They pay for a gift card via `loop_asset`. Assert `user_credits` is debited by the order's chargeMinor and a matching `type='spend'` ledger row exists.
- Add an integration test: user with cashback-source credit attempts a `paymentMethod='credit'` purchase. Either it's rejected (option (a) — credit removed) or it drains only the non-cashback portion (option (b) — tagged credit).
- Add a financial-invariant test: total `pending_payouts` confirmed amount + total `user_credits` balance + total spend ledger - total cashback ledger = 0. Currently this invariant fails under linked-wallet usage.
