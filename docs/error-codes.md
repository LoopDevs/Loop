# Error-code taxonomy

Single source of truth for every `{ code, message }` response the
Loop backend emits. Kept in sync with `packages/shared/src/api.ts`
(`ApiErrorCode` enum) ‚Äî the enum is the TypeScript surface, this
doc is the operator + integrator reference.

Closes A2-1011. Previously, handlers shipped codes ad-hoc and
client switch-ladders tracked drift informally in code review.

## How to read the table

- **Code** ‚Äî the literal string in the response body's `code` field.
- **Status** ‚Äî the HTTP status the handler pairs with it. A client
  should branch on `code`, not status ‚Äî multiple codes can share a
  status (e.g. several 400s), and status alone is coarser than
  needed for UX branching.
- **Where** ‚Äî the handler families that emit it. Not exhaustive;
  `grep "code: 'X'"` gives the current list.
- **Client guidance** ‚Äî what the web / mobile UX is expected to do
  when it receives this code. "Retryable" = safe to try again after
  some backoff; "terminal" = don't retry without fixing input /
  state first.

## Request-validation family (400)

| Code                                      | Status  | Where                                                                                                                                                                                                   | Client guidance                                                                                                                                                                                                                                                                           |
| ----------------------------------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `VALIDATION_ERROR`                        | 400     | Every handler ‚Äî generic body / param / query validation                                                                                                                                                 | Terminal. Surface `message` to the user; fix input before retry.                                                                                                                                                                                                                          |
| `IDEMPOTENCY_KEY_REQUIRED`                | 400     | All admin writes (ADR 017)                                                                                                                                                                              | Terminal. The client is missing the `Idempotency-Key` header or sent one outside the 16-128 char range. Generate a UUID + retry.                                                                                                                                                          |
| `HOME_CURRENCY_LOCKED`                    | 409     | `POST /api/users/me/home-currency` after first order                                                                                                                                                    | Terminal. User placed an order; support must unlock. Surface the "contact support" path.                                                                                                                                                                                                  |
| `INSUFFICIENT_CREDIT`                     | 400     | `POST /api/orders/loop` with `paymentMethod=credit`                                                                                                                                                     | Terminal. User's cashback balance is below the order amount. Prompt them to top up via another method.                                                                                                                                                                                    |
| `CREDIT_METHOD_RETIRED`                   | 400     | `POST /api/orders/loop` with `paymentMethod=credit` by a wallet-activated user (ADR-036 OQ3)                                                                                                            | Terminal. The balance IS the tokens once the wallet is activated ‚Äî spend via token redemption (`loop_asset` / `POST /api/orders/loop/:id/redeem`).                                                                                                                                        |
| `LOOP_ASSET_UNAVAILABLE_PHASE_1`          | 400     | `POST /api/orders/loop` with `paymentMethod=loop_asset`, and `POST /api/orders/loop/:id/redeem` for any `loop_asset` order, while `LOOP_PHASE_1_ONLY=true` (AUDIT-2 finding B, 2026-07 hardening)       | Terminal while the flag is on. loop_asset is a Phase-2 spend surface (redemption of on-chain LOOP); use `credit`, `xlm`, or `usdc` instead. Goes away when `LOOP_PHASE_1_ONLY` flips to `false`.                                                                                          |
| `INSUFFICIENT_BALANCE`                    | 400     | `POST /api/admin/users/:userId/emissions` (ADR-024/036); `POST /api/orders/loop/:id/redeem` (ADR-030 C3) when the on-chain LOOP balance is below the charge                                             | Terminal. Admin path: emission larger than the mirror balance. Redeem path: prompt the user to pay another way.                                                                                                                                                                           |
| `ORDER_NOT_PAYABLE`                       | 400     | `POST /api/orders/loop/:id/redeem` (ADR-030 C3)                                                                                                                                                         | Terminal. The order is not a `loop_asset` order awaiting payment (wrong method, or terminal failed/expired state). Already-paid states replay 200.                                                                                                                                        |
| `PAYMENT_IN_FLIGHT`                       | 400     | `POST /api/orders/loop/:id/redeem` (ADR-030 C3)                                                                                                                                                         | Retryable after the in-flight call resolves (seconds). A concurrent redeem for the same order is being processed.                                                                                                                                                                         |
| `WALLET_NOT_ACTIVATED`                    | 400     | `POST /api/orders/loop/:id/redeem` (ADR-030 C3)                                                                                                                                                         | Retryable once provisioning completes. The embedded wallet isn't `activated` yet ‚Äî poll `GET /api/me/wallet` and re-offer the button when it is.                                                                                                                                          |
| `REFUND_ALREADY_ISSUED`                   | 409     | `POST /api/admin/users/:userId/refunds`                                                                                                                                                                 | Terminal. DB partial unique index caught a duplicate refund for the same order. Surface the existing refund's id.                                                                                                                                                                         |
| `DEPOSIT_NOT_REFUNDABLE`                  | 409     | `POST /api/admin/deposits/:paymentId/refund` (A6) ‚Äî the skip is not `abandoned`, its stored payment is unparseable, or it has no sender/amount, or the operator Stellar signer is unconfigured          | Terminal for this deposit. Not an abandoned late deposit (or missing data). Surface the detail; don't retry blindly.                                                                                                                                                                      |
| `REFUND_SUBMIT_FAILED`                    | 502     | `POST /api/admin/deposits/:paymentId/refund` (A6) ‚Äî the Stellar submit failed; the claim is released back to `abandoned`                                                                                | Retryable. Transient Horizon/network or a rebuild-needed submit error ‚Äî re-trigger the refund; idempotency (CF-18 hash) prevents a double-send.                                                                                                                                           |
| `ORDER_NOT_REDRIVABLE`                    | 400     | `POST /api/admin/orders/:orderId/redrive` (A5-1) ‚Äî order is not `paid`                                                                                                                                  | Terminal for the current order state. `fulfilled`/`failed`/`expired` orders are already terminal; `pending_payment` orders haven't been paid yet ‚Äî nothing to redrive. (`procuring` has its own 409.)                                                                                     |
| `ORDER_NOT_REFUNDABLE`                    | 400/409 | `POST /api/admin/orders/:orderId/refund` (A5-4) ‚Äî order is `pending_payment`/`expired` (400), a not-yet-stale `procuring` order still actively procuring (400), or its state changed concurrently (409) | Terminal-ish. `pending_payment`/`expired` have nothing to reverse. A fresh `procuring` order may have a live worker mid-flight paying CTX ‚Äî wait for it to fulfil/fail or age into the 15-min recovery sweep. On a concurrent state change, refresh + retry with a fresh Idempotency-Key. |
| `ORDER_REFUND_ATTESTATION_REQUIRED`       | 400     | `POST /api/admin/orders/:orderId/refund` (A5-4) ‚Äî a `fulfilled` order with no attestation in the body                                                                                                   | Terminal until re-sent. Refunding a fulfilled order (customer already has the code) requires a code-unused attestation: `{ codeUnused: true, attestationNote }`. Confirm the code is unused, then retry.                                                                                  |
| `ORDER_ALREADY_REFUNDED`                  | 409     | `POST /api/admin/orders/:orderId/refund` (A5-4) ‚Äî a refund already exists for this order (INV-8)                                                                                                        | Terminal. The order was already refunded (the migration-0013 partial unique index / the deposit-refund claim guard caught the duplicate). Surface the existing refund.                                                                                                                    |
| `ORDER_REFUND_CTX_ALREADY_PAID`           | 409     | `POST /api/admin/orders/:orderId/refund` (A5-4) ‚Äî the order is `procuring` and Loop has already paid CTX                                                                                                | Terminal for this endpoint. Refunding now would double-lose money (CTX paid AND the customer refunded). Escalate for manual reconcile; a genuinely-stuck row is resolved by the recovery sweep.                                                                                           |
| `ORDER_REFUND_UNSUPPORTED_PAYMENT_METHOD` | 409     | `POST /api/admin/orders/:orderId/refund` (A5-4) ‚Äî the order is `loop_asset`-funded                                                                                                                      | Terminal for this endpoint. `loop_asset` refunds need coordinated on-chain re-mint/re-credit (not built ‚Äî matches the R3-2 fail-closed posture). Escalate for a manual money-review refund.                                                                                               |
| `ORDER_REFUND_SUBMIT_FAILED`              | 502     | `POST /api/admin/orders/:orderId/refund` (A5-4) ‚Äî the on-chain refund-to-sender submit failed, or the order predates the payment-snapshot columns                                                       | Retryable. Transient Horizon/network, or a pre-migration order with no payment snapshot (refund manually). The order is already fenced to `failed`, so a retry via this endpoint is safe.                                                                                                 |
| `PAYLOAD_TOO_LARGE`                       | 413     | bodyLimit middleware (A2-1005)                                                                                                                                                                          | Terminal. Body exceeded 1 MB. Client should split the payload.                                                                                                                                                                                                                            |

## Auth family (401 / 403 / 404)

| Code                       | Status | Where                                                                                                                                 | Client guidance                                                                                                                          |
| -------------------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `UNAUTHORIZED`             | 401    | `requireAuth` middleware, admin handlers without actor context                                                                        | Retryable after re-auth. Trigger refresh-token rotation, then retry once. If still 401, sign the user out.                               |
| `STEP_UP_REQUIRED`         | 401    | `requireAdminStepUp` gate (ADR 028) ‚Äî no `X-Admin-Step-Up` header                                                                     | Re-prompt the admin to confirm (OTP), mint a step-up token, replay the request with the header set.                                      |
| `STEP_UP_INVALID`          | 401    | `requireAdminStepUp` ‚Äî token malformed / expired / wrong signature / wrong audience, or no Loop-native subject to pin against         | Same flow as `STEP_UP_REQUIRED`: re-prompt and replay. If the session is legacy CTX-proxy auth, sign in through Loop-native auth first.  |
| `STEP_UP_SUBJECT_MISMATCH` | 401    | `requireAdminStepUp` ‚Äî the token's `sub` is a different admin than the bearer                                                         | Sign the admin out ‚Äî a step-up minted for another session was replayed.                                                                  |
| `STEP_UP_PURPOSE_MISMATCH` | 401    | `requireAdminStepUp(action)` (CF-08) ‚Äî a token narrowed to one action class was replayed against a different one                      | Re-prompt and mint a step-up for the correct action, then replay.                                                                        |
| `NOT_FOUND`                | 404    | Merchant / order / payout not found. Also: feature-flag-off (`LOOP_AUTH_NATIVE_ENABLED=false` surfaces the Loop-native routes as 404) | Terminal. Don't retry ‚Äî the resource doesn't exist or the caller isn't authorised to see it (enumeration-defence masks the distinction). |
| `ORDER_NOT_FOUND`          | 404    | `POST /api/admin/users/:userId/refunds` (CF-06) ‚Äî the bound order id matches no order                                                 | Terminal. Verify the order id before retrying the refund.                                                                                |
| `USER_NOT_FOUND`           | 404    | `POST /api/admin/users/:userId/home-currency` (ADR 015 support-mediated change) ‚Äî the path user id matches no user                    | Terminal. Verify the user id before retrying.                                                                                            |
| `MERCHANT_NOT_FOUND`       | 404    | `POST /api/users/me/favorites` ‚Äî the merchant id is not in the live catalog                                                           | Terminal. The merchant id is stale or evicted; refresh the catalog before retrying.                                                      |

## Well-known verification family (404)

| Code                        | Status | Where                                                                                                                                                                  | Client guidance                                                                                                                                                                                                                                                                                                                           |
| --------------------------- | ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `WELL_KNOWN_NOT_CONFIGURED` | 404    | `GET /.well-known/apple-app-site-association` when `APPLE_TEAM_ID` is unset; `GET /.well-known/assetlinks.json` when `ANDROID_CERT_SHA256` is unset (M-3 deep linking) | Terminal until ops sets the gating env var post-enrollment (go-live-plan L1-4 / L1-5). ## Rate-limit family (429)

| Code                              | Status | Where                                                                                                                                                                                                                                                                                                                                                                                                      | Client guidance                                                                                                                                                |
| --------------------------------- | ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `RATE_LIMITED`                    | 429    | Every rate-limited route (see AGENTS.md middleware stack entry #6)                                                                                                                                                                                                                                                                                                                                         | Retryable. Respect `Retry-After`. The limiter uses a fixed-window counter per IP ‚Äî a burst that hits the limit waits the remainder of the window.              |
| `DAILY_LIMIT_EXCEEDED`            | 429    | `POST /api/admin/users/:userId/credit-adjustments` (per-admin cap on signed adjustments), `POST /api/admin/payouts/:id/compensate` (fleet-wide A4-020 compensation cap), `POST /api/admin/users/:userId/refunds` (fleet-wide CF-06 refund cap), and `POST /api/admin/users/:userId/emissions` (hardening A1 ‚Äî fleet-wide per-currency cap; failed emission rows created today still count against the day) | Terminal for the current UTC day unless an operator raises the cap; retrying the same write will not help.                                                     |
| `TOO_MANY_ATTEMPTS`               | 429    | `POST /api/auth/verify-otp` (native path) ‚Äî hardening B5. The per-EMAIL failed-attempt counter crossed its threshold (10 wrong guesses per 15-min window), so verify is locked for that email regardless of how many fresh codes are requested. Decoupled from the per-IP rate limit and from OTP rows.                                                                                                    | Retryable after the lockout. Respect `Retry-After` (15 min). A successful verify (or the window lapsing) clears the counter.                                   |
| `OTP_LOCKOUT_CLEAR_RATE_EXCEEDED` | 429    | `POST /api/admin/users/:userId/clear-otp-lockout` (A5-3). The PER-TARGET velocity cap: this account's B5 lockout has already been cleared `CLEAR_LOCKOUT_MAX_PER_TARGET_PER_DAY` (5) times in the trailing 24h. This is the control that bounds a compromised admin bearer's clear‚Üíguess‚Üíclear loop against one account (the per-IP route limit can't).                                                    | Terminal for ~24h for this target. If a legitimate user is still locked out after 5 clears in a day, escalate rather than retry ‚Äî repeated clears won't apply. |

## Client-only codes (never sent by the backend)

Two entries on the shared enum exist for the client's own fetch
layer so its error path carries the same switch-ladder shape:

- `NETWORK_ERROR` ‚Äî transport-level failure (DNS, connection refused).
- `TIMEOUT` ‚Äî client-side `AbortSignal.timeout` tripped.

## Adding a new code

1. Add the literal to `ApiErrorCode` in `packages/shared/src/api.ts` ‚Äî grouped under the appropriate category comment block, with a trailing inline comment if the handler-side rationale isn't obvious.
2. Emit it from the handler as `{ code: 'XYZ', message: '...' }` paired with an HTTP status consistent with its row in this table (or add a new row if it's a new family).
3. Register the response in `apps/backend/src/openapi.ts` against the endpoint ‚Äî status code, ErrorResponse schema, description that names the code literal. The A2-1507 drift check (`scripts/lint-docs.sh` ¬ß9) catches handlers that ship without openapi registration.
4. Update this doc. A missing row here is a tracker-sweep item waiting to happen.
