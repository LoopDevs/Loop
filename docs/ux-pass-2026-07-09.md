# Full customer-journey UX/visual pass — 2026-07-09

> Closes **U-1** (`docs/readiness-backlog-2026-07-03.md` Tier 10). A systematic
> walk of the customer journey at desktop (1440×900) and mobile (390×844)
> viewports, screenshotting + noting every broken / empty / loading / error
> state, per the U-1 surface list.

## Method

Two environments, per the U-1 instructions:

- **Live beta** (`https://beta.loopfinance.io`, API `api.loopfinance.io`) —
  **read-only** browse of Home, Directory/cashback, Map, Search, Merchant
  detail, Onboarding (view only, no OTP submit), Auth screen (view only),
  Calculator, locale variants (`/us/en`, `/gb/en`, `/ca/en`), 404. No
  sign-up, no order creation. This is the real production data/imagery
  state users see today.
- **Local mocked stack** (`tests/e2e-mocked/` harness — mock-ctx `:9091`,
  backend `:8081` with `NODE_ENV=test` + `DATABASE_URL` on a scratch
  Postgres, web dev server `:5174`, per `playwright.mocked.config.ts` /
  `tools/load-test/run-local.sh`) — interactive walk of Onboarding
  (email → OTP → wrong-OTP error → "use a different email"), the inline
  purchase-card auth (email → OTP → wrong-OTP error → correct OTP), amount
  selection, the payment step (address/QR/memo/countdown), redemption
  reveal, order history + order detail, and Settings/privacy
  (unauthenticated gate + authenticated).

A throwaway Playwright script drove each pass (not committed — screenshots
and scripts stayed in the session scratchpad, per the task instructions).
~70 screenshots were captured and reviewed directly.

**Mock vs real:** every finding below reproduces on **live beta** unless
explicitly marked "mock-stack only." The mocked stack's nav shows an extra
"Rates" item and un-gated cashback copy because the local harness doesn't
set `LOOP_PHASE_1_ONLY=true` — that specific nav difference is a harness
artifact, not a product bug (noted inline where relevant). One suspected
bug (a fixed bottom tab bar appearing to overlap mid-page content) was
investigated and **ruled out** as a `fullPage` Playwright screenshot
stitching artifact for `position: fixed` elements — verified separately
with a real scroll-to-bottom screenshot, which shows correct clearance.

**Known, already-tracked issue — not re-filed here:** brand/product
imagery renders as a blue placeholder + single-letter fallback across the
home page, search results, and product pages (both US and GB locales,
both viewports). This is **T0-2** (`docs/readiness-backlog-2026-07-03.md`
Tier 0, diagnosis-gated on creds) and reproduced identically in this pass
— e.g. `/us/en` "Top cashback rates" cards, `/gb/en/gift-card/xbox-game-pass-ultimate-3m-gb`.
No new information beyond confirming it's still open as of 2026-07-09;
see T0-2 for the fix.

## Findings

### UX-01 · Onboarding marketing copy unconditionally promises Phase-2 cashback — P1

- **Surface:** Onboarding (native trust screens + web `/onboarding`)
- **Viewport:** both
- **Repro:** `https://beta.loopfinance.io/onboarding` (or any locale — the
  screen is not locale-scoped), first screen. Also reproduces identically
  on the mocked stack.
- **Description:** The onboarding slideshow's first screen shows a mocked
  "TOTAL CASHBACK $2,847.00" card under the headline "Shop. Save. Repeat."
  with the subcopy _"The smart way to pay — earn cashback on every
  purchase, paid by instant bank transfer."_ This is Phase-2 language
  (cashback + bank-transfer settlement) shown to every new sign-up on a
  Phase-1 (discount-gift-card) build. It's inconsistent with the home
  page, which correctly branches its hero copy on `phase1Only`
  (`apps/web/app/routes/home.tsx:64,73,143,146,160,240` — Phase 1 shows
  "Up to 15% off, instantly" / "Save up to 15% instantly," not cashback
  promises).
  - Root cause: `COPY[1]` in
    `apps/web/app/components/features/onboarding/Onboarding.tsx:34-38`
    (`"Shop. Save.\nRepeat."` / "earn cashback on every purchase, paid by
    instant bank transfer") feeds `TrustWelcome`, which both the native
    `Onboarding.tsx` (`renderStep`, idx 0) and the web
    `OnboardingDesktop.tsx`'s `SlidePanel` render unconditionally. The
    existing `phase1Only` skip logic
    (`Onboarding.tsx:354-359`) only skips the currency-picker (step 5)
    and wallet-intro (step 7) screens — it never touches the trust-screen
    copy (steps 0-2, `COPY[1]`/`COPY[2]`/`COPY[3]`), and
    `OnboardingDesktop.tsx`'s `SlidePanel` doesn't reference `phase1Only`
    at all.
  - Impact: brand-new users' very first interaction with the product
    over-promises a feature (cashback paid by bank transfer) that isn't
    live yet — a real expectations/trust risk right at the top of the
    signup funnel.
- **Suggested fix:** branch `COPY[1]`/`COPY[2]`/`COPY[3]` (or the
  `TrustWelcome`/`TrustHowItWorks`/`TrustMerchants` screens themselves) on
  `phase1Only`, mirroring the copy home.tsx already uses for Phase-1 vs
  Phase-2 ("save up to X% instantly" vs "earn cashback"). Filed as **U-2**.

### UX-02 · `/calculator` is reachable in Phase 1 and shows a misleading empty state — P1

- **Surface:** Cashback calculator
- **Viewport:** both
- **Repro:** `https://beta.loopfinance.io/us/en/calculator` (also
  `/calculator` top-level).
- **Description:** Unlike `/cashback`, which correctly Phase-1-gates with
  a clean "Coming soon — this part of Loop is under construction" message
  (verified working correctly at `/us/en/cashback`), `/calculator` has no
  such gate: `apps/web/app/routes/calculator.tsx` contains no
  `phase1Only`/`Phase2Gate` check. It renders live, with the heading
  "Cashback calculator" and subcopy _"Pick a merchant and see what you'd
  earn on Loop — paid in LOOP-asset stablecoin you can spend on your next
  order,"_ followed by **"No merchants available right now. Check back
  shortly."** — which reads as a temporary data glitch, not "this feature
  isn't launched yet." A Phase-1 user who finds this page (it's linked
  from the merchant-detail footer nav) gets a broken-looking dead end
  instead of the same clear "coming soon" messaging `/cashback` already
  has.
- **Suggested fix:** wrap `/calculator` in the same `Phase2Gate` (or
  equivalent `phase1Only` redirect/coming-soon render) used by
  `/cashback`. Filed as **U-3**.

### UX-03 · `/calculator` heading renders clipped under the sticky navbar — P2

- **Surface:** Cashback calculator
- **Viewport:** desktop (not checked on mobile — page is Phase-2 and low
  priority; verify on the same pass as UX-02's fix)
- **Repro:** `https://beta.loopfinance.io/us/en/calculator`, page load,
  no scroll.
- **Description:** The "Cashback calculator" H1 renders with its top
  ~15px behind/under the fixed navbar on initial load — the tops of the
  letters are visibly cut off. Other routes account for the fixed
  navbar's height with top padding; this route appears to be missing it.
- **Suggested fix:** add the standard top-padding/offset used by sibling
  routes so the heading clears the fixed navbar. Small enough to bundle
  with the UX-02 fix (U-3) rather than filing separately.

### UX-04 · Home page renders the entire merchant catalog unpaginated — P2

- **Surface:** Home
- **Viewport:** desktop (mechanism is identical on mobile; the desktop
  grid duplicate inflates the effective DOM count further — see UX-tooling
  note below)
- **Repro:** `https://beta.loopfinance.io/us/en`, scroll to bottom, or
  inspect `document.body.scrollHeight`.
- **Description:** The "All merchants" section below "Top cashback rates"
  dumps the **entire** catalog into one grid with no pagination, "load
  more," or virtualization. Measured live: **2,416** `a[href*="/gift-card/"]`
  nodes in the DOM and a **95,971px**-tall page on a 1440×900 viewport
  (about 107 screen-heights of scroll). Some of that count is inflated by
  the desktop grid staying in the DOM alongside the mobile carousel (see
  UX-tooling note in Method — both match the same selector), but even
  halved this is roughly 1,200 simultaneously-mounted card components
  with images, well past what a "browse the catalog" page needs to render
  eagerly.
- **Impact:** heavy initial DOM/paint cost, especially on lower-end
  mobile hardware; makes "networkidle" essentially unreachable in normal
  browsing (dozens of image requests keep firing well after the fold is
  interactive) and likely compounds any image-loading issues (T0-2) by
  sheer request volume.
- **Suggested fix:** paginate or virtualize the "All merchants" grid
  (windowing library, or a "Load more" button / infinite scroll with a
  real fetch boundary instead of client-side over-render).

### UX-05 · Navbar search input has no accessible name — P2

- **Surface:** Search (desktop navbar)
- **Viewport:** desktop
- **Repro:** any page ≥ lg width; inspect the "Search brands" input in
  `apps/web/app/components/features/Navbar.tsx:187-198`.
- **Description:** The input has `placeholder="Search brands"` but no
  `aria-label`, `aria-labelledby`, or associated `<label>` — grepping the
  whole file, the only `aria-label` present is on the unrelated "Account
  menu" button. Placeholder-only naming is a known accessibility
  anti-pattern (many screen readers don't reliably announce it as the
  field's name, and it disappears once typing starts). Confirmed
  empirically: `page.getByLabel(...)` cannot find this input by any
  accessible-name query in Playwright either.
- **Suggested fix:** add `aria-label="Search brands"` (or wire a visually-hidden
  `<label>`) to the input — a one-line fix. Ties into the broader B-2
  accessibility backlog item; not filed as its own tracker item given the
  small scope.

### UX-06 · Search shows no "no results" state — P2

- **Surface:** Search (desktop navbar)
- **Viewport:** desktop (mobile's separate `MobileHome` search — see note
  below — likely shares the same underlying results-list component;
  not independently re-verified)
- **Repro:** desktop navbar search, type a query with no matches (e.g.
  `zzzzzznonexistentmerchant`).
- **Description:** The results dropdown is gated entirely on
  `results.length > 0` (`Navbar.tsx:238`) — there's no `else` branch. When
  a query matches nothing, the dropdown just doesn't render: no "No
  results for…" message, nothing. This is indistinguishable from "the
  search hasn't run yet," which reads as the feature being unresponsive
  rather than having searched and found nothing.
- **Suggested fix:** render an explicit empty state ("No brands match
  '{query}'") when `open && results.length === 0 && query.length > 0`.

### UX-07 · Web onboarding has no explicit "resend code" action — P2

- **Surface:** Onboarding (web `/onboarding`, desktop split-layout)
- **Viewport:** desktop (the form-only mobile/`<lg` layout shares the
  same `OnboardingDesktop` OTP step, so this applies there too)
- **Repro:** `https://beta.loopfinance.io/onboarding` → enter email → OTP
  step.
- **Description:** The web onboarding OTP screen
  (`apps/web/app/components/features/onboarding/OnboardingDesktop.tsx:205-215`)
  offers only **"Use a different email,"** which resets to the email step
  (forcing a full re-entry + re-submit) — there's no direct "resend to
  the same address" action. The native/mobile multi-screen `Onboarding.tsx`
  has an explicit `handleResend` wired to a "Resend code" button
  (`signup-tail.tsx:299`). The inline purchase-card sign-in form (used
  from merchant-detail pages) has neither an explicit label match for
  "Continue"/"Verify" the way this analysis first assumed — worth noting
  the two auth surfaces (`/onboarding` vs. the merchant-card inline form)
  use different button copy ("Send verification code" / "Verify &
  continue" vs. "Continue" / "Verify"), a minor copy inconsistency
  alongside the missing resend action.
- **Suggested fix:** add a "Resend code" action to `OnboardingDesktop`'s
  OTP step, reusing `useOnboardingAuth`'s `sendOtp` the way the native
  flow's `handleResend` does. Low priority — "use a different email" with
  the same address works as a de facto resend, just with worse copy/UX.

### UX-08 · Map view's initial viewport isn't locale-aware — P2

- **Surface:** Map
- **Viewport:** desktop (not independently re-checked on mobile, but the
  underlying map-init logic is shared)
- **Repro:** compare `https://beta.loopfinance.io/us/en/map` and
  `https://beta.loopfinance.io/ca/en/map` — both center/zoom on the exact
  same North-America-wide view.
- **Description:** Clustering itself works well (numbered cluster
  markers, branded pins at higher zoom — see "What looked good"), but
  switching the country selector to Canada doesn't re-center or zoom the
  map toward Canada — a CA user gets the identical initial view a US user
  does, just with the header flag/currency swapped.
- **Suggested fix:** derive the map's initial center/zoom from the active
  locale's country (a rough per-country centroid + zoom table would be
  enough; doesn't need to be precise).

### UX-09 · "Welcome back, there" personalization shown to anonymous visitors — P2

- **Surface:** Home (mobile card layout, `MobileHome.tsx`)
- **Viewport:** mobile
- **Repro:** `https://beta.loopfinance.io/us/en` at a mobile viewport,
  fresh/logged-out session.
- **Description:** The mobile home hero always renders "WELCOME BACK" /
  a greeting name (falling back to the literal string **"there"** for
  anonymous visitors — this is deliberate, per the code comment at
  `MobileHome.tsx:81-85`: _"back to 'there' for unauth / no-email
  visitors"_) plus a "$0.00 saved / 0 orders" stat row and an avatar
  showing "T" (the first letter of the fallback name) that links to
  `/auth`. Functionally fine, but the copy is confusing for a genuine
  first-time visitor: "Welcome **back**" implies returning/logged-in
  status they don't have, and "T" as an avatar initial (derived from the
  word "there," not any real identity) reads oddly under scrutiny.
- **Suggested fix:** for the anonymous case, swap "Welcome back" for a
  neutral "Welcome" (or hide the personalization card entirely
  pre-auth in favor of a plain CTA), and use a generic icon instead of a
  letter avatar when there's no real name to initial.

## What looked good

- **Map clustering** — numbered cluster pins with accurate counts,
  individual branded pins at zoom, smooth pan/zoom, no missing-tile or
  broken-render states across US/CA.
- **404 page** — clean, on-brand, single clear "Go home" CTA, both
  locale-scoped and top-level 404s render identically correct.
- **OTP error handling** — the wrong-code path is consistent and
  well-built everywhere it's used (onboarding, inline purchase-card
  auth): inline red error text, the user stays on the OTP step, the
  Verify button re-enables immediately, no dead-end or full-page error.
- **Payment step** — thoughtful details: QR code, copyable address,
  a clearly-boxed "Required memo" with its own copy button, a live
  countdown ("Time remaining: 29:59" ticking down in real time), and a
  graceful expiry escape hatch ("Need more time? Start over with a fresh
  payment window").
- **Order history / order detail** — the full loop (pay → mock-fulfil →
  redemption reveal → `/orders` list → order detail) rendered correctly
  end-to-end with a proper "Completed" status badge and working
  pagination controls (correctly disabled with only one order present).
- **Settings/privacy** — complete GDPR-surface implementation (data
  export, sign-out-everywhere, typed-confirmation account deletion) with
  a clean unauthenticated gate ("Sign in to export your data or delete
  your account").
- **Search autocomplete** (desktop) — fast, relevant, grouped results
  ("Amazon — 15 options" / "Amazon UAE incentive — 0.3% off") with real
  logos where the catalog has them and clean initials fallback where it
  doesn't.
- **`/cashback`'s Phase-1 gate** — a clean, correctly-worded "Coming
  soon" screen; this is the pattern UX-02 (`/calculator`) should be
  brought in line with.
- **Locale/currency formatting** — `/gb/en` product pages correctly show
  GBP (`£44.99`) instead of USD; country flag + currency stayed in sync
  across US/GB/CA locale switches everywhere except the map's initial
  view (UX-08).
- **Disabled-state button styling** — the lighter-blue "Continue" button
  before a required field is filled reads clearly as non-interactive
  without looking broken.

## Surfaces not independently assessed

- **In-store/barcode redemption reveal** — the mocked stack's
  `mark-fulfilled` test hook only exercises the URL-type redemption
  (`MOCK-CHALLENGE-...` + "Open redemption page"); the barcode/code+PIN
  reveal path wasn't separately walked in this pass. Tracked separately
  by C2-2 (`docs/go-live-plan.md` §T1-E — the redeemUrl/Code/Pin
  fulfilled-but-null bug).
- **Keyboard-only navigation** — this pass was mouse/tap-driven
  screenshotting, not a keyboard walkthrough; B-2 (`docs/readiness-backlog-2026-07-03.md`)
  is the existing accessibility-audit item that should own a full
  keyboard/focus pass.
- **Mobile FixedSearchButton** — `apps/web/app/components/features/FixedSearchButton.tsx`
  (aria-label "Search merchants") is defined but not imported by any
  route in the app — dead code, superseded by `MobileHome`'s own inline
  search. Zero user impact (never rendered), so not filed as a UX
  finding — noted here only because it caused a scripting dead-end
  during this pass before the actual mobile search UI was located.
