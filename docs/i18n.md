# i18n — how to add a translatable string, and how to add a language

Framework decision + rationale: `docs/adr/043-i18n-framework.md`. Locale
routing (`/:country/:lang`) decision + rationale: `docs/adr/034-path-based-locale-routing.md`.
This doc is the mechanical "how", not the "why."

Loop ships **English only** today (`SUPPORTED_LANGS` in
`packages/shared/src/countries.ts`). The i18next framework + a first
extraction tranche are in place (ADR 043) so that adding a second language
later is a catalog drop, not a refactor — but no second language is wired up
yet, and picking one is a 🧭 operator decision tracked in
`docs/readiness-backlog-2026-07-03.md` B-6 / `docs/go-live-plan.md` §T1-BS B-6.

## Files

```
apps/web/app/i18n/
├── i18next.ts        ← the bootstrap: creates + initializes the i18next
│                        singleton, bundles every namespace's resources,
│                        exports the instance as default
├── locales/en/
│   ├── common.json    ← strings shared verbatim across ≥2 surfaces
│   ├── footer.json    ← Footer.tsx
│   ├── notFound.json  ← not-found.tsx / not-found-ssr.tsx
│   ├── home.json      ← routes/home.tsx
│   ├── auth.json      ← routes/auth.tsx
│   ├── onboarding.json ← Onboarding.tsx + screens-trust.tsx
│   ├── settings.json  ← settings.privacy.tsx / settings.wallet.tsx /
│   │                     settings.cashback.tsx (one namespace, one
│   │                     sub-key per page — see "How to add" step 1)
│   ├── orders.json    ← routes/orders.tsx / routes/orders.$id.tsx /
│   │                     components/features/orders/LoopOrdersList.tsx /
│   │                     OrdersSummaryHeader.tsx
│   └── giftCard.json  ← routes/gift-card.$name.tsx
├── locale.ts          ← ADR 034: useLocale()/useActiveLocale()/localizedHref()
│                        — the URL-param seam. UNRELATED to string translation;
│                        i18next reads the *result* of this (the active
│                        language) but doesn't replace it.
├── format.ts          ← ADR 034: Intl currency/number/date formatting.
│                        UNRELATED to string translation — a separate live
│                        seam, don't confuse the two.
└── seo.ts              ← ADR 034: canonical/hreflang URL builders.
```

One JSON file per feature area ("namespace" in i18next terms), not one giant
flat file — `i18next.ts`'s `ns: [...]` array + `resources.en` object is the
registration point when you add a new namespace file.

## How to add a translatable string (mechanical steps)

You're touching a **customer-facing** surface (admin stays English-only —
see AGENTS.md; internal ops tooling doesn't need translation). Say you're
adding copy to a component under `apps/web/app/components/features/` or
`apps/web/app/routes/`.

1. **Find (or create) the right namespace JSON** under
   `apps/web/app/i18n/locales/en/`. Reuse an existing file if your component
   is a sub-part of a surface that already has one (e.g. a new card on the
   auth Account view goes in `auth.json`; a new `/settings/*` page goes in
   `settings.json` under its own top-level sub-key, matching how
   `privacy`/`wallet`/`cashback` are split there). A wholly new surface
   (e.g. `MobileHome.tsx`, not yet extracted) gets its own file — add it to
   both the `import` list and the `resources.en` / `ns` arrays in
   `apps/web/app/i18n/i18next.ts`.
2. **Add the key**, matching the existing dotted-nesting convention (i18next
   nests JSON objects by default — `t('account.heading')` resolves
   `{"account": {"heading": "..."}}`). Keep the **exact current English
   text** if you're doing a pure extraction (no copy changes bundled into an
   i18n PR — that's a separate, reviewable-on-its-own change).
3. **In the component**, add `import { useTranslation } from 'react-i18next';`
   and call `const { t } = useTranslation('yourNamespace');` — then replace
   the literal with `{t('your.key')}`. Every rendering function component
   gets its **own** `useTranslation()` call (cheap — it's a memoized context
   read, not a new subscription per string); don't thread `t` down as a prop
   unless the callee is a plain non-component helper function (see
   `ledgerLabel(t, type)` in `routes/auth.tsx` for that pattern, or
   `getOnboardingCopy(t, phase1Only)` in `Onboarding.tsx`).
4. **Interpolation**: `t('key', { name: value })` against a JSON string
   containing `{{name}}` — e.g. `footer.json`'s
   `"copyright": "© {{year}} Loop. All rights reserved."` called as
   `t('copyright', { year: new Date().getFullYear() })`. Interpolated values
   are **not** HTML-escaped by i18next (`interpolation: { escapeValue: false }`
   in `i18next.ts`) because React already escapes everything it renders as
   text — don't reintroduce manual escaping.
5. **Markup embedded in a string** (bold, a link, a `<br/>` mid-sentence):
   for a **short, simple** split (a highlighted word, a line break), splitting
   into two or three `t()` keys with the JSX markup authored around them is
   fine and is what this first tranche does throughout (see `home.tsx`'s
   `hero.titlePhase1Prefix` / `titlePhase1Highlight` / `titlePhase1Suffix`).
   For anything where splitting would break word order in a language that
   reorders clauses relative to English (most European languages, for
   phrases longer than a headline), use react-i18next's `<Trans>` component
   instead — it's already an installed dependency, just not yet used in this
   codebase:
   ```tsx
   <Trans i18nKey="yourNamespace:some.key">
     Click <strong>here</strong> to continue.
   </Trans>
   ```
   with the catalog value written as `"some.key": "Click <1>here</1> to continue."`
   (the child-element numbering `<1>`, `<2>`, ... follows the order
   `<Trans>`'s children were written in — verify by rendering, don't
   hand-guess the numbers).
6. **Non-component contexts** (`meta()`, `loader()` functions — anywhere
   outside the React tree, so no hooks): import the singleton instance
   directly, `import i18n from '~/i18n/i18next';`, and call `i18n.t(...)`
   instead of the hook. See `routes/not-found-ssr.tsx`'s `meta()` for the
   pattern. Prefix the key with its namespace (`i18n.t('notFound:meta.title')`)
   since there's no `useTranslation(ns)` call setting a default namespace in
   this context.
7. **Trademarked / brand terms don't get translated** — "Face ID", "Touch
   ID", "iPhone", etc. stay hardcoded string literals (see `routes/auth.tsx`'s
   `BiometricLockRow` for the pattern: the Apple product names are literals,
   only the generic "Biometrics" fallback is catalogue-driven).
8. **Tests**: if a test renders your component and asserts on the exact
   rendered string, nothing extra is needed — `apps/web/vitest.setup.ts` is
   wired into every test file via `vitest.config.ts#test.setupFiles` and
   initializes the i18next singleton (with all bundled English resources)
   before your test runs. If you're testing the catalog lookup itself
   (rare — usually only relevant when adding a whole new namespace), see
   `apps/web/app/i18n/__tests__/i18next.test.tsx` for the pattern.
9. **Run `npm run verify`** before opening a PR — same as any other change.

## Admin / internal surfaces

Do **not** extract copy under `apps/web/app/routes/admin*` or
`apps/web/app/components/features/admin/`. Admin is operator-facing tooling,
stays English-only by design (see AGENTS.md), and mixing it into a
customer-facing i18n pass adds churn with no launch value.

## How to add a language later (once the 🧭 decision lands)

This is deliberately **not** done yet — recorded here so it's a mechanical
checklist when the language-set decision is made, not a redesign.

1. **Routing**: add the language code to `SUPPORTED_LANGS` in
   `packages/shared/src/countries.ts` (ADR 034's `/:country/:lang` routing
   already validates against this — an unsupported `lang` segment 404s).
   `isSupportedLang()` picks it up automatically everywhere it's already
   called.
2. **RTL** (only if the language is Arabic/Hebrew/Farsi/Urdu, or another
   RTL script): confirm it's in `apps/web/app/i18n/locale.ts`'s `RTL_LANGS`
   set (Arabic/Hebrew/Farsi/Urdu already are). `<html dir>` flips
   automatically via the existing `root.tsx` effect — no i18next-side work
   needed for direction. Do budget real UI verification here: mirrored
   layouts (icons, chevrons, numeric fields), not just the `dir` attribute.
3. **Translate every catalog file** under `apps/web/app/i18n/locales/en/` —
   create the equivalent `apps/web/app/i18n/locales/<lang>/*.json` files with
   the same keys, translated values. This is the actual translation work
   (professional translation service or equivalent) — out of scope for any
   engineering ADR.
4. **Register the new resources** in `apps/web/app/i18n/i18next.ts`:
   ```ts
   import commonDe from './locales/de/common.json';
   // ...
   resources: {
     en: { common, footer, notFound, home, auth, onboarding },
     de: { common: commonDe, footer: footerDe, /* ...every namespace */ },
   },
   ```
5. **SSR concurrency — read before shipping a second language.** The
   i18next instance in `i18next.ts` is a **single module-scope singleton**,
   shared across every SSR request on a given server process. Today
   (English-only) that's safe because every request resolves to the same
   `lng: 'en'` — `changeLanguage('en')` is idempotent no matter which
   request calls it. The moment a second language ships, **concurrent SSR
   requests for two different locales race on this shared instance's active
   language** — request A for `/de/de/...` and request B for `/us/en/...`
   arriving close together could each read the other's language mid-render.
   Fix this **before** flipping on language #2: either (a) `i18n.cloneInstance({
lng })` per SSR request in `entry.server.tsx` (the officially-recommended
   react-i18next SSR pattern — a cheap clone that shares the loaded resource
   store but gets its own `language` state), or (b) verify Fly's deployment
   topology genuinely serializes requests per-process (it doesn't — Node's
   event loop interleaves concurrent request handling, so this needs the
   real fix, not an assumption). This is the single most important gotcha in
   this whole doc — don't skip it.
6. **Extend the extraction** to any surface the first tranche didn't cover
   (see ADR 043 "What's left") — a language rollout is a good forcing
   function to finish the remaining customer-facing tranches too, though
   it's not strictly required (untranslated surfaces just fall back to
   English via `fallbackLng: 'en'`).
7. **Country ↔ language pairing**: ADR 034's `/:country/:lang` model allows
   any routed country to pair with any supported language segment (e.g.
   `/de/en` — a German visitor who prefers English). Don't assume
   country-locked language; the UI/country-selector work for exposing a
   language switcher (independent of the country selector) is itself a
   follow-up — not built yet, since there's only one language to switch to.

## What NOT to do

- Don't add `i18next-browser-languagedetector` — the active language is
  route-derived (ADR 034), not detected from `navigator.language`. See ADR
  043 for why (the "US flash" ADR 034 already killed for country).
- Don't add `i18next-http-backend` / lazy-loaded namespaces without
  re-reading ADR 043's SSR section first — the current synchronous-init
  design is load-bearing for both the SSR and static-mobile-export paths.
- Don't bundle a copy change into an extraction PR — extraction preserves
  the exact current English text; wording changes are a separate,
  independently-reviewable diff.
- Don't translate trademarked product names.
- Don't touch `apps/web/app/i18n/format.ts` or `locale.ts` for string-catalog
  work — they're a different, already-live seam (Intl formatting / URL
  locale routing), not string translation.
