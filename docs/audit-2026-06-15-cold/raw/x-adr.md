## Findings (invariant violated or status wrong)

### F-ADR-1 — ADR 035 order-path gap: display-only markets are advertised as orderable — **P1 (P0 if launched)**

ADR 035 §Decision-1 + §Consequences state AE/IN/SA/AU/MX get "price display + **ordering** (the XLM rail)" and "**No backend change**." In code the Loop-native order path hard-blocks them three ways: `apps/backend/src/orders/loop-handler.ts:259` (`isHomeCurrency` → 400 "currency must be USD, GBP, or EUR"), the FX feeds (`payments/price-feed-fx.ts`, `price-feed.ts`) only fetch USD/GBP/EUR, and five DB CHECK constraints (`orders_currency_known`, `orders.charge_currency`, `users.home_currency`, `credit_transactions`, `user_credits`; `schema.ts:89,141,220,548,554` + `migrations/0021_orders_currency_check.sql`) lock to `('USD','GBP','EUR')`. Result: ~286 merchants are surfaced on the marketing site but every Loop-native order 400s/constraint-rolls-back. **Impact:** broken purchase funnel for the new markets; ADR/code disagree. **Fix:** either widen CHECKs across every `*_currency_known` table + add rates-API conversion + money-format, or amend ADR 035 to state ordering is deferred. Already in `comprehensive-audit-2026-06-11` Part IV Phase 3 + roadmap follow-up; **not yet remediated.**

### F-ADR-2 — ADR 027 binary-tamper deferral trigger is MET but ADR records it unmet — **P1**

ADR 027 defers binary-tamper detection with Phase-2 trigger "distribution path moves outside the official stores (e.g. … web install)." The Tranche-1 deliverable explicitly ships **Android APK sideload via direct link / Google Drive / Diawi** (`docs/tranche-1-launch.md:276,288,330`). The trigger is therefore already satisfied. `docs/roadmap.md` orphaned-work register line 34 flags this as a decision needed; ADR 027's decision table still reads the control as out-of-scope/not-triggered. ADR 007's own revisit trigger (sideloaded Android bypassing store review) is likewise active. **Impact:** an undefended deferral now in scope; either implement soft tamper detection for the sideload build or record a dated acceptance in ADR 027. **Status-accurate = NO.**

### F-ADR-3 — ADR 036 lifecycle unimplemented on main → ledger-divergence + watcher wedge — **P1 (latent P0)**

ADR 036 pins the canonical emission/redemption/burn model; it is fully built only on `fix/adr036-emission-burn` (not the audit base, not main). On main: (a) `credits/withdrawals.ts` still debits `user_credits` at send time, so a user holding withdrawal-sourced LOOP has a zeroed mirror; (b) `orders/transitions.ts:40` documents the issuer-return burn but **no code performs it**. Consequences (comp-audit CRIT #2 + HIGH): redeeming withdrawal-LOOP drives the `user_credits_non_negative` CHECK and (pre-2026-06-11 hardening) wedged the entire deposit watcher; the asset-drift watcher counts deposit-held LOOP as circulation → permanent positive drift that pins the alert. **Impact:** financial-integrity invariant ("on-chain LOOP always has a matching off-chain half") is violated on main until ADR 036 merges. **Fix:** merge the ADR 036 branch (emission re-scope + issuer-return burn + drift subtraction).

### F-ADR-4 — ADR 016 hardening asymmetry: LOOP-asset payout worker idempotency unprotected — **P2**

`payments/payout-worker-pay-one.ts:128-149` converges (`markPayoutConfirmed`) on a memo + to-address match alone; the memo+amount+asset verification added by the stranded-order-hardening to `orders/pay-ctx.ts` was not mirrored here. Risk is low (`generatePayoutMemo` is a fresh 100-bit random base32 string, destination is the user's own address → ~2⁻¹⁰⁰ collision), but for class-consistency the same guard should be applied. **Fix:** add memo+amount+asset check in `payout-worker-pay-one.ts`.

### F-ADR-5 — ADR 028 step-up gap on balance-moving siblings — **P2 (advisory)**

`refunds` (`routes/admin-credit-writes.ts:58`) and `payouts/:id/compensate` (`routes/admin-payouts.ts:99`) both move user balances but are NOT behind `requireAdminStepUp()`. This is faithful to ADR 028's literal Phase-1 pinned set (credit-adjust / withdrawal / payout-retry), so not a spec violation — but they are financially equivalent surfaces. **Fix:** fold into the ADR 028 Phase-2 "expand the gated surface" item.

### F-ADR-6 — Append-only ledger is convention-only — **P3**

ADR 009/017 invariant #4 ("never UPDATE ledger history, always append") is honoured by code (grep finds no `UPDATE credit_transactions`) but is not DB-enforced (no UPDATE-blocking trigger/RULE/REVOKE). A future buggy path would not be rejected. **Fix (optional):** add a `BEFORE UPDATE` reject trigger or `REVOKE UPDATE` on `credit_transactions`.

### F-ADR-7 — ADR 021 order-rejection status drift — **P3**

ADR 021 says order creation rejects absent-from-catalog merchants with **404**; `orders/loop-handler.ts:220-221` returns **400** "Unknown or disabled merchant". Rejection itself is correctly enforced; only the wording/status drifts. **Fix:** align status or amend ADR text.

### F-ADR-8 — ADR 005 stale revisit triggers — **P3**

§2 (barcode-redemption: trigger "barcode-primary merchant added") and §11 (token-transport: "revisit at Phase 2") were not re-checked after the June supplier program (SVS is in-store/barcode-heavy) and after ADR-013 loop-native auth shipped (`LOOP_AUTH_NATIVE_ENABLED=true` in the Tranche-1 runbook). **Fix:** dated re-check pass over all 11 ADR-005 entries. (comp-audit P2574.)
