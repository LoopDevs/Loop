Payout confirmation only updates `pending_payouts`; it does not insert a withdrawal/spend/debit ledger row and does not decrement `user_credits`:

apps/backend/src/credits/pending-payouts-transitions.ts:85-99
  export async function markPayoutConfirmed(args: { id: string; txHash: string; }): Promise<PendingPayout | null> {
    const [row] = await db
      .update(pendingPayouts)
      .set({
        state: 'confirmed',
        confirmedAt: sql`NOW()`,
        txHash: args.txHash,
        lastError: null,
      })
      .where(and(eq(pendingPayouts.id, args.id), eq(pendingPayouts.state, 'submitted')))
      .returning();
    return row ?? null;
  }

Search result:

- `markPayoutConfirmed` is the only payout-success transition.
- The only `type: 'withdrawal'` production writer is `credits/withdrawals.ts`, which is for admin-initiated withdrawals and not order-cashback payout confirmation.
- No production code inserts a negative ledger row for `kind='order_cashback'` payout confirmation.

Current unit tests encode the double-write side of fulfillment:

apps/backend/src/orders/__tests__/transitions.test.ts:271-298 asserts fulfillment writes `type='cashback'` and upserts `user_credits`.
apps/backend/src/orders/__tests__/transitions.test.ts:321-334 asserts the same qualifying fulfillment also inserts a pending payout row.

Impact reasoning:

A user with a linked Stellar address can receive a LOOP asset payout for order cashback while the same cashback remains in `user_credits`. That off-chain balance is returned in `/api/users/me` and can later fund a credit order or an admin withdrawal. The ledger invariant checker still passes because it only compares `user_credits` to `credit_transactions`; it does not reconcile confirmed `order_cashback` payouts as debits.
