### A2-2001 — **High** — Admin idempotency-key race allows double-credit on concurrent replay

**Surface:** Phase 18 §1 row 4; amplifies A2-902 (phase 6.5) and A2-612 (phase 5c).

**Files:** `apps/backend/src/admin/credit-adjustments.ts:111-188`; `apps/backend/src/admin/idempotency.ts:44-99`.

**Evidence:** The handler reads the idempotency row (`lookupIdempotencyKey`, L111-115), **then** calls `applyAdminCreditAdjustment` (L137-143), **then** stores the snapshot (`storeIdempotencyKey`, L180-188) via `onConflictDoUpdate`. Two concurrent requests with the same `(adminUserId, key)` both see a null lookup, both execute the ledger write, both upsert the snapshot (last writer wins at the snapshot table). The `admin_idempotency_keys` row constrains single-occupancy via a unique index (schema.ts:600), but nothing in the write path locks on it before the ledger write. No `SELECT ... FOR UPDATE` on the key row; no advisory lock; the `admin_idempotency_keys` row isn't even part of the same transaction as the `credit_transactions` insert.

**Exploit / impact:** a double-click on the admin UI ("Apply adjustment"), or a retrying HTTP client on a marginally-slow response, produces two `credit_transactions` rows for the same intended adjustment. `user_credits.balance_minor` is incremented twice. The admin's Discord audit channel fires twice with non-replayed flags — detectable post-hoc but not prevented. Cross-references A2-902 (no DB UNIQUE for idempotency keying on `credit_transactions`) which would catch the second insert at the DB layer. ADR 017 #2 "idempotency — repeat → snapshot replay" is therefore violated under concurrency.

**Proposed remediation:** reverse the ordering. Before the ledger write, **pre-insert** the `admin_idempotency_keys` row with `INSERT ... ON CONFLICT DO NOTHING RETURNING`. If zero rows returned, another request owns the key — wait, then re-read and replay. Alternatively, wrap the entire handler body (lookup + write + store) inside a `db.transaction` so Postgres serialises by the key's unique index. Equivalent to `UPSERT-FOR-OWNERSHIP` pattern. Cost: one extra round-trip on happy path; exact correctness under concurrency.

### A2-2002 — **Medium** — `users.email` accepts Unicode confusables / is not NFKC-normalised

**Surface:** Phase 18 §1 row 6; amplifies A2-706 (phase 6 — no unique index).

**Files:** `apps/backend/src/db/schema.ts:58, 75`; `apps/backend/src/db/users.ts:85-107`; `apps/backend/src/auth/otps.ts` (email → OTP → user resolution).

**Evidence:** `users.email` is `text` with a non-unique btree `users_email` index. Phase 6 A2-706 flagged the missing unique index. This adds: there is no NFKC normalisation or confusable-folding anywhere on the email input path. Probed analytically: `admin@loop.com` (Latin `a`, byte `0x61`) and `аdmin@loop.com` (Cyrillic `а`, byte sequence `0xD0 0xB0` in UTF-8) are two different byte sequences. Postgres text equality is byte-equality, so both rows would coexist even if a unique index were added. `grep -rE "normalize\(|NFKC|NFC" apps/backend/src` → 0 hits.

**Exploit / impact:** an attacker registers `аdmin@loop.com` (Cyrillic first letter) and talks to a support human who visually sees `admin@loop.com` and treats it as the admin. Cross-channel phishing aid. Also: OTP flow keys by the exact email string, so a Cyrillic-а user gets their own OTP stream that isn't collision-prone with the Latin-а user, but that's exactly the confusion vector — two accounts, both "admin@loop.com" to the eye. ICU-based confusable detection (e.g. `unicode-confusables`) or punycode-aware comparison is the standard fix. The scope extends beyond `users.email`: `identities.email_at_link`, `admin_audit` logs, Discord notification bodies all carry the raw bytes.

**Proposed remediation:** add NFKC normalisation at insertion (in `auth/email.ts` or `auth/otps.ts`'s `createOtp`). Reject emails whose NFKC form contains non-ASCII characters unless we intentionally support IDN-local-parts (we don't, per ADR 013). Alternative: maintain `users.email_normalised` as a generated column + unique index on that.

### A2-2003 — **Medium** — Customer-facing `POST /api/orders` does not require `Idempotency-Key`

**Surface:** Phase 18 §1 row 10; G5-39 (HTTP method safety), ADR 017 scope.

**Files:** `apps/backend/src/orders/handler.ts:createOrderHandler` (legacy); `apps/backend/src/orders/loop-handler.ts:loopCreateOrderHandler`; `apps/backend/src/app.ts:798, 805`.

**Evidence:** neither `POST /api/orders` nor `POST /api/orders/loop` parses an `Idempotency-Key` header. ADR 017 declares idempotency a primitive for ADMIN writes; customer writes are out of scope. But the loop-native order-create writes a Postgres `orders` row and — on a following payment-received tick — a `credit_transactions` cashback row, plus a `pending_payouts` row. A customer double-click or a retrying-client pattern generates two orders, each with their own `payment_memo`. Economically, the user is told to pay one order and the second is a stuck `pending_payment` that expires at 24h. No double-charge on user, but:

- two Discord `orders` channel embeds (noise).
- a confusing UX: the user sees two orders in `/account/orders`, one is a duplicate ghost.
- pre-Phase-2 cashback doesn't fire until payment, so the duplicate order never costs Loop — **but** post-Phase-2 once the user pays, if a race somehow pays both, both cashback rows fire. State-machine prevents that today but the defense is sequential.

**Exploit / impact:** UX noise, admin-side reconciliation gremlins ("why are there 2 orders from user X with identical face values 50ms apart?"), and a latent financial-correctness risk when the order state machine evolves. Not exploitable for gain today.

**Proposed remediation:** optional `Idempotency-Key` header on order-create; when absent the call still works (back-compat) but when present the handler follows the ADR 017 snapshot-replay pattern. Over time, require it on mobile builds once the client is updated.

### A2-2004 — **Low** — Discord `escapeMarkdown` does not escape deceptive-link construction or zero-width / RTL-override characters

**Surface:** Phase 18 §1 row 18; G5-124.

**Files:** `apps/backend/src/discord.ts:27-31`.

**Evidence:** `escapeMarkdown(value).replace(/([\\`_\_~|>])/g, '\\$1')`. Escape set = `\`, backtick, `_`, `\_`, `~`, `|`, `>`. **Missing:** `[`, `]`, `(`, `)`(markdown link construction);`\u202E`(right-to-left override);`\u200B..\u200D`(zero-width spaces); emoji escapes. A merchant name`click [here](https://evil.phish)`renders in Discord as a blue clickable link to the phish site. A merchant name`safe\u202Egnp.exe`renders as`safe exe.gnp` in the embed (RTL-override display spoofing).

**Exploit / impact:** Two vectors. (a) CTX upstream serves a merchant catalog with a hostile merchant name; our backend forwards it to Discord via `notifyOrderCreated` (L96-117 of discord.ts). Ops clicking the embed's link lands on the phish site. Pre-launch: ops channel only; low probability. (b) A user-controlled text (e.g. a future "note" field on an order) that flows into a Discord embed without additional sanitisation becomes an active phishing channel. `allowed_mentions: { parse: [] }` prevents `@everyone` but not link/override tricks.

**Proposed remediation:** extend the regex to `/([\\`\*\_~|>\[\]()])/g`. Additionally strip or escape Unicode control-class characters before the markdown escape: `value.replace(/[\u200B-\u200F\u202A-\u202E\u2066-\u2069]/g, '')`. Ship unit tests that pin both behaviours.

### A2-2005 — **Medium** — CTX OTP-request path has no per-email rate limit (IP-only)

**Surface:** Phase 18 §1 row 20; G5-72.

**Files:** `apps/backend/src/auth/handler.ts:requestOtpHandler` (CTX branch L59-106); compare `apps/backend/src/auth/native.ts:49-67` (native branch has `OTP_REQUESTS_PER_EMAIL_PER_MINUTE=3`).

**Evidence:** `app.ts:744` — `/api/auth/request-otp` has 5/min per-IP. Native path adds a per-email cap (3/min, `otps.ts:27`). CTX path does NOT add an equivalent — it forwards to upstream CTX's `/login` which has its own rate limits we don't control. An attacker cycling IPs past the 5/min per-IP ceiling can generate unlimited OTP-email traffic to a target victim's inbox via the CTX pathway.

**Exploit / impact:** email-spam DoS against a target inbox; brand damage ("why am I getting 200 OTPs from Loop today?"); in extremis, triggers spam-filter blacklisting that breaks legit logins. CTX's own rate limit is the backstop — they may or may not catch this, and we have zero observability into when they do.

**Proposed remediation:** in `requestOtpHandler` (CTX branch), before the circuit-breaker call, add the same `countRecentOtpsForEmail` check that the native path uses. That check counts rows in our local `otps` table; the CTX path never writes to it, so we'd need a sibling counter table (`ctx_otp_requests_by_email`) or a per-email in-memory sliding-window LRU keyed on email. The in-memory LRU is sufficient pre-launch; migrate to Postgres once we have multi-instance cadence.

### A2-2006 — **Medium** — No tamper-evident audit trail for admin actions (DB DELETE is silent; reads are unaudited)

**Surface:** Phase 18 §2 malicious-admin walk; G5-121.

**Files:** `apps/backend/src/db/schema.ts:586-605` (`admin_idempotency_keys`); `apps/backend/src/admin/audit-tail.ts`; `apps/backend/src/discord.ts:notifyAdminAudit` (sends to `DISCORD_WEBHOOK_ADMIN_AUDIT`).

**Evidence:** admin-write observability stack = (1) insert to `admin_idempotency_keys` (retention: 24h per phase 6 A2-721 sweep plan), (2) Discord `admin-audit` embed (retention: Discord's own UX defaults; deletable by any Discord admin). **No append-only, cryptographically-chained, independently-retained audit log.** There is no DB-level trigger preventing DELETE on `admin_idempotency_keys`; a DBA (or a future admin handler) can redact rows. Admin READS — CSV exports, `/api/admin/users/by-email`, `/api/admin/user-search` — emit no Discord, no DB row, nothing. A malicious admin can bulk-dump the user table zero-trace.

**Exploit / impact:** "silent balance manipulation, cover tracks, leak PII" per plan §1. The malicious-admin adversary is not a theoretical one — it's the named worst-case in the threat model. Current controls are "admin is honest and Discord audit is watched." Pre-launch acceptable because no customers; post-launch this fails any basic SOC-2 / PCI-adjacent audit.

**Proposed remediation:** (1) add an append-only `admin_audit` table separate from `admin_idempotency_keys`, with: `actor_id`, `action_verb` (read/write/export), `path`, `row_ids_touched`, `at_ts`, and a **DB trigger that denies DELETE/UPDATE** (only SYSTEM role can prune). (2) Instrument every admin reader — a single middleware on `/api/admin/*` that intercepts the handler's response and records `{actor, method, path, status, response_row_count}`. (3) Treat the Discord channel as a fan-out of this table, not the source of truth. (4) Require the admin panel's "export CSV" button to prompt for a written reason that flows into the audit row.

### A2-2007 — **Low** — `/api/public/*` endpoints are commercially-sensitive but only IP-rate-limited

**Surface:** Phase 18 §1 row 28; G5-82.

**Files:** `apps/backend/src/app.ts:699-725` (`/api/public/cashback-stats`, `/cashback-preview`, `/top-cashback-merchants`, `/flywheel-stats`, `/merchants/:id`); ADR 020 `public-api-surface.md`.

**Evidence:** each endpoint carries `rateLimit(60, 60_000)` per IP. From a residential ISP with IP rotation (typical US ISPs rotate /64 blocks every 24h) or a commercial proxy service ($5/month for 1000 rotating IPs), an attacker can fully map the merchant catalog, cashback percentages, monthly stats, and flywheel data in hours. That data is the core commercial signal of the business.

**Exploit / impact:** competitive intelligence exfiltration. A competitor pre-launch-scrapes our cashback configs → builds a matching offering with their own capital. Low severity because: (a) the data is intended for public consumption on the marketing site anyway, (b) the margin structure is in-catalog but not private, and (c) pre-launch there's no competitive harm yet. Would become Medium once we're positioned against a specific competitor.

**Proposed remediation:** add a Cloudflare/Fly-edge bot-detection layer in front of `/api/public/*`; optional CAPTCHA for high-rate callers; consider an `Authorization: Bearer <anon-token>` cookie that gates full-depth responses vs preview. Not urgent; file for post-launch.

### A2-2008 — **Medium** — Admin read-only endpoints emit no audit trail whatsoever

**Surface:** Phase 18 §2 malicious-admin walk; cross-cut with A2-2006.

**Files:** all `apps/backend/src/admin/*.ts` handlers except the write-side (`credit-adjustments.ts`, `merchants-resync.ts`, `payouts.ts:adminRetryPayout`). Representative: `admin/users-list.ts`, `admin/user-credits-csv.ts`, `admin/reconciliation.ts`, `admin/audit-tail.ts`.

**Evidence:** `grep -rE "notifyAdminAudit|admin_idempotency_keys" apps/backend/src/admin` → instrumentation exists only on the three write endpoints. Every other admin route (≥ 65 handlers) returns data silently. `admin/user-credits-csv.ts` streams every user's balance to a CSV with no Discord embed and no DB row.

**Exploit / impact:** a malicious admin can bulk-exfiltrate user PII and the entire credit ledger without leaving a single HTTP-layer trace. Detection requires reading access logs (Pino-backed at `app.ts:350-360`) which live in Fly's log stream and are retention-bound. If an admin is compromised, the reconstruction effort for "what did they look at" is forensic-grade, not operational-grade.

**Proposed remediation:** see A2-2006 §Proposed remediation item (2) — one `/api/admin/*` middleware that records every admin read. Explicitly call out CSV exports with `response_row_count` and `query_filters_json` so a "export all users" action is as visible as a "credit-adjustment" write.

## 6. Pre-launch "top five if this shipped today" risks

Ranked by the question _"what's the worst thing an attacker could do between launch-midnight and morning-standup?"_:

1. **A2-550 unverified CTX JWT → A2-551 stellar-address redirect.** Attacker with any bearer token (logs, screenshots, shared-device) forges a JWT for another user and redirects their Stellar payouts. **Critical — ships broken.**
2. **A2-1608 refresh-token reuse-detected-but-not-revoked.** A stolen refresh token keeps issuing access tokens for up to 30 days even after the legit user's next login warns us the token is compromised. **High — silent session continuation.**
3. **A2-619 cross-currency watcher mismatch.** First non-home-currency order post-launch fails payment verification unconditionally. **Critical for cross-currency users; dormant until one arrives.**
4. **A2-2006 no tamper-evident admin audit trail + A2-2008 reads unaudited.** Malicious or compromised admin exfiltrates the user table or silent-balances a confederate with zero trace. **High for a regulated-finance-adjacent product.**
5. **A2-2001 idempotency race on admin credit adjustments.** A double-click by a well-intentioned admin doubles a credit. **High — silent financial drift.**

(All five are pre-existing except #4 and #5 of which #4 synthesises across A2-2006/A2-2008 and #5 is net-new from this phase.)
