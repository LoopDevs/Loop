### 1.3 Error-propagation faithfulness

`apiRequest`/`authenticatedRequest` (`apps/web/app/services/api-client.ts:69-99`): if the backend body isn't `{code,message}`-shaped, the client synthesises `code: 'UPSTREAM_ERROR'` (line 85, 93, 96). `clusters.ts:56-81` repeats the same logic by hand (duplication already noted as phase-10-shared consolidation candidate). Behaviour is consistent — if a handler emitted a non-string code, the client would clobber it to `UPSTREAM_ERROR` and downstream `switch (err.code)` branches in the UI would never see the backend intent. This is defensive but hides server misbehaviour.

### 1.4 Findings

- **A2-1500 (Medium)** — `OrderState` redeclared in 4 web files rather than imported from `@loop/shared/order-state.ts`. A future CHECK-constraint widening (e.g. adding `refunded`) updates schema + shared tuple but leaves every web literal-union intact — TypeScript then rejects assignment of the new backend response into the stale union, failing at runtime when the DB returns a valid state. See table 1.2 for exact locations.
- **A2-1501 (Medium)** — `PayoutState` redeclared in `user.ts:93` and `admin.ts:151`. Same risk as A2-1500.
- **A2-1502 (Medium)** — `OrderPaymentMethod` redeclared in `orders-loop.ts:19`, `user.ts:366`, `admin.ts:112`, `admin.ts:1165`.
- **A2-1503 (Medium)** — `LoopAssetCode` redeclared as an inline literal in `user.ts:174` and `public-stats.ts:102`. Same drift risk when a new LOOP asset is configured.
- **A2-1504 (High)** — `CreateLoopOrderBody`/`CreateLoopOrderResponse`/`LoopOrderView` have **no** shared representation — backend zod (`orders/loop-handler.ts`) and web (`orders-loop.ts:21-74`) independently declare them. A backend-side field addition or rename goes silently unnoticed by the web until runtime. ADR 019 three-part test says "two consumers → consolidate"; here the consumers are backend + web, same shape, no shared module. Phase 7 already noted these routes are **not registered in openapi.ts** (drift F-1), compounding the risk — no generated types exist either.
- **A2-1505 (High)** — `/api/users/me*` response shapes (13 of them listed in table 1.2) are redeclared in `user.ts` only. No shared types, no compile-time tie to `openapi.ts` or to the handler implementations.
- **A2-1506 (High)** — `/api/admin/*` response shapes (~30 shapes across 1948 LOC of `admin.ts`) exist only in the web consumer; backend side is handler-local types + `openapi.ts` z-schemas. With 77 admin routes and ~19 unregistered in openapi.ts (phase-7-api §3.1), the drift surface is largest here.

### 1.5 Continuous drift detector (G5-68)

`.github/workflows/ci.yml` was fully read. Job inventory:

1. `quality` — typecheck + lint + format:check + lint:docs
2. `test-unit` — vitest with coverage
3. `audit` — `npm audit --audit-level=high`
4. `build` — builds backend, SSR web, mobile web
5. `test-e2e` — Playwright against real CTX, PR-only
6. `test-e2e-mocked` — Playwright mocked
7. `notify` — Discord webhook

No job compares `openapi.ts`-derived shapes against `apps/web/app/services/**` consumer types. `scripts/lint-docs.sh` only enforces `.env.example` ↔ `env.ts` parity. There is **no** continuous drift detector for the web↔backend contract.

- **A2-1507 (High)** — No CI job fails when `apps/backend/src/openapi.ts` output diverges from `apps/web/app/services/**` consumer types. G5-68 expectation unmet. With 148 routes and ≈19 already undocumented in openapi.ts (phase-7-api F-1), drift is detectable only by downstream runtime breakage. Proposed remediation: add a CI job that (a) boots backend, curls `/openapi.json`, (b) runs `openapi-typescript` or `openapi-zod-client` to generate consumer types into a tempdir, (c) `diff` against web's re-declared shapes or fail-if-missing. Alternative: expose a `@loop/shared-api` package holding zod schemas used by both sides (inverts the current direction where web duplicates).

### 2.3 Findings

- **A2-1508 (High — cross-references phase-5c.A2-622)** — CTX procurement POST (`orders/procurement.ts:243`) has no `Idempotency-Key` / request-id. Crash between POST ack and `markOrderFulfilled` double-bills Loop on the next retry tick. Already logged; Phase 11 confirms no client-side mitigation.
- **A2-1509 (Medium)** — No continuous contract-check against CTX schema drift. Every handler zod-validates on each call (good), but `G4-07` notes a silent CTX field-rename is caught only when a request happens to fail. No canary + alert; no published CTX openapi snapshot pinned in-repo. Pre-launch acceptable; noted.
- **A2-1510 (Medium)** — No enforced per-request timeout in `operatorFetch` (grep `AbortSignal.timeout` in `ctx/operator-pool.ts`). CTX stall → backend stall past the client's 30s limit → 502 with stale in-flight.
- **A2-1511 (Low)** — `upstream.ts:19-45` validates path input but accepts arbitrary paths — no allowlist of known CTX endpoints. A future code path mistyping `upstreamUrl("/admin")` would construct a request to CTX's admin surface with operator creds. Defense-in-depth: a `upstreamUrl` enum or allowlist would pin the surface.

### 3.3 Findings

- **A2-1512 (High — cross-references phase-5c.A2-602 and .A2-603)** — Boundary-level observation: when Horizon returns 503, the backend flips the row to `submitted` but no downstream worker re-scans `submitted` rows, so the payout is stuck. From a cross-app contract standpoint, the web's `/api/users/me/pending-payouts` will forever report `state=submitted` to the user with no path to `confirmed`. The user-facing service type `UserPendingPayoutView.state` admits all 4 values (user.ts:104) but the user has no remediation action. Compounding: no Discord notifier for "payout stuck in submitted for > N min".
- **A2-1513 (Medium)** — `payout-worker.ts:287` reads `process.env.LOOP_STELLAR_HORIZON_URL` directly rather than via `env.ts` (already A2-608). From a cross-boundary perspective: deployment-time misconfiguration (e.g. env var typo) goes undetected — `env.ts` doesn't fail boot, and the worker silently falls back to mainnet Horizon.
- **A2-1514 (Low)** — Horizon's memo field has a 28-byte limit for `MEMO_TEXT`. Outbound memo is sliced to 28 chars; inbound is 20 chars. Cross-boundary: a retro change to 32-char memos on one side silently truncates on the other. No shared constant.

### 6.1 Consistency verdicts

- **Minor-unit handling:** three algorithms coexist:
  1. bigint-safe pad+slice (`discord.ts`, shared `formatMinorCurrency`, `LoopOrdersList`, `LoopPaymentStep`)
  2. `Number(minor)/100` (≈12 files) — precision loss past `2^53 - 1`; for per-user balances acceptable, for fleet totals risky
  3. Raw `bigint.toString()` with no scaling (all backend CSVs + logs)
- **Symbol source:** `Intl.NumberFormat` (≈22 call sites) vs hard-coded `CURRENCY_SYMBOLS` map (Discord). Intl's `narrowSymbol` returns `"£"` for GBP; Discord's map returns `"£"`. For `CAD` Intl returns `"CA$"` (narrowSymbol), Discord returns `"C$"`. Drift.
- **Symbol vs code ordering:** Intl output usually puts symbol before digits (`"$1,234.00"` with `currencyDisplay: 'code'` → `"USD 1,234.00"`). Discord renders `<symbol><digits> <code>` (e.g. `"£10.00 GBP"` — both symbol and code). `LoopOrdersList` and `LoopPaymentStep` render bare `<digits>` and append currency at the call site.
- **Locale:** ≈14 call sites pin `'en-US'`, ≈7 pass `undefined` (browser default), Discord pins `'en-US'` for grouping only. A GBP user on a French browser sees `"£1,000.00"` from the en-US sites and `"1 000,00 £"` from the undefined-locale sites.

### 6.2 Findings

- **A2-1520 (Medium)** — Three distinct minor→major conversion algorithms across the repo (bigint-safe, Number-cast, raw). Admin fleet-total UI surfaces use `Number(minor)/100` (#5..#13 in matrix) — vulnerable to `2^53` precision loss once lifetime-cashback-minor exceeds 9e13 (≈$9e11). Pre-launch fine, post-launch a few years out is not.
- **A2-1521 (Medium)** — Locale inconsistency: ≈14 sites hard-code `'en-US'`, ≈7 use browser default. Two widgets on the same admin page can render "£1,000.00" and "1 000,00 £" depending on browser locale. Pick one.
- **A2-1522 (Medium)** — Symbol inconsistency: Discord `CURRENCY_SYMBOLS` hard-codes `{USD,EUR,GBP,CAD}` and omits every other ISO code, while web uses Intl for all codes. For non-listed currencies Discord shows `"10.00 NOK"` and web shows `"kr 10"` — same data, different renderings in two surfaces ops and users might cross-reference. ADR 015 only sets home-currencies to USD/GBP/EUR so live exposure is low; catalog currencies (phase-5c.A2-619) are broader.
- **A2-1523 (Medium)** — CSV exports emit raw minor-unit integers (e.g. `"100000"` for £1,000.00) without a scale column or header marker. A support/compliance user opening a spreadsheet without reading the doc will misread amounts by 100×. Consider either formatted major units or a column suffix `_minor` to make it explicit, and/or a second column for the scale.
- **A2-1524 (Low)** — The shared canonical helper `packages/shared/src/money-format.ts` is only used by 4 web files (FlywheelChip, CashbackSummaryChip, test files) out of ≈29 money-formatting call sites. The "web-only" adoption matches phase-10-shared A2-815 — the three-part test is not met because backend doesn't consume it. Phase 11 cross-boundary lens: consolidation would fix A2-1520/1521/1522 in one stroke.

## 8. Client version skew tolerance (G5-69)

### 8.1 Fact pattern

- `apps/web/app/services/api-client.ts:219` sends `Authorization: Bearer <token>` and `X-Client-Id: loopweb|loopios|loopandroid` — no `X-Client-Version` or `User-Agent` override.
- `apps/backend/src/**` was grepped for `X-Client-Version|clientVersion|apiVersion` — **zero matches**. Backend has no version negotiation / deprecation header.
- No `/api/v1/*` or `/api/v2/*` path prefixes; all routes are unversioned under `/api/*`.
- No build-time commit-SHA or app-version stamped into the mobile bundle that the client returns to the backend.
- Capacitor app is static-exported and bundled into the native binary (phase-9-mobile.md). An old app version on a user's device → the bundled `/api/services/*` code is frozen at the bundle's commit.

### 8.2 Implicit contract assumptions (old client hitting new backend)

| Class of change                                                    | Backward-compatible?                                                                                                                                                                                                                                         | Risk                                                               |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------ |
| Backend adds a new response field                                  | Yes — web ignores unknown keys                                                                                                                                                                                                                               | safe                                                               |

| Backend narrows an enum                                            | Yes — old client still sends valid subset                                                                                                                                                                                                                    | safe                                                               |
| Backend adds a new required request field                          | **No** — old client gets 400 on POST                                                                                                                                                                                                                         | medium                                                             |
| Backend adds a new endpoint                                        | Yes — old client never calls it                                                                                                                                                                                                                              | safe                                                               |
| Backend changes error-envelope shape                               | **No** — old `api-client.ts` normaliser catches `{code, message}` drift, but unknown envelopes become `UPSTREAM_ERROR`                                                                                                                                       | low                                                                |
| Backend rotates `LOOP_JWT_SIGNING_KEY` without `_PREVIOUS` overlap | **No** — old sessions die                                                                                                                                                                                                                                    | medium                                                             |

### 8.3 Findings

- **A2-1529 (High)** — No client-version header, no deprecation header, no `/api/v1` prefix. The backend cannot detect "old client" to emit a feature-flag or graceful-degradation response, nor can ops tell "how many 30-day-old clients are still hitting prod?" Cross-boundary: this is a compounding risk against every A2-1500..1506 shape drift — web re-declarations are frozen in the mobile binary, and without version awareness the backend can't selectively downgrade a response. Proposed: (a) stamp a commit-SHA or semver into the mobile bundle at build time; (b) forward it as `X-Client-Version` via `api-client.ts`; (c) backend logs + exposes a `/metrics` counter keyed on version.
- **A2-1530 (Medium)** — Backend has **no backward-compat promise document**. ADR 013 covers auth, ADR 019 covers shared types, but there's no single doc saying "fields may only be added, never removed or renamed without a 6-month window". Phase 11 lens: A2-1506 (~30 admin shapes in web-only) are especially at risk because no contract says "these are stable". For admin surface this is less severe (employee-controlled clients); for user-facing `orders*`/`user*` services it's higher.

Summary: 9 switches over closed-set unions; **1 (row 7) uses `default:`**, defeating compile-time exhaustiveness. 8 are compile-time-exhaustive but **zero use `assertNever(...)`** — a grep of the entire monorepo for `assertNever` returns zero hits. This matters because a compiler-exhaustive switch that returns a value (as in rows 1, 2, 3, 4, 8, 9) is exhaustive; but a switch used for side-effects with no return would silently allow fall-through on widen.

ESLint rule `@typescript-eslint/switch-exhaustiveness-check` is **not** enabled in `eslint.config.js` (full read of the file — only `react-hooks/exhaustive-deps` is turned on at line 85). So new switch statements are not protected from widen drift.

### 9.1 Findings

- **A2-1531 (High)** — `apps/web/app/components/features/orders/LoopOrdersList.tsx:221-230` has `switch (state)` with `default:` falling back to the yellow "in-flight" pill. Cross-boundary: if the backend widens `OrderState` (e.g. `refunded`, `chargeback`), fulfilled/failed/expired rows render correctly, but refunded rows display as "in-flight" yellow. User confusion, not data corruption.
- **A2-1532 (High)** — `assertNever(...)` utility exists nowhere in the codebase. No switch returning a value protects against an enum widen; no switch returning `void` is exhaustiveness-checked. `@typescript-eslint/switch-exhaustiveness-check` not enabled. Remediation: (a) add lint rule, (b) add a `packages/shared/src/assert-never.ts` helper, (c) require a default branch in non-returning switches that calls it.
- **A2-1533 (Medium)** — All OK-verdicted switches above are only exhaustive because they return a value _and_ every case has an explicit return. A code reviewer adding a `log.info` above the return statements (or converting one case to a no-op) silently drops compile-time exhaustiveness. Brittle — A2-1532's remediation is the mitigation.
