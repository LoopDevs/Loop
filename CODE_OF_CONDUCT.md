# Code of Conduct

Closes A2-1801.

Loop is a small pre-launch team shipping a financial product. The
standard we hold ourselves to is that anyone contributing or
reviewing code — whether they are an employee, a contractor, a
security researcher, or a visitor commenting on a public PR — can
do so without being attacked, demeaned, or made to feel unwelcome.

## The short version

Be straightforward. Be kind. Attack ideas, not people. Assume the
other person is doing their best with the information they have.

## What we expect

- **Useful feedback over cleverness.** A review comment that
  explains a concrete problem and a concrete next step is worth
  more than a memorable one-liner that leaves the author guessing.
- **Disagreement in technique is normal; contempt is not.** We
  will ship code that later turns out to be wrong. When that
  happens we fix it; we do not pretend the author was careless.
- **Respect people's time.** If you're blocking on something, say
  what you need. If you're offering a review, keep the nit pile
  small and the must-change pile specific.
- **Be careful with other people's context.** A PR review is not
  the place to litigate someone's scope decisions if the original
  ask didn't include that scope.
- **Credit where it's due.** If you borrowed an approach from a
  prior PR / ADR / external writeup, say so.

## What we won't tolerate

- Harassment, threats, or personal attacks — in PRs, issues,
  Discord channels, or any other surface we operate.
- Discriminatory language about a person's race, gender, sexuality,
  disability, nationality, age, religion, or appearance.
- Sustained disruption — flooding a channel with off-topic content,
  brigading a PR, or repeatedly raising the same rejected argument
  after the decision has been made.
- Sharing private information (another contributor's real name,
  address, employer, or other identifying details) without their
  explicit consent.
- Using Loop's access privileges (repo write, Discord admin, cloud
  credentials, operator accounts, CI secrets) to intimidate,
  retaliate against, or silence another contributor.

## Scope

This applies anywhere Loop operates as a project: this GitHub repo,
our Discord channels, Loop-hosted forums or chat rooms we add in
future, and in-person events run under the Loop name. It also
applies to contributor-to-contributor conduct on third-party
platforms (email, social media) when one party is representing Loop
or when the interaction directly references Loop work.

## Reporting

If you experience or witness behaviour that violates this code,
report it privately to any of:

1. **`conduct@loopfinance.io`** — the standing address. Read by the
   Loop operations team (currently one person — see
   `.github/CODEOWNERS` for the active maintainer).
2. **DM to a maintainer on the Loop Discord** — for something you
   want to raise fast without an email trail.
3. **GitHub Security Advisory** at
   `github.com/LoopDevs/Loop/security/advisories/new` — use this
   if the conduct issue is itself a security concern (e.g., a
   contributor abusing access to exfiltrate data).

What to include: a short description of what happened, when, where
(channel / PR / issue URL), who was involved, and any supporting
messages or screenshots. Reports are confidential by default.

## How reports are handled

- **Acknowledged within 2 business days.** We don't leave people in
  silence.
- **Assessed within 7 days.** Response is proportionate: a
  clarifying conversation, a warning, temporary loss of repo /
  Discord access, or permanent removal from the project. For
  severe violations (threats, doxxing, discrimination) the
  response is immediate and skips escalation steps.
- **Reporter protection.** Retaliation against a good-faith
  reporter is itself a violation and is treated as such.
- **Transparent outcomes.** The reporter is told what action was
  taken. A public post-mortem is only published with the
  reporter's consent.

## Maintainer conduct

Maintainers — anyone with repo write, Discord admin, or deploy
access — hold themselves to a stricter bar:

- Use access only for project work, never to settle scores.
- Recuse from enforcement decisions in which you are a party.
- If ambiguous, escalate to another maintainer rather than acting
  unilaterally.
